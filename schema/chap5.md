# 제5장: LangChain과 MCP의 연결: "도구를 쓰는 에이전트" — 집필 계획서

## 장 개요

이 장은 4장에서 만든 MCP 서버를 LangChain 에이전트와 연결하여, LLM이 도구를 선택하고 호출하는 흐름을 구현한다. 학습자는 (1) 에이전트가 도구를 선택하는 메커니즘, (2) 툴 선택 실패를 줄이는 프롬프트 구조, (3) 관측 가능성(로그/트레이스), (4) 입력 검증을 직접 구현하며, "도구를 쓰는 에이전트"의 기본 패턴을 익힌다. 실습에서는 OpenAI API와 LangChain을 사용하여 MCP 도구 2개 이상을 연결한 파이프라인을 구현하고, 실제 실행 결과를 파일로 저장한다.

### 학습 목표

1) LangChain 에이전트가 도구를 선택하고 호출하는 흐름을 설명한다.
2) 툴 선택 실패를 줄이기 위한 프롬프트 구조와 도구 설명 작성 원칙을 적용한다.
3) 에이전트 실행 과정을 로그와 트레이스로 추적 가능하게 만든다.
4) 도구 입력을 검증하여 잘못된 호출을 사전에 차단한다.
5) 실습 산출물(에이전트 실행 로그, 최종 결과 JSON)을 저장하고 해석한다.

---

## 5.1 에이전트와 도구의 관계: LLM이 도구를 선택하는 방식

### 집필 방향

에이전트가 도구를 "알아서" 선택하는 것처럼 보이지만, 실제로는 LLM이 도구 설명을 읽고 적합한 도구를 판단하는 과정임을 설명한다. 이 판단이 실패하는 경우(잘못된 도구 선택, 불필요한 도구 호출)를 다루며, 도구 설명과 프롬프트가 왜 중요한지 강조한다.

### 강화 항목(필수)

- 에이전트의 "생각(reasoning)" → "행동(action)" → "관찰(observation)" 루프
- 도구 설명이 LLM의 도구 선택에 미치는 영향
- 도구가 여러 개일 때 선택 충돌 문제

### 표(권장)

**표 5.1** 에이전트 실행 루프의 단계

| 단계 | 설명 | 예시 |
|-----|------|------|
| 생각(Thought) | 현재 상황을 분석하고 다음 행동 결정 | "날씨 정보가 필요하다" |
| 행동(Action) | 선택한 도구와 입력 파라미터 지정 | `get_weather(lat=37.5, lon=127)` |
| 관찰(Observation) | 도구 실행 결과 확인 | `{"temp": 15, "condition": "Clear"}` |
| 반복/종료 | 목표 달성 여부 판단 | 최종 답변 생성 또는 다음 행동 |

## 5.2 툴 선택 실패를 줄이는 프롬프트 구조

### 집필 방향

도구 선택 실패의 원인을 유형별로 분류하고, 각각에 대한 대응 전략을 제시한다. 도구 설명 작성의 모범 사례, 시스템 프롬프트에서 도구 사용 지침을 명시하는 방법, 도구 이름과 파라미터 명명 규칙을 다룬다.

### 강화 항목(필수)

- 도구 설명 작성 원칙: 목적, 입력, 출력, 제약을 명확히
- 유사한 도구가 있을 때 구분 방법
- 도구를 호출하지 말아야 할 상황 명시

### 핵심 코드(3-5줄)

```python
tools = [
    Tool(
        name="get_weather",
        description="현재 날씨를 조회합니다. 위도와 경도가 필요합니다.",
        func=weather_tool
    )
]
```

## 5.3 관측 가능성: 로그와 트레이스

### 집필 방향

에이전트가 "블랙박스"처럼 동작하면 디버깅이 어렵다. LangChain의 콜백 시스템과 LangSmith(또는 자체 로깅)를 활용하여 에이전트의 각 단계를 추적하는 방법을 설명한다. 운영 환경에서 문제를 재현하고 개선하기 위한 로깅 전략을 다룬다.

### 강화 항목(필수)

- LangChain 콜백을 활용한 단계별 로깅
- 토큰 사용량, 응답 시간 기록
- 실패 시 컨텍스트 보존

### 핵심 코드(3-5줄)

```python
from langchain.callbacks import StdOutCallbackHandler

agent.invoke(
    {"input": query},
    config={"callbacks": [StdOutCallbackHandler()]}
)
```

## 5.4 입력 검증: 잘못된 도구 호출 방지

### 집필 방향

LLM이 생성한 도구 입력이 항상 올바른 것은 아니다. 타입 오류, 범위 초과, 필수 파라미터 누락 등의 문제가 발생할 수 있다. Pydantic을 활용한 입력 검증, 검증 실패 시 에이전트에게 피드백하는 패턴을 다룬다.

### 강화 항목(필수)

- Pydantic 모델로 도구 입력 스키마 정의
- 검증 실패 시 명확한 에러 메시지 반환
- 에이전트가 에러를 보고 재시도하도록 유도

### 핵심 코드(3-5줄)

```python
from pydantic import BaseModel, Field

class WeatherInput(BaseModel):
    latitude: float = Field(..., ge=-90, le=90)
    longitude: float = Field(..., ge=-180, le=180)
```

## 5.5 실습: OpenAI + LangChain으로 도구를 쓰는 에이전트 만들기

### 실습 목표

1) OpenAI API와 LangChain을 사용하여 에이전트를 구현한다.
2) 2개 이상의 도구(날씨 조회, 파일 저장)를 연결한다.
3) 에이전트 실행 과정을 로그로 기록하고, 최종 결과를 JSON 파일로 저장한다.

### 도구 구성

1. **날씨 조회 도구**: 위도/경도를 받아 현재 날씨 정보 반환 (4장 MCP 서버 재활용 또는 직접 API 호출)
2. **파일 저장 도구**: 텍스트 내용을 지정된 파일에 저장
3. (선택) **도시 좌표 조회 도구**: 도시 이름을 받아 위도/경도 반환

### 실습 구성

1. **환경 설정**: `.env`에 `OPENAI_API_KEY` 설정
2. **도구 정의**: LangChain Tool 형식으로 도구 래핑
3. **에이전트 생성**: OpenAI 함수 호출 기반 에이전트
4. **실행 및 로깅**: 콜백을 통한 단계별 로깅
5. **결과 저장**: 최종 결과를 `data/output/`에 JSON으로 저장

### 필요 라이브러리

- `langchain`: 에이전트 프레임워크
- `langchain-openai`: OpenAI 통합
- `openai`: OpenAI API 클라이언트
- `python-dotenv`: 환경 변수 관리
- `pydantic`: 입력 검증

### 실습 코드

- `practice/chapter5/code/5-5-langchain-agent.py`: 에이전트 구현
- `practice/chapter5/code/requirements.txt`: 의존성 목록

### 실습 산출물(필수)

- `practice/chapter5/data/output/ch05_agent_log.txt`: 에이전트 실행 로그
- `practice/chapter5/data/output/ch05_result.json`: 최종 실행 결과
- `practice/chapter5/data/output/ch05_weather_report.txt`: 에이전트가 생성한 날씨 보고서

## 5.6 실패 사례와 디버깅

### 집필 방향

에이전트 실행 중 발생할 수 있는 실패 유형(잘못된 도구 선택, 무한 루프, 토큰 초과 등)과 디버깅 방법을 다룬다. 실제 실패 사례를 통해 원인 분석과 해결 과정을 보여준다.

### 강화 항목(필수)

- 잘못된 도구 선택 시 로그 분석
- 무한 루프 방지를 위한 최대 반복 횟수 설정
- 토큰 한도 초과 시 대응

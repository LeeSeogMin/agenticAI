# 제3장: MCP 개념과 도구/리소스 설계 — 집필 계획서

## 장 개요

이 장은 MCP(Model Context Protocol)를 “특정 제품 기능”이 아니라, 에이전트가 외부 세계와 상호작용하기 위한 **표준화된 인터페이스(도구/리소스)** 관점에서 소개한다. 학습자는 이 장을 통해 (1) 도구(tool) 설계의 기본 원칙(입출력, 오류, 제약, 안전), (2) 리소스(resource)로 컨텍스트를 관리하는 방식, (3) 구현 전에 반드시 작성해야 하는 ‘명세서’를 익힌다. 이후 제4장에서 외부 API 래핑을 실제 MCP 서버 구현으로 확장할 수 있도록 연결한다.

### 학습 목표

1) MCP에서 tool/resource가 각각 무엇을 의미하는지 설명한다.  
2) 좋은 tool 인터페이스의 조건(입출력/에러/제약/멱등성/보안)을 체크리스트로 정리한다.  
3) 구현에 앞서 “도구 명세(JSON)”를 작성하고, 이를 기반으로 최소 스켈레톤 서버를 구성한다.  
4) 실습 산출물(명세/요청·응답 로그)을 파일로 저장한다.

---

## 3.1 MCP를 도입하는 이유: 도구 표준화와 운영 가능성

### 집필 방향

“도구를 붙인다”는 말을 기능 나열로 풀지 않고, 운영 관점(일관된 I/O, 실패 처리, 감사 가능성)으로 풀어 설명한다. 제1장의 핵심(검증 가능한 산출물)과 연결해, MCP가 단순 편의 기능이 아니라 ‘검증 가능한 도구 호출’을 가능케 하는 구조라는 점을 강조한다.

### 강화 항목(필수)

도구 호출이 늘어날수록 인터페이스 불일치가 비용이 되므로, 표준화(스키마/에러/제약)가 왜 중요한지 사례로 설명한다(숫자·통계는 사용하지 않는다).

---

## 3.2 Tool 설계 원칙: 입출력·에러·제약을 먼저 쓴다

### 집필 방향

좋은 tool은 “기능”보다 “계약(contract)”이 먼저이며, 최소한 다음을 명시해야 한다는 흐름으로 서술한다.

- 입력 스키마(필수/선택, 타입, 범위)  
- 출력 스키마(성공/실패 형태)  
- 오류 코드/메시지 정책  
- 시간/비용 제약(타임아웃, 레이트 리밋)  
- 안전(멱등성, 파괴적 동작 조건, 승인 필요 여부)

### 표(권장)

tool 명세서 템플릿 표 1개(“항목/의미/예”)를 포함한다.

---

## 3.3 Resource로 컨텍스트를 다루는 방식

### 집필 방향

리소스를 “파일처럼 읽는 컨텍스트”로 설명하고, tool과 비교해 언제 resource가 유리한지(재사용, 캐싱, 접근 범위 제한)를 설명한다. 본 저장소의 규칙(프로젝트 범위 filesystem 서버, `mcp.json`)과 연결한다.

---

## 3.4 실습: 도구 명세를 만들고 최소 스켈레톤으로 검증하기

### 실습 목표

학습자가 도구 명세를 먼저 작성하고, 그 명세대로 요청/응답을 흉내 내며 “계약 중심 개발”을 경험하게 한다. 이 실습은 MCP SDK 의존 없이도 수행 가능해야 하므로, 표준 라이브러리만 사용하는 “스켈레톤 서버”를 제공한다. 본문에서는 이것이 **완전한 MCP 구현이 아니라 설계 연습용 최소 골격**임을 명확히 적는다.

### 실습 산출물(필수)

- `practice/chapter3/data/output/ch03_tool_spec.json`  
- `practice/chapter3/data/output/ch03_demo_request.json`  
- `practice/chapter3/data/output/ch03_demo_response.json`

### 실습 코드

- `practice/chapter3/code/3-4-tool-spec-and-skeleton.py`
- 요구사항:
  - tool 명세 JSON 생성
  - 샘플 요청 JSON 생성
  - 요청을 읽어 응답 JSON 생성(간단한 유효성 검사 + 에러 형태 포함)
  - 모든 산출물을 `data/output/`에 저장

### 본문 “핵심 코드”(3–5줄)

명세 파일 경로 구성 및 저장(출력 폴더 생성 포함) 또는 입력 검증의 핵심 3–5줄만 발췌한다.

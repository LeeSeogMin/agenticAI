# 11주차. MCP + Skills 통합 프로젝트

> **1회차** (강의 90분): MCP와 Skills의 역할 분담, 3가지 설계 패턴, 디버깅과 트레이싱
> **2회차** (실습 90분): 주간 보고서 자동 생성 에이전트 구축

---

## 학습목표

1. MCP(도구 접근)와 Skills(지식/절차)의 결합이 만드는 시너지를 구조적으로 이해한다
2. 세 가지 설계 패턴(단일 도구, 파이프라인, 감독자)을 상황에 맞게 선택하고 적용한다
3. 에이전트 동작의 전 과정을 추적하고 문제 발생 시 체계적으로 디버깅한다
4. MCP + Skills를 결합하여 실용적인 업무 자동화 에이전트를 설계하고 구현한다

## 선수지식

9주차에서 다룬 SKILL.md 파일의 구조와 작성 방법, 10주차에서 다룬 스킬 설계 패턴(체크리스트형, 워크플로우형, 가드레일형)을 숙지하고 있어야 한다. 또한 5주차에서 실습한 MCP 서버 연결 방법과 `.vscode/mcp.json` 설정 경험이 필수적이다. GitHub MCP 서버의 도구(`list_issues`, `list_pull_requests` 등)를 직접 호출해 본 경험이 있으면 이번 주차의 통합 프로젝트를 원활하게 수행할 수 있다.

## 1회차: 강의

### 11.1 MCP와 Skills의 역할 분담

9주차와 10주차에서 Skills를, 4~7주차에서 MCP를 각각 독립적으로 학습하였다. 이 절에서는 두 요소가 결합했을 때 나타나는 시너지를 구조적으로 분석한다.

#### 두 요소의 본질적 차이

MCP는 Copilot이 "무엇을 할 수 있는가"를 결정한다. GitHub API에서 이슈를 조회하거나, 파일 시스템에서 문서를 읽거나, 데이터베이스에 쿼리를 실행하는 것은 모두 MCP 서버가 제공하는 도구(Tool)를 통해 이루어진다. MCP가 없으면 Copilot은 IDE 내부의 파일 편집과 터미널 명령만 수행할 수 있다. MCP는 이 경계를 외부 시스템으로 확장하는 역할을 한다.

Skills는 Copilot이 "무엇을 알고 있는가"를 결정한다. 보고서의 형식 규칙, 코드 리뷰의 기준, 특정 프로젝트의 명명 규칙 등은 도구가 아니라 지식이다. SKILL.md에 이런 절차와 규칙을 기술하면, Copilot은 해당 도메인의 전문가처럼 행동한다. 도구 없이 지식만 있으면 "무엇을 해야 하는지 알지만 실행할 수 없는" 상태가 된다.

두 요소를 결합하면 "도메인 지식을 갖춘 채로 외부 시스템을 조작할 수 있는 에이전트"가 탄생한다. 이것이 MCP + Skills 통합의 핵심 가치이다. 비유하자면, MCP는 에이전트의 손과 발이고 Skills는 에이전트의 두뇌이다. 손발만 있으면 방향 없이 움직이고, 두뇌만 있으면 생각만 하고 행동하지 못한다.

**표 11.1** MCP와 Skills의 특성 비교

| 구분 | MCP | Skills |
|------|-----|--------|
| 제공하는 것 | 도구(외부 시스템 접근) | 지식(절차, 규칙, 형식) |
| 정의 위치 | `.vscode/mcp.json` + 외부 서버 | `.github/skills/*/SKILL.md` |
| 없을 때의 한계 | IDE 내부 작업만 가능하다 | 범용적 응답만 생성한다 |
| 개선 방법 | 새 MCP 서버를 추가한다 | SKILL.md의 규칙을 정교화한다 |
| 변경 빈도 | 인프라 변경 시 (드물다) | 요구사항 변경 시 (자주) |

#### 처리 흐름에서의 이중 개입

다음 다이어그램은 사용자의 요청이 처리되는 과정에서 Skills와 MCP가 어떻게 역할을 분담하는지 보여 준다.

```
사용자 요청
    |
    v
[Skills: 어떻게 처리할지 절차 파악]  ← 1차 개입: 계획 수립
    |
    v
[MCP Tools: 외부 데이터 조회 / 작업 실행]
    |
    v
[Skills: 결과를 규칙에 맞게 가공]  ← 2차 개입: 품질 적용
    |
    v
최종 산출물
```

**그림 11.1** MCP와 Skills의 역할 분담 흐름

이 흐름에서 주목할 점은 Skills가 두 번 개입한다는 것이다. 첫 번째 개입에서 Skills는 사용자의 요청을 해석하고 어떤 데이터를 어떤 순서로 수집해야 하는지 절차를 결정한다. MCP 도구가 데이터를 가져온 뒤, 두 번째 개입에서 Skills는 그 데이터를 정해진 형식과 규칙에 맞게 가공한다. 도구 호출의 전후에 지식이 작용하는 이 구조가 단순한 도구 사용과 전문가 수준의 자동화를 구분하는 핵심이다.

#### 결합 없이 발생하는 문제

결합의 가치를 더 명확히 이해하기 위해, 한쪽만 존재할 때의 한계를 구체적으로 살펴본다.

MCP만 있고 Skills가 없는 경우를 생각해 보자. Copilot에게 "이번 주 주간 보고서를 작성해 줘"라고 요청하면, GitHub MCP를 통해 PR과 이슈 데이터를 가져올 수는 있다. 그러나 보고서의 형식, 우선순위 분류 기준, 작성 언어 등을 알지 못하므로 매번 다른 형태의 결과물이 나온다. 팀에서 정한 양식과 맞지 않아 수동으로 재정리해야 하는 상황이 반복된다.

반대로 Skills만 있고 MCP가 없는 경우에는 어떠한가. SKILL.md에 보고서 형식과 분류 규칙을 완벽하게 정의했더라도, 실제 GitHub 데이터에 접근할 수 없다. Copilot은 "이런 형식으로 보고서를 작성해야 합니다"라고 안내하거나, 가상의 데이터로 보고서 껍데기를 만들 뿐이다. 실제 데이터가 비어 있는 산출물은 실무에서 사용할 수 없다.

두 요소가 결합되어야 비로소 "실제 데이터로 채워진, 규칙에 맞는 보고서"가 만들어진다.

### 11.2 세 가지 설계 패턴

MCP + Skills 통합 프로젝트를 여러 번 수행하다 보면, 반복적으로 등장하는 구조적 패턴이 있다는 것을 발견하게 된다. 이 절에서는 복잡도 순으로 세 가지 대표적인 설계 패턴을 정리한다.

#### 패턴 1: 단일 도구(Single-Tool)

가장 단순한 형태이다. 하나의 MCP 도구를 호출하고, 하나의 스킬로 결과를 가공하는 구조이다. 입력과 출력이 1:1로 대응하며, 도구 호출 사이에 의존성이 없다.

```
[사용자 요청] → [MCP: 단일 도구 호출] → [Skills: 결과 가공] → [산출물]
```

**그림 11.2** 단일 도구 패턴의 흐름

이 패턴의 대표 사례는 "특정 저장소의 open 이슈 목록을 정리해 줘"와 같은 요청이다. GitHub MCP의 `list_issues` 도구 하나만 호출하고, 스킬에 정의된 형식(우선순위별 분류, 한국어 요약 등)으로 결과를 정리한다. 구현이 간단하고 디버깅이 용이하다는 것이 장점이다. 다만 여러 데이터 소스를 결합해야 하는 복잡한 작업에는 적합하지 않다.

단일 도구 패턴을 선택해야 하는 상황은 다음과 같다. 필요한 데이터가 하나의 API 호출로 충분할 때, 데이터 가공의 복잡도가 낮을 때, 에이전트 개발의 초기 프로토타입 단계에서 빠르게 동작을 검증하고 싶을 때이다.

#### 패턴 2: 파이프라인(Pipeline)

여러 MCP 도구를 순차적으로 호출하고, 각 단계의 출력이 다음 단계의 입력이 되는 구조이다. Skills는 전체 파이프라인의 순서와 각 단계의 처리 규칙을 정의한다.

```
[MCP: 도구 A] → [Skills: 중간 가공] → [MCP: 도구 B] → [Skills: 최종 가공] → [산출물]
```

**그림 11.3** 파이프라인 패턴의 흐름

2회차에서 구현할 주간 보고서 에이전트가 이 패턴의 전형적인 사례이다. 먼저 GitHub MCP로 병합된 PR 목록을 조회하고(도구 A), 그 결과에서 각 PR의 상세 정보를 추가 조회하며(도구 B), 미해결 이슈를 별도로 조회한 뒤(도구 C), 스킬에 정의된 보고서 구조와 분류 규칙에 따라 최종 보고서를 생성한다. 시장 조사 보고서, 경쟁사 분석, 학술 문헌 요약 등도 동일한 구조로 구현할 수 있다.

파이프라인 패턴에서 가장 중요한 것은 Skills에 "무엇을 어떤 순서로 수집할 것인가"와 "각 단계에서 어떻게 가공할 것인가"를 명확하게 분리하여 기술하는 것이다. 순서가 모호하면 에이전트가 불필요한 도구를 호출하거나 필요한 도구를 누락할 수 있다.

파이프라인 패턴의 주의점은 단계 간 의존성 관리이다. 도구 A의 결과가 도구 B의 입력으로 사용되므로, 도구 A가 실패하면 전체 파이프라인이 중단된다. 각 단계에서 실패 시 어떻게 처리할지(재시도, 기본값 사용, 사용자에게 알림 등)를 스킬에 미리 정의해 두는 것이 안정적인 운영의 핵심이다.

#### 패턴 3: 감독자(Supervisor)

가장 복잡한 형태이다. 스킬이 "감독자" 역할을 수행하며, 상황에 따라 어떤 MCP 도구를 호출할지 동적으로 결정하는 구조이다. 단순히 정해진 순서대로 도구를 호출하는 것이 아니라, 중간 결과를 평가하고 분기하며, 필요에 따라 이전 단계로 되돌아가는 자기 교정(Self-Correction) 루프를 포함한다.

```
[Skills: 상황 판단] → [MCP: 도구 선택 + 호출]
        ^                      |
        |                      v
        +---[Skills: 결과 평가 + 분기 결정]
                               |
                               v
                          [산출물 또는 재시도]
```

**그림 11.4** 감독자 패턴의 흐름

감독자 패턴의 사례를 두 가지 살펴본다.

첫 번째 사례는 이슈 자동 분류 에이전트이다. GitHub MCP로 새로운 이슈가 등록되었는지 감시하고, 스킬에 정의된 분류 규칙에 따라 라벨을 자동으로 부착한다. 긴급 이슈의 경우 담당자에게 알림을 보내고, 일반 이슈는 백로그에 추가한다. 여기서 핵심은 "긴급 이슈란 무엇인가"를 스킬에 명확하게 정의하는 것이다. 기준이 모호하면 에이전트의 판단이 불안정해진다.

두 번째 사례는 코드 생성-검증-수정 에이전트이다. 에이전트가 코드를 생성한 뒤, 스킬에 정의된 품질 기준에 따라 스스로 검증하고, 기준에 미달하는 부분을 자동으로 수정하는 자기 교정 루프이다. 코드를 생성한 뒤 린트(Lint) 검사를 실행하여 스타일 위반을 수정하거나, 문서를 작성한 뒤 용어 일관성 규칙에 따라 재검토하는 것이 이 패턴의 예시이다.

**표 11.2** MCP + Skills 설계 패턴 요약

| 패턴 | 복잡도 | MCP 역할 | Skills 역할 | 대표 사례 |
|------|--------|---------|------------|----------|
| 단일 도구 | 낮음 | 하나의 도구 호출 | 결과 가공 | 이슈 목록 정리 |
| 파이프라인 | 중간 | 순차적 다중 도구 호출 | 순서 정의, 단계별 가공 | 주간 보고서, 시장 분석 |
| 감독자 | 높음 | 조건부 도구 호출 | 판단 기준, 분기 결정, 품질 검증 | 이슈 자동 분류, 코드 자동 수정 |

세 가지 패턴은 상호 배타적이지 않다. 하나의 에이전트가 여러 패턴을 조합하여 동작하는 것이 일반적이다. 주간 보고서 에이전트에 검증 단계를 추가하면 파이프라인과 감독자가 결합된 형태가 된다. 패턴을 인식하면 새로운 프로젝트를 설계할 때 기존 구조를 재활용할 수 있으므로, 개발 속도와 안정성이 모두 향상된다.

#### 패턴 선택 가이드

어떤 패턴을 선택할지 결정할 때는 다음 세 가지 질문을 던진다.

1. 필요한 MCP 도구가 몇 개인가? 하나라면 단일 도구 패턴을 선택한다.
2. 도구 호출 사이에 의존성이 있는가? 이전 도구의 결과가 다음 도구의 입력이 된다면 파이프라인 패턴이 적합하다.
3. 중간 결과에 따라 분기가 필요한가? 조건부 판단이나 자기 교정 루프가 필요하다면 감독자 패턴을 적용한다.

프로토타입 단계에서는 항상 가장 단순한 패턴으로 시작하고, 요구사항이 명확해지면 점진적으로 복잡한 패턴으로 확장하는 것이 실무에서의 올바른 접근 방식이다.

### 11.3 디버깅과 트레이싱

에이전트가 기대와 다르게 동작할 때, 문제의 원인을 빠르게 찾아내는 능력은 통합 프로젝트의 성공을 결정짓는 핵심 역량이다. 이 절에서는 MCP + Skills 에이전트의 디버깅 방법을 체계적으로 다룬다.

#### VS Code Output 패널 활용

가장 먼저 확인해야 할 곳은 VS Code의 출력(Output) 패널이다. 상단 메뉴에서 "View > Output"을 선택한 뒤, 드롭다운에서 "MCP" 채널을 선택하면 MCP 서버와의 통신 로그를 실시간으로 확인할 수 있다. 이 로그에는 어떤 도구가 어떤 인자로 호출되었는지, 서버가 어떤 응답을 반환했는지가 모두 기록된다. 에이전트가 잘못된 데이터를 사용하고 있다면, 이 로그에서 도구 호출의 입력과 출력을 비교하여 원인을 추적할 수 있다.

Output 패널에서 확인할 수 있는 정보는 다음과 같다.

- **요청 로그**: 도구 이름, 전달된 매개변수, 호출 시각
- **응답 로그**: 반환된 데이터, 응답 시간, 오류 메시지(있을 경우)
- **서버 상태**: MCP 서버의 시작/종료, 연결 상태 변화

#### 분리 테스트 전략

에이전트의 동작이 복잡해질수록 문제를 분리하여 테스트하는 것이 효과적이다. 이 전략은 세 단계로 구성된다.

첫째, MCP 도구를 단독으로 테스트한다. Copilot Chat에서 "이 저장소의 지난 7일간 merged PR 목록을 보여 줘"처럼 도구 호출만 유발하는 간단한 요청을 보내 데이터가 올바르게 반환되는지 확인한다.

둘째, 스킬의 규칙을 단독으로 테스트한다. 이미 수집된 데이터를 직접 붙여 넣고 "이 데이터를 주간 보고서 형식으로 정리해 줘"라고 요청하여 스킬의 형식 규칙이 제대로 적용되는지 검증한다.

셋째, 두 가지가 각각 정상이라면 통합하여 다시 실행한다. 이 분리 테스트 방식은 문제의 원인이 MCP 쪽인지 Skills 쪽인지를 빠르게 특정할 수 있게 해 준다.

#### 디버깅 체크리스트

**표 11.3** 에이전트 디버깅 체크리스트

| 증상 | 확인 위치 | 대응 방법 |
|------|----------|----------|
| 도구가 호출되지 않는다 | Copilot Chat 도구 아이콘 | MCP 서버 등록과 도구 설명(description)을 점검한다 |
| 잘못된 도구가 호출된다 | Output 패널 > MCP 채널 | 도구 이름과 설명을 더 명확하게 수정한다 |
| 데이터는 맞지만 형식이 틀리다 | 생성된 산출물 | SKILL.md의 규칙을 더 구체적으로 보강한다 |
| 스킬이 활성화되지 않는다 | SKILL.md의 title/description | 사용자 프롬프트와 매칭되는 키워드를 추가한다 |
| 도구 호출이 실패한다 | Output 패널 > MCP 채널 | 환경 변수, 인증 토큰, 네트워크 상태를 확인한다 |
| 파이프라인 중간에 멈춘다 | Output 패널 로그 타임스탬프 | 실패한 단계를 식별하고 해당 도구만 격리 테스트한다 |
| 감독자 루프가 무한 반복한다 | Chat 패널 응답 길이 | 스킬의 종료 조건을 더 명확하게 정의한다 |

#### 도구 미호출 문제 해결

도구가 전혀 호출되지 않는 경우가 가장 흔한 문제이다. 이 경우 Copilot Chat 패널의 도구 아이콘을 클릭하여 사용 가능한 도구 목록을 확인한다. MCP 서버가 정상적으로 시작되었는지, 도구가 올바르게 노출되고 있는지를 점검한다. 도구 목록에는 나타나지만 호출되지 않는 경우에는 도구의 설명(description)이 사용자 요청과 의미적으로 연결되지 않을 가능성이 높다. MCP 서버의 도구 설명을 수정하여 Copilot이 적절한 맥락에서 해당 도구를 선택하도록 유도한다.

#### 스킬 비활성화 문제 해결

스킬이 활성화되지 않는 문제도 빈번하게 발생한다. Copilot은 사용자의 프롬프트와 SKILL.md의 `title`, `description` 필드를 비교하여 관련 스킬을 자동으로 활성화한다. 프롬프트에 "주간 보고서"라는 표현을 사용했는데 스킬의 title이 "weekly summary"로만 되어 있다면 매칭에 실패할 수 있다. 해결 방법은 두 가지이다. 스킬의 메타데이터에 다양한 표현(한국어, 영어, 약어 등)을 포함하거나, 프롬프트에서 스킬 이름을 직접 언급하는 것이다.

#### 메모리 MCP를 활용한 컨텍스트 유지

세션 간 컨텍스트 유지가 필요한 프로젝트에서는 메모리 MCP(Memory MCP) 서버를 활용할 수 있다. 이 서버는 에이전트가 학습한 정보를 지속적으로 저장하고, 다음 세션에서 이를 불러와 활용하도록 한다. 예를 들어 첫 번째 보고서 생성 시 팀원의 담당 영역을 파악했다면, 이 정보를 메모리에 저장하여 이후 보고서에서 자동으로 반영할 수 있다. 이를 통해 에이전트가 반복 사용될수록 결과의 품질이 향상되는 학습 효과를 기대할 수 있다.

## 2회차: 실습

### 실습 1: 주간 보고서 자동 생성 에이전트 구축

이 실습에서는 MCP와 Skills를 결합한 실전 프로젝트를 단계별로 구현한다. 목표는 GitHub 저장소의 한 주간 활동을 자동으로 수집하여 정해진 양식의 주간 보고서를 생성하는 에이전트를 만드는 것이다.

#### Step 1: MCP 서버 구성

에이전트가 외부 데이터에 접근하려면 먼저 MCP 서버를 등록해야 한다. 이 프로젝트에서는 두 가지 MCP 서버를 사용한다. 첫째는 GitHub MCP 서버로, 이슈와 풀 리퀘스트(Pull Request), 커밋 이력을 조회하는 데 사용한다. 둘째는 파일 시스템(Filesystem) MCP 서버로, 프로젝트 문서와 보고서 템플릿을 읽는 데 사용한다.

`.vscode/mcp.json` 파일을 다음과 같이 구성한다.

```json
{
  "servers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem",
               "${workspaceFolder}"]
    }
  }
}
```

GitHub MCP 서버는 `GITHUB_TOKEN` 환경 변수를 필요로 한다. 이 토큰은 GitHub 설정의 Developer Settings에서 Personal Access Token(Fine-grained)을 생성하여 `.env` 파일에 저장한다. 토큰에는 저장소의 이슈와 풀 리퀘스트 읽기 권한이 포함되어야 한다.

MCP 서버가 올바르게 등록되었는지 확인하려면 Copilot Chat 패널의 도구 아이콘을 클릭한다. 사용 가능한 도구 목록에 GitHub 관련 도구(`list_issues`, `list_pull_requests`, `get_file_contents` 등)와 파일 시스템 관련 도구(`read_file`, `list_directory` 등)가 나타나면 정상이다.

#### Step 2: 보고서 작성 스킬 만들기

MCP 서버가 데이터를 가져올 수 있는 "손"이라면, 이제 그 데이터를 어떻게 정리할지 알려주는 "두뇌"를 만들 차례이다. `.github/skills/weekly-report/` 폴더를 생성하고 그 안에 `SKILL.md` 파일을 작성한다.

```markdown
---
title: 주간 보고서 작성
description: GitHub 활동을 기반으로 주간 보고서를 생성한다
---
# 주간 보고서 작성 스킬

## 데이터 수집 절차
1. GitHub MCP의 list_pull_requests로 이번 주 merged PR을 조회한다
2. GitHub MCP의 list_pull_requests로 현재 open PR을 조회한다
3. GitHub MCP의 list_issues로 신규 및 미해결 이슈를 조회한다
4. 각 PR/이슈의 라벨을 확인하여 우선순위를 분류한다

## 보고서 구조
1. 기간 (이번 주 월요일~일요일)
2. 완료된 작업 (merged PR 기준)
3. 진행 중인 작업 (open PR 기준)
4. 신규/미해결 이슈 (라벨별 분류)
5. 다음 주 계획 (milestone 기반)

## 우선순위 분류 규칙
- critical/bug 라벨 → 긴급
- feature 라벨 → 일반
- docs/chore 라벨 → 낮음

## 작성 규칙
- 한국어로 작성한다
- 각 PR/이슈는 한 줄 요약으로 기술한다
- 날짜 형식: YYYY-MM-DD
- PR 번호를 반드시 포함한다 (예: #42)
- 담당자 이름을 괄호 안에 표기한다
```

이 SKILL.md에는 네 가지 핵심 요소가 담겨 있다. 첫째는 데이터 수집 절차이다. 어떤 MCP 도구를 어떤 순서로 호출할지 명시한다. 이 절차가 있으면 에이전트가 불필요한 도구를 호출하거나 순서를 잘못 잡는 것을 방지할 수 있다. 둘째는 보고서의 구조이다. 어떤 섹션이 어떤 순서로 나와야 하는지 명확하게 정의한다. 셋째는 우선순위 분류 규칙이다. GitHub 이슈의 라벨을 보고 어떤 기준으로 분류할지 판단하는 도메인 지식이다. 넷째는 작성 규칙이다. 언어, 요약 방식, 날짜 형식 등 산출물의 품질을 결정하는 세부 사항이다.

필요에 따라 `assets/report-template.md` 파일에 보고서의 마크다운 템플릿을 별도로 작성하고, SKILL.md에서 이를 참조하도록 구성할 수도 있다. 템플릿을 분리하면 보고서 형식 변경 시 SKILL.md를 수정하지 않고 템플릿 파일만 교체하면 되므로 유지보수가 용이하다.

#### Step 3: 에이전트 실행

MCP 서버와 Skills가 모두 준비되었으면 Copilot Agent Mode에서 에이전트를 실행한다. Copilot Chat을 Agent 모드로 전환하고 다음과 같이 입력한다.

```
이번 주 주간 보고서를 작성해 줘.
대상 저장소는 owner/repo-name이다.
```

이 요청을 받은 Copilot은 다음과 같은 과정을 자율적으로 수행한다.

1. "주간 보고서"라는 키워드에 매칭되는 `weekly-report` 스킬을 활성화한다
2. 스킬의 데이터 수집 절차를 파악한다
3. GitHub MCP 서버의 도구를 순서대로 호출한다
   - 이번 주에 병합된 풀 리퀘스트 조회
   - 현재 열려 있는 풀 리퀘스트 확인
   - 신규 및 미해결 이슈를 라벨별로 조회
4. 스킬에 정의된 보고서 구조와 분류 규칙에 따라 보고서를 생성한다

이 전체 과정에서 사용자가 개입할 필요는 거의 없다. MCP 도구 호출 시 터미널 명령이 아닌 서버 통신이 이루어지므로, 대부분의 단계가 자동으로 진행된다. 다만 Copilot이 도구 호출 승인을 요청하는 경우에는 호출 내용을 확인한 뒤 승인한다.

#### Step 4: 결과 검증과 반복 개선

에이전트가 생성한 보고서를 검토할 때는 세 가지 관점에서 평가한다.

첫째, 데이터의 정확성이다. 보고서에 나열된 PR과 이슈가 실제 GitHub 저장소의 내용과 일치하는지 확인한다. 누락되거나 잘못된 항목이 있다면 MCP 도구 호출에 문제가 있었을 가능성이 높다. Output 패널에서 도구 호출 로그를 확인한다.

둘째, 형식의 준수 여부이다. SKILL.md에 정의한 보고서 구조, 우선순위 분류, 작성 규칙이 제대로 반영되었는지 검토한다. 형식이 기대와 다르다면 스킬의 지시문이 불충분하거나 모호한 것이므로, SKILL.md를 수정하여 지시를 더 구체적으로 만든다.

셋째, 활용 가능성이다. 생성된 보고서가 실제로 팀 회의나 상급자 보고에 바로 사용할 수 있는 수준인지 판단한다. 부족한 부분이 있다면 스킬에 추가 규칙을 넣거나 보고서 템플릿을 개선한다.

이 반복적 개선 과정이 MCP + Skills 통합 프로젝트의 본질이다. 한 번에 완벽한 에이전트를 만드는 것이 아니라, 실행 결과를 보고 스킬을 점진적으로 정교화하는 것이 실무에서의 올바른 접근 방식이다.

### 실습 2: 나만의 에이전트 설계

이 실습에서는 수강생 각자가 자신만의 MCP + Skills 통합 에이전트를 설계한다. 실습 1에서 경험한 파이프라인 패턴을 기반으로, 다른 설계 패턴을 적용해 보는 것이 목표이다.

#### 제안 프로젝트 3가지

첫 번째 제안은 코드 품질 감시 에이전트이다. GitHub MCP 서버로 새로 생성된 풀 리퀘스트를 감지하고, 코드 리뷰 스킬에 정의된 기준(함수 길이, 타입 힌트 유무, 테스트 포함 여부 등)에 따라 자동 리뷰 코멘트를 작성한다. 감독자 패턴에 해당한다.

두 번째 제안은 문서 동기화 에이전트이다. 파일 시스템 MCP 서버로 소스 코드 디렉토리의 변경을 감지하고, 문서 업데이트 스킬에 정의된 규칙에 따라 관련 문서를 자동으로 갱신한다. 함수 시그니처가 변경되면 API 문서를 업데이트하고, 새 모듈이 추가되면 목차를 갱신하는 식이다. 파이프라인 패턴에 해당한다.

세 번째 제안은 학습 도우미 에이전트이다. 웹 검색 MCP 서버로 특정 기술 주제에 관한 최신 자료를 수집하고, 학습 가이드 스킬에 정의된 교육 방법론(개념 설명 -> 예시 코드 -> 연습 문제 순서)에 따라 맞춤형 학습 자료를 생성한다. 파이프라인 패턴의 변형이다.

수강생은 이 중 하나를 선택하거나 자유 주제로 설계할 수 있다.

#### 설계서 작성 양식

**표 11.4** 에이전트 설계서 작성 양식

| 항목 | 작성 내용 |
|------|----------|
| 에이전트 이름 | 에이전트의 용도를 나타내는 명칭 |
| 해결하는 문제 | 수동으로 수행할 때의 불편함이나 비효율 |
| MCP 서버 목록 | 서버 이름, 용도, 사용할 주요 도구 |
| SKILL.md 핵심 규칙 | 판단 기준, 출력 형식, 품질 기준 |
| 설계 패턴 | 단일 도구 / 파이프라인 / 감독자 |
| 예상 한계 | 에이전트가 처리하지 못하는 상황과 대응 방안 |

이 양식에 따라 설계서를 작성하면, 에이전트의 구조를 체계적으로 정리할 수 있고 동료에게 설계 의도를 명확하게 전달할 수 있다. 설계서 작성이 완료되면 MCP 서버 설정(`mcp.json`)과 SKILL.md 파일을 실제로 작성하여 프로토타입을 구현한다. 시간이 허락하면 에이전트를 실행하여 결과를 확인한다.

3. **보고서 결과물**: 에이전트가 생성한 주간 보고서 마크다운 파일
4. **디버깅 로그**: Output 패널에서 캡처한 MCP 통신 로그 스크린샷 (최소 2장)
5. **에이전트 설계서**: 실습 2에서 작성한 나만의 에이전트 설계서 (표 11.4 양식)

평가 기준:
- MCP 서버 연결과 도구 호출의 정상 동작 여부 (30%)
- SKILL.md의 규칙 정의 구체성과 명확성 (25%)
- 보고서 결과물의 형식 준수 여부와 데이터 정확성 (20%)
- 나만의 에이전트 설계서의 완성도 (15%)
- 디버깅 과정의 체계성 (10%)

## 핵심 정리

- MCP는 "능력(도구)"을, Skills는 "지식(절차)"을 제공한다. 두 요소를 결합하면 도메인 전문가 수준의 자동화가 가능하다.
- Skills는 도구 호출 전후에 두 번 개입한다. 첫째 절차를 결정하고, 둘째 결과를 규칙에 맞게 가공한다.
- 세 가지 설계 패턴(단일 도구, 파이프라인, 감독자)은 복잡도 순으로 정렬되며, 프로토타입은 항상 가장 단순한 패턴에서 시작한다.
- 디버깅의 핵심 도구는 VS Code Output 패널의 MCP 채널이다. 분리 테스트 전략(MCP 단독 -> Skills 단독 -> 통합)으로 문제의 원인을 빠르게 특정한다.
- 스킬의 품질이 에이전트 출력의 품질을 결정한다. 스킬을 점진적으로 정교화하는 것이 올바른 접근이다.

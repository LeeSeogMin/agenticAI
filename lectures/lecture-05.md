# 제5강. MCP + Skills 통합 프로젝트

## 학습목표

- MCP(도구 접근)와 Skills(지식/절차)의 결합이 만드는 시너지를 이해한다
- 두 요소를 조합하여 실용적인 자동화 에이전트를 설계하고 구현한다
- 에이전트 동작의 전 과정을 추적하고 디버깅한다

## 선수지식

제2강에서 다룬 MCP 서버 연결 방법과 제4강에서 다룬 Skills 작성 방법을 숙지하고 있어야 한다. `.vscode/mcp.json`으로 MCP 서버를 등록하는 과정과 `SKILL.md` 파일의 구조를 직접 실습한 경험이 있으면 이 강의의 통합 프로젝트를 원활하게 수행할 수 있다.

---

## 5.1 MCP와 Skills의 역할 분담

이전 강의에서 MCP와 Skills를 각각 독립적으로 학습했다. 이 절에서는 두 요소가 결합했을 때 나타나는 시너지를 구조적으로 이해한다.

MCP는 Copilot이 "무엇을 할 수 있는가"를 결정한다. GitHub API에서 이슈를 조회하거나, 파일 시스템에서 문서를 읽거나, 데이터베이스에 쿼리를 실행하는 것은 모두 MCP 서버가 제공하는 도구(Tool)를 통해 이루어진다. MCP가 없으면 Copilot은 IDE 내부의 파일 편집과 터미널 명령만 수행할 수 있다. MCP는 이 경계를 외부 시스템으로 확장하는 역할을 한다.

Skills는 Copilot이 "무엇을 알고 있는가"를 결정한다. 보고서의 형식 규칙, 코드 리뷰의 기준, 특정 프로젝트의 명명 규칙 등은 도구가 아니라 지식이다. SKILL.md에 이런 절차와 규칙을 기술하면, Copilot은 해당 도메인의 전문가처럼 행동한다. 도구 없이 지식만 있으면 "무엇을 해야 하는지 알지만 실행할 수 없는" 상태가 된다.

두 요소를 결합하면 "도메인 지식을 갖춘 채로 외부 시스템을 조작할 수 있는 에이전트"가 탄생한다. 이것이 MCP + Skills 통합의 핵심 가치이다. 비유하자면, MCP는 에이전트의 손과 발이고 Skills는 에이전트의 두뇌이다. 손발만 있으면 방향 없이 움직이고, 두뇌만 있으면 생각만 하고 행동하지 못한다.

이 관계를 표로 정리하면 다음과 같다.

**표 5.0** MCP와 Skills의 특성 비교

| 구분 | MCP | Skills |
|------|-----|--------|
| 제공하는 것 | 도구(외부 시스템 접근) | 지식(절차, 규칙, 형식) |
| 정의 위치 | `.vscode/mcp.json` + 외부 서버 | `.github/skills/*/SKILL.md` |
| 없을 때의 한계 | IDE 내부 작업만 가능하다 | 범용적 응답만 생성한다 |
| 개선 방법 | 새 MCP 서버를 추가한다 | SKILL.md의 규칙을 정교화한다 |

다음 다이어그램은 사용자의 요청이 처리되는 과정에서 Skills와 MCP가 어떻게 역할을 분담하는지 보여 준다.

```
사용자 요청
    │
    ▼
[Skills: 어떻게 처리할지 절차 파악]
    │
    ▼
[MCP Tools: 외부 데이터 조회 / 작업 실행]
    │
    ▼
[Skills: 결과를 규칙에 맞게 가공]
    │
    ▼
최종 산출물
```

**그림 5.1** MCP와 Skills의 역할 분담 흐름

이 흐름에서 주목할 점은 Skills가 두 번 개입한다는 것이다. 첫 번째 개입에서 Skills는 사용자의 요청을 해석하고 어떤 데이터를 어떤 순서로 수집해야 하는지 절차를 결정한다. MCP 도구가 데이터를 가져온 뒤, 두 번째 개입에서 Skills는 그 데이터를 정해진 형식과 규칙에 맞게 가공한다. 도구 호출의 전후에 지식이 작용하는 이 구조가 단순한 도구 사용과 전문가 수준의 자동화를 구분하는 핵심이다.

## 5.2 프로젝트: 주간 보고서 자동 생성 에이전트

이 절에서는 MCP와 Skills를 결합한 실전 프로젝트를 단계별로 구현한다. 목표는 GitHub 저장소의 한 주간 활동을 자동으로 수집하여 정해진 양식의 주간 보고서를 생성하는 에이전트를 만드는 것이다.

### Step 1: MCP 서버 구성

에이전트가 외부 데이터에 접근하려면 먼저 MCP 서버를 등록해야 한다. 이 프로젝트에서는 두 가지 MCP 서버를 사용한다. 첫째는 GitHub MCP 서버로, 이슈와 풀 리퀘스트(Pull Request), 커밋 이력을 조회하는 데 사용한다. 둘째는 파일 시스템(Filesystem) MCP 서버로, 프로젝트 문서와 보고서 템플릿을 읽는 데 사용한다.

`.vscode/mcp.json` 파일을 다음과 같이 구성한다.

```json
{
  "servers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem",
               "${workspaceFolder}"]
    }
  }
}
```

GitHub MCP 서버는 `GITHUB_TOKEN` 환경 변수를 필요로 한다. 이 토큰은 GitHub 설정의 Developer Settings에서 Personal Access Token(Fine-grained)을 생성하여 `.env` 파일에 저장한다. 토큰에는 저장소의 이슈와 풀 리퀘스트 읽기 권한이 포함되어야 한다.

MCP 서버가 올바르게 등록되었는지 확인하려면 Copilot Chat 패널의 도구 아이콘을 클릭한다. 사용 가능한 도구 목록에 GitHub 관련 도구(`list_issues`, `list_pull_requests`, `get_file_contents` 등)와 파일 시스템 관련 도구(`read_file`, `list_directory` 등)가 나타나면 정상이다.

### Step 2: 보고서 작성 스킬 만들기

MCP 서버가 데이터를 가져올 수 있는 "손"이라면, 이제 그 데이터를 어떻게 정리할지 알려주는 "두뇌"를 만들 차례이다. `.github/skills/weekly-report/` 폴더를 생성하고 그 안에 `SKILL.md` 파일을 작성한다.

```markdown
---
title: 주간 보고서 작성
description: GitHub 활동을 기반으로 주간 보고서를 생성한다
---
# 주간 보고서 작성 스킬

## 보고서 구조
1. 기간 (이번 주 월요일~일요일)
2. 완료된 작업 (merged PR 기준)
3. 진행 중인 작업 (open PR 기준)
4. 신규/미해결 이슈 (라벨별 분류)
5. 다음 주 계획 (milestone 기반)

## 우선순위 분류 규칙
- critical/bug 라벨 → 긴급
- feature 라벨 → 일반
- docs/chore 라벨 → 낮음

## 작성 규칙
- 한국어로 작성한다
- 각 PR/이슈는 한 줄 요약으로 기술한다
- 날짜 형식: YYYY-MM-DD
```

이 SKILL.md에는 세 가지 핵심 요소가 담겨 있다. 첫째는 보고서의 구조이다. 어떤 섹션이 어떤 순서로 나와야 하는지 명확하게 정의한다. Copilot은 이 구조를 따라 보고서의 골격을 만든다. 둘째는 우선순위 분류 규칙이다. GitHub 이슈의 라벨을 보고 어떤 기준으로 분류할지 판단하는 도메인 지식이다. 이 규칙이 없으면 Copilot은 이슈를 나열만 할 뿐, 중요도에 따른 분류를 수행하지 못한다. 셋째는 작성 규칙이다. 언어, 요약 방식, 날짜 형식 등 산출물의 품질을 결정하는 세부 사항이다.

필요에 따라 `assets/report-template.md` 파일에 보고서의 마크다운 템플릿을 별도로 작성하고, SKILL.md에서 이를 참조하도록 구성할 수도 있다. 템플릿을 분리하면 보고서 형식 변경 시 SKILL.md를 수정하지 않고 템플릿 파일만 교체하면 되므로 유지보수가 용이하다.

### Step 3: 에이전트 실행

MCP 서버와 Skills가 모두 준비되었으면 Copilot Agent Mode에서 에이전트를 실행한다. Copilot Chat을 Agent 모드로 전환하고 다음과 같이 입력한다.

```
이번 주 주간 보고서를 작성해 줘.
대상 저장소는 owner/repo-name이다.
```

이 요청을 받은 Copilot은 다음과 같은 과정을 자율적으로 수행한다. 먼저 "주간 보고서"라는 키워드에 매칭되는 `weekly-report` 스킬을 활성화한다. 스킬의 보고서 구조를 파악한 뒤, 필요한 데이터를 수집하기 위해 GitHub MCP 서버의 도구를 호출한다. 이번 주에 병합된 풀 리퀘스트를 조회하고, 현재 열려 있는 풀 리퀘스트를 확인하며, 신규 및 미해결 이슈를 라벨별로 가져온다. 데이터 수집이 완료되면 스킬에 정의된 우선순위 분류 규칙과 작성 규칙에 따라 보고서를 생성한다.

이 전체 과정에서 사용자가 개입할 필요는 거의 없다. MCP 도구 호출 시 터미널 명령이 아닌 서버 통신이 이루어지므로, 대부분의 단계가 자동으로 진행된다. 다만 Copilot이 도구 호출 승인을 요청하는 경우에는 호출 내용을 확인한 뒤 승인한다.

에이전트가 생성한 보고서는 마크다운 형식으로 Chat 패널에 출력되며, 필요에 따라 파일로 저장할 수 있다. 보고서의 각 항목이 실제 GitHub 데이터와 일치하는지, 우선순위 분류가 라벨 기준에 맞게 이루어졌는지, 작성 규칙이 준수되었는지를 확인하는 것이 이 단계의 핵심이다.

여기서 MCP와 Skills의 시너지가 분명하게 드러난다. MCP만 있고 Skills가 없었다면, Copilot은 GitHub 데이터를 가져올 수는 있지만 어떤 형식으로 정리해야 하는지, 어떤 라벨이 긴급을 의미하는지 알지 못한다. 반대로 Skills만 있고 MCP가 없었다면, 보고서의 형식은 완벽하지만 실제 데이터가 비어 있는 껍데기만 생성된다. 두 요소가 결합되어야 비로소 "실제 데이터로 채워진, 규칙에 맞는 보고서"가 만들어진다.

### Step 4: 결과 검토와 디버깅

에이전트가 생성한 보고서를 검토할 때는 세 가지 관점에서 평가한다.

첫째, 데이터의 정확성이다. 보고서에 나열된 PR과 이슈가 실제 GitHub 저장소의 내용과 일치하는지 확인한다. 누락되거나 잘못된 항목이 있다면 MCP 도구 호출에 문제가 있었을 가능성이 높다.

둘째, 형식의 준수 여부이다. SKILL.md에 정의한 보고서 구조, 우선순위 분류, 작성 규칙이 제대로 반영되었는지 검토한다. 형식이 기대와 다르다면 스킬의 지시문이 불충분하거나 모호한 것이므로, SKILL.md를 수정하여 지시를 더 구체적으로 만든다.

셋째, 활용 가능성이다. 생성된 보고서가 실제로 팀 회의나 상급자 보고에 바로 사용할 수 있는 수준인지 판단한다. 부족한 부분이 있다면 스킬에 추가 규칙을 넣거나 보고서 템플릿을 개선한다.

이 반복적 개선 과정이 MCP + Skills 통합 프로젝트의 본질이다. 한 번에 완벽한 에이전트를 만드는 것이 아니라, 실행 결과를 보고 스킬을 점진적으로 정교화하는 것이 실무에서의 올바른 접근 방식이다.

## 5.3 디버깅과 추적

에이전트가 기대와 다르게 동작할 때, 문제의 원인을 빠르게 찾아내는 능력은 통합 프로젝트의 성공을 결정짓는 핵심 역량이다. 이 절에서는 MCP + Skills 에이전트의 디버깅 방법을 체계적으로 다룬다.

가장 먼저 확인해야 할 곳은 VS Code의 출력(Output) 패널이다. 상단 메뉴에서 "View > Output"을 선택한 뒤, 드롭다운에서 "MCP" 채널을 선택하면 MCP 서버와의 통신 로그를 실시간으로 확인할 수 있다. 이 로그에는 어떤 도구가 어떤 인자로 호출되었는지, 서버가 어떤 응답을 반환했는지가 모두 기록된다. 에이전트가 잘못된 데이터를 사용하고 있다면, 이 로그에서 도구 호출의 입력과 출력을 비교하여 원인을 추적할 수 있다.

**표 5.1** 에이전트 디버깅 체크리스트

| 증상 | 확인 위치 | 대응 방법 |
|------|----------|----------|
| 도구가 호출되지 않는다 | Copilot Chat 도구 아이콘 | MCP 서버 등록과 도구 설명(description)을 점검한다 |
| 잘못된 도구가 호출된다 | Output 패널 > MCP 채널 | 도구 이름과 설명을 더 명확하게 수정한다 |
| 데이터는 맞지만 형식이 틀리다 | 생성된 산출물 | SKILL.md의 규칙을 더 구체적으로 보강한다 |
| 스킬이 활성화되지 않는다 | SKILL.md의 title/description | 사용자 프롬프트와 매칭되는 키워드를 추가한다 |
| 도구 호출이 실패한다 | Output 패널 > MCP 채널 | 환경 변수, 인증 토큰, 네트워크 상태를 확인한다 |

도구가 전혀 호출되지 않는 경우가 가장 흔한 문제이다. 이 경우 Copilot Chat 패널의 도구 아이콘을 클릭하여 사용 가능한 도구 목록을 확인한다. MCP 서버가 정상적으로 시작되었는지, 도구가 올바르게 노출되고 있는지를 점검한다. 도구 목록에는 나타나지만 호출되지 않는 경우에는 도구의 설명(description)이 사용자 요청과 의미적으로 연결되지 않을 가능성이 높다. MCP 서버의 도구 설명을 수정하여 Copilot이 적절한 맥락에서 해당 도구를 선택하도록 유도한다.

스킬이 활성화되지 않는 문제도 빈번하게 발생한다. Copilot은 사용자의 프롬프트와 SKILL.md의 `title`, `description` 필드를 비교하여 관련 스킬을 자동으로 활성화한다. 프롬프트에 "주간 보고서"라는 표현을 사용했는데 스킬의 title이 "weekly summary"로만 되어 있다면 매칭에 실패할 수 있다. 스킬의 메타데이터에 다양한 표현을 포함하거나, 프롬프트에서 스킬 이름을 직접 언급하는 것이 확실한 해결책이다.

세션 간 컨텍스트 유지가 필요한 프로젝트에서는 메모리 MCP(Memory MCP) 서버를 활용할 수 있다. 이 서버는 에이전트가 학습한 정보를 지속적으로 저장하고, 다음 세션에서 이를 불러와 활용하도록 한다. 예를 들어 첫 번째 보고서 생성 시 팀원의 담당 영역을 파악했다면, 이 정보를 메모리에 저장하여 이후 보고서에서 자동으로 반영할 수 있다.

디버깅의 실전 요령을 하나 더 소개한다. 에이전트의 동작이 복잡해질수록 문제를 분리하여 테스트하는 것이 효과적이다. 먼저 MCP 도구만 단독으로 테스트한다. Copilot Chat에서 "이 저장소의 지난 7일간 merged PR 목록을 보여 줘"처럼 도구 호출만 유발하는 간단한 요청을 보내 데이터가 올바르게 반환되는지 확인한다. 도구가 정상이라면, 다음으로 스킬의 규칙을 테스트한다. 이미 수집된 데이터를 직접 붙여 넣고 "이 데이터를 주간 보고서 형식으로 정리해 줘"라고 요청하여 스킬의 형식 규칙이 제대로 적용되는지 검증한다. 두 가지가 각각 정상이라면 통합하여 다시 실행한다. 이 분리 테스트 방식은 문제의 원인이 MCP 쪽인지 Skills 쪽인지를 빠르게 특정할 수 있게 해 준다.

## 5.4 설계 패턴

MCP + Skills 통합 프로젝트를 여러 번 수행하다 보면, 반복적으로 등장하는 구조적 패턴이 있다는 것을 발견하게 된다. 이 절에서는 세 가지 대표적인 설계 패턴을 정리한다.

첫 번째 패턴은 조사-분석-보고(Research-Analyze-Report)이다. 이 패턴에서 MCP 도구는 외부 소스에서 데이터를 수집하고, Skills는 수집된 데이터를 분석 기준에 따라 정리하여 보고서를 작성한다. 앞서 구현한 주간 보고서 에이전트가 이 패턴의 전형적인 사례이다. 시장 조사 보고서, 경쟁사 분석, 학술 문헌 요약 등도 동일한 구조로 구현할 수 있다. 이 패턴의 핵심은 Skills에 "무엇을 수집할 것인가"와 "어떻게 분석할 것인가"를 명확하게 분리하여 기술하는 것이다.

```
[MCP: 데이터 수집] → [Skills: 분석 기준 적용] → [산출물: 보고서]
```

**그림 5.2** 조사-분석-보고 패턴의 흐름

두 번째 패턴은 감시-판단-실행(Monitor-Decide-Act)이다. 이 패턴에서 MCP 도구는 특정 시스템의 상태를 모니터링하고, Skills는 상태 변화에 대한 판단 기준을 제공하며, 판단 결과에 따라 MCP 도구가 후속 조치를 실행한다. 예를 들어 GitHub MCP로 새로운 이슈가 등록되었는지 감시하고, 스킬에 정의된 분류 규칙에 따라 라벨을 자동으로 부착하며, 긴급 이슈의 경우 담당자에게 알림을 보내는 에이전트가 이 패턴에 해당한다. 이 패턴에서 가장 중요한 것은 판단 기준의 명확성이다. "긴급 이슈란 무엇인가"를 모호하게 정의하면 에이전트의 판단이 불안정해진다.

세 번째 패턴은 생성-검증-수정(Generate-Verify-Fix)이다. 에이전트가 산출물을 생성한 뒤, 스킬에 정의된 품질 기준에 따라 스스로 검증하고, 기준에 미달하는 부분을 자동으로 수정하는 자기 교정(Self-Correction) 루프이다. 코드를 생성한 뒤 린트(Lint) 검사를 실행하여 스타일 위반을 수정하거나, 문서를 작성한 뒤 용어 일관성 규칙에 따라 재검토하는 것이 이 패턴의 예시이다. 이 패턴은 제1강에서 다룬 에이전트 모드의 자기 치유 루프와 동일한 원리이며, Skills가 품질 기준을 제공함으로써 그 루프의 정밀도를 높이는 구조이다.

**표 5.2** MCP + Skills 설계 패턴 요약

| 패턴 | MCP 역할 | Skills 역할 | 대표 사례 |
|------|---------|------------|----------|
| 조사-분석-보고 | 데이터 수집 | 분석 기준, 보고서 형식 | 주간 보고서, 시장 분석 |
| 감시-판단-실행 | 상태 모니터링, 후속 조치 | 판단 기준, 분류 규칙 | 이슈 자동 분류, 알림 |
| 생성-검증-수정 | 실행 환경 제공 | 품질 기준, 검증 규칙 | 코드 자동 수정, 문서 교정 |

세 가지 패턴은 상호 배타적이지 않다. 하나의 에이전트가 여러 패턴을 조합하여 동작하는 것이 일반적이다. 주간 보고서 에이전트에 검증 단계를 추가하면 조사-분석-보고와 생성-검증-수정이 결합된 형태가 된다. 패턴을 인식하면 새로운 프로젝트를 설계할 때 기존 구조를 재활용할 수 있으므로, 개발 속도와 안정성이 모두 향상된다.

## 5.5 도전 과제: 나만의 에이전트 설계

이 절에서는 수강생 각자가 자신만의 MCP + Skills 통합 에이전트를 설계한다. 기획부터 구현, 발표까지의 전 과정을 수행하며, 앞에서 학습한 설계 패턴을 실제로 적용하는 경험을 쌓는다.

에이전트를 설계할 때는 세 가지를 명확하게 정의해야 한다. 첫째, 어떤 MCP 서버를 사용하고 각 서버에서 어떤 도구를 활용할 것인가이다. 둘째, 어떤 스킬을 만들고 그 스킬에 어떤 지식과 규칙을 담을 것인가이다. 셋째, 에이전트가 최종적으로 어떤 산출물을 생성하며 그 품질을 어떻게 평가할 것인가이다.

아래는 세 가지 제안 프로젝트이다. 수강생은 이 중 하나를 선택하거나 자유 주제로 설계할 수 있다.

첫 번째 제안은 코드 품질 감시 에이전트이다. GitHub MCP 서버로 새로 생성된 풀 리퀘스트를 감지하고, 코드 리뷰 스킬에 정의된 기준(함수 길이, 타입 힌트 유무, 테스트 포함 여부 등)에 따라 자동 리뷰 코멘트를 작성한다. 감시-판단-실행 패턴과 생성-검증-수정 패턴이 결합된 구조이다.

두 번째 제안은 문서 동기화 에이전트이다. 파일 시스템 MCP 서버로 소스 코드 디렉토리의 변경을 감지하고, 문서 업데이트 스킬에 정의된 규칙에 따라 관련 문서를 자동으로 갱신한다. 함수 시그니처가 변경되면 API 문서를 업데이트하고, 새 모듈이 추가되면 목차를 갱신하는 식이다. 감시-판단-실행 패턴에 해당한다.

세 번째 제안은 학습 도우미 에이전트이다. 웹 검색 MCP 서버로 특정 기술 주제에 관한 최신 자료를 수집하고, 학습 가이드 스킬에 정의된 교육 방법론(개념 설명 -> 예시 코드 -> 연습 문제 순서)에 따라 맞춤형 학습 자료를 생성한다. 조사-분석-보고 패턴의 변형이다.

설계가 완료되면 다음 사항을 포함하여 발표를 준비한다. 에이전트의 목적과 대상 사용자, 사용하는 MCP 서버 목록과 각 서버의 역할, SKILL.md의 주요 내용, 적용한 설계 패턴, 그리고 예상되는 한계와 개선 방향이다. 에이전트를 직접 실행하여 실제 산출물을 보여 주면 더욱 효과적인 발표가 된다.

**표 5.3** 에이전트 설계서 작성 양식

| 항목 | 작성 내용 |
|------|----------|
| 에이전트 이름 | 에이전트의 용도를 나타내는 명칭 |
| 해결하는 문제 | 수동으로 수행할 때의 불편함이나 비효율 |
| MCP 서버 목록 | 서버 이름, 용도, 사용할 주요 도구 |
| SKILL.md 핵심 규칙 | 판단 기준, 출력 형식, 품질 기준 |
| 설계 패턴 | 조사-분석-보고 / 감시-판단-실행 / 생성-검증-수정 |
| 예상 한계 | 에이전트가 처리하지 못하는 상황과 대응 방안 |

이 양식에 따라 설계서를 작성하면, 에이전트의 구조를 체계적으로 정리할 수 있고 동료에게 설계 의도를 명확하게 전달할 수 있다. 실습 과정에서 작성한 설계서는 향후 유사한 프로젝트의 참고 자료로 활용할 수 있으므로, 충분히 구체적으로 기술하는 것이 좋다.

## 핵심정리

- MCP는 "능력(도구)"을, Skills는 "지식(절차)"을 제공한다
- 두 요소를 결합하면 도메인 전문가 수준의 자동화가 가능하다
- 디버깅의 핵심 도구: VS Code Output 패널의 MCP 채널에서 도구 호출을 확인한다
- 스킬의 품질이 에이전트 출력의 품질을 결정한다 -- 스킬을 점진적으로 정교화하는 것이 올바른 접근이다
- 세 가지 설계 패턴(조사-분석-보고, 감시-판단-실행, 생성-검증-수정)을 인식하면 새 프로젝트의 구조를 빠르게 잡을 수 있다

## 3.1 MCP 서버의 구조

2강에서는 이미 만들어진 MCP 서버를 Copilot에 연결하는 방법을 다루었다. GitHub MCP 서버, Filesystem 서버처럼 커뮤니티가 제공하는 도구를 설치하고 사용하는 것만으로도 Copilot의 능력을 크게 확장할 수 있었다. 그러나 기존 서버만으로는 자신의 업무에 특화된 기능을 제공하기 어렵다. 이번 강에서는 한 단계 더 나아가, MCP 서버의 내부를 열어 보고 직접 만들어 본다. MCP 서버를 직접 만들 수 있다는 것은 곧 Copilot에 원하는 능력을 자유롭게 추가할 수 있다는 뜻이다.

MCP 서버는 세 가지 핵심 요소(Primitive)로 구성된다. 첫째는 도구(Tool), 둘째는 리소스(Resource), 셋째는 프롬프트(Prompt)이다. 이 세 요소의 역할을 정확히 이해하는 것이 MCP 서버 설계의 출발점이다.

### 3.1.1 도구(Tool): 에이전트가 수행하는 행동

도구는 에이전트가 외부 세계에 영향을 미치는 행동을 정의한다. API를 호출하거나, 데이터베이스에 쿼리를 보내거나, 파일을 생성하는 것이 모두 도구에 해당한다. 각 도구는 네 가지 구성 요소를 가진다. 이름(name)은 LLM이 도구를 식별하는 데 사용하고, 설명(description)은 LLM이 도구의 용도를 판단하는 근거가 된다. 입력 스키마(input schema)는 JSON Schema 형식으로 파라미터의 이름, 타입, 설명을 정의하며, 핸들러 함수(handler)는 실제 로직을 실행하는 비동기 함수이다.

도구를 설계할 때 가장 중요한 점은 LLM이 도구를 선택하는 과정을 이해하는 것이다. LLM은 도구의 이름과 설명만을 읽고 "이 도구를 지금 호출해야 하는가"를 판단한다. 코드를 직접 읽지 않는다. 따라서 이름은 명확하고 서술적이어야 하며, 설명은 도구가 무엇을 하고 어떤 상황에서 사용해야 하는지를 구체적으로 기술해야 한다.

### 3.1.2 리소스(Resource): 에이전트가 읽는 데이터

리소스는 에이전트가 읽기 전용으로 접근할 수 있는 데이터를 정의한다. 파일 내용, 데이터베이스 레코드, 설정 정보 등이 해당한다. 리소스는 URI(Uniform Resource Identifier) 기반으로 접근하며, `config://settings`이나 `db://users/123`처럼 프로토콜과 경로로 식별한다. 도구가 "행동"이라면 리소스는 "데이터"에 해당하며, 에이전트가 작업에 필요한 맥락 정보를 얻는 통로가 된다.

도구와 리소스의 구분이 처음에는 모호하게 느껴질 수 있다. 판단 기준은 "이 기능이 외부 상태를 변경하는가"이다. 상태를 변경한다면(메시지 전송, 파일 생성, API 호출 등) 도구로, 이미 존재하는 데이터를 그대로 반환한다면 리소스로 설계하는 것이 적절하다. 예를 들어, 사용자 프로필을 조회하는 것은 리소스에 가깝고, 사용자 프로필을 수정하는 것은 도구에 가깝다.

### 3.1.3 프롬프트(Prompt): 재사용 가능한 템플릿

프롬프트는 자주 사용하는 프롬프트 패턴을 서버 측에서 템플릿으로 제공하는 기능이다. 예를 들어 "요약" 프롬프트나 "코드 리뷰" 프롬프트를 미리 정의해 두면, 클라이언트가 이를 호출하여 일관된 형식의 지시를 LLM에 전달할 수 있다. 다만 현재 MCP 생태계에서 프롬프트는 도구나 리소스에 비해 활용 빈도가 낮으므로, 이번 실습에서는 도구와 리소스에 집중한다.

다음은 이 세 요소가 하나의 MCP 서버 안에서 어떻게 구성되는지를 보여 주는 구조도이다.

```
MCP Server
├── Tools (행동)
│   ├── get_weather: 날씨 조회
│   └── send_notification: 알림 전송
├── Resources (데이터)
│   └── config://settings: 설정 정보
└── Prompts (템플릿)
    └── summarize: 요약 프롬프트
```

**그림 3.1** MCP 서버의 세 가지 핵심 요소(Primitive)

### 3.1.4 좋은 도구 설계의 원칙

도구를 설계할 때는 네 가지 원칙을 따른다. 첫째, 이름과 설명을 명확하게 작성한다. LLM은 이 정보만으로 도구 호출 여부를 결정하기 때문이다. 둘째, 입력 파라미터에 JSON Schema를 정확하게 정의한다. 타입, 필수 여부, 설명을 빠짐없이 기술해야 LLM이 올바른 인자를 구성할 수 있다. 셋째, 오류 발생 시 사람이 아닌 LLM이 읽을 수 있는 구체적인 오류 메시지를 반환한다. LLM은 이 메시지를 읽고 다음 행동을 결정하기 때문이다. 넷째, 가능한 한 멱등성(Idempotency)을 유지한다. 같은 입력으로 여러 번 호출해도 동일한 결과가 나오도록 설계하면 재시도가 안전해진다. LLM은 네트워크 오류 등으로 도구 호출이 실패했다고 판단하면 동일한 호출을 다시 시도할 수 있기 때문이다.

## 3.2 개발 환경 준비

MCP 서버를 만들기 위해 Python 프로젝트를 하나 생성한다. 가상환경을 만들고 필요한 패키지를 설치하는 과정은 다음과 같다.

```bash
mkdir my-mcp-server && cd my-mcp-server
python3 -m venv venv && source venv/bin/activate  # Windows: venv\Scripts\activate
pip install mcp httpx python-dotenv
```

`mcp` 패키지는 MCP 서버를 구현하기 위한 공식 Python SDK이다. 이 패키지 안에 FastMCP 클래스, 전송 계층(transport), JSON-RPC 핸들링 등 서버 구현에 필요한 모든 기능이 포함되어 있다. `httpx`는 비동기 HTTP 클라이언트로, 외부 API를 호출할 때 사용한다. Python 표준 라이브러리의 `urllib`와 달리 `async`/`await` 구문을 네이티브로 지원하므로 MCP 서버의 비동기 핸들러와 잘 어울린다. `python-dotenv`는 `.env` 파일에서 API 키와 같은 환경 변수를 불러오는 데 활용한다.

프로젝트 폴더 안에 `.env` 파일을 생성하고 API 키를 기록해 둔다. 이 파일은 `.gitignore`에 반드시 추가하여 저장소에 커밋되지 않도록 한다. API 키가 공개 저장소에 노출되면 타인이 악용할 수 있으므로 이 단계는 반드시 지켜야 한다.

```
OPENWEATHER_API_KEY=발급받은_API_키
```

OpenWeatherMap은 무료 플랜으로 분당 60회 호출을 허용하므로 학습 용도로 충분하다. 회원가입 후 API 키를 발급받으면 즉시 사용할 수 있다. 발급 직후에는 활성화까지 몇 분이 걸릴 수 있으므로, 환경 준비 단계에서 미리 발급해 두는 것을 권장한다.

## 3.3 실습: 날씨 API MCP 서버

이제 실제로 동작하는 MCP 서버를 만들어 본다. 목표는 도시 이름을 받아 현재 날씨를 반환하는 도구와, 지원 도시 목록을 반환하는 리소스를 갖춘 서버를 구현하는 것이다.

### 3.3.1 서버 기본 뼈대

`weather_server.py` 파일을 생성하고 서버의 뼈대를 작성한다. MCP Python SDK에서 제공하는 FastMCP 패턴을 사용하면 데코레이터(Decorator) 한 줄로 도구와 리소스를 등록할 수 있다.

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather-server")
```

이 두 줄이 MCP 서버의 시작점이다. `FastMCP` 클래스의 인자로 전달하는 문자열은 서버의 이름으로, 클라이언트가 여러 서버를 구분할 때 사용한다. 하나의 VS Code 프로젝트에 여러 MCP 서버를 동시에 연결할 수 있으므로, 서버 이름은 기능을 직관적으로 나타내는 것이 좋다.

### 3.3.2 도구 등록: get_weather

날씨 조회 도구를 등록하는 코드의 핵심 구조는 다음과 같다.

```python
@mcp.tool()
async def get_weather(city: str) -> str:
    """지정한 도시의 현재 날씨를 조회한다."""
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"{BASE_URL}?q={city}&appid={API_KEY}&units=metric&lang=kr")
    data = resp.json()
    return f"{city}: {data['main']['temp']}°C, {data['weather'][0]['description']}"
```

`@mcp.tool()` 데코레이터를 함수 위에 붙이면 해당 함수가 MCP 도구로 자동 등록된다. SDK는 함수의 이름(`get_weather`), 독스트링(docstring), 타입 힌트(`city: str`)를 읽어 도구의 이름, 설명, 입력 스키마를 자동으로 생성한다. 이것이 FastMCP 패턴의 핵심적인 장점이다. 별도의 JSON Schema를 수동으로 작성할 필요가 없다.

### 3.3.3 리소스 등록: 지원 도시 목록

리소스도 도구와 동일한 패턴으로 등록한다.

```python
@mcp.resource("cities://list")
async def list_cities() -> str:
    """지원하는 도시 목록을 반환한다."""
    cities = ["서울", "부산", "대구", "인천", "광주", "대전", "울산", "제주"]
    return ", ".join(cities)
```

`@mcp.resource()` 데코레이터의 인자로 URI를 전달한다. 클라이언트는 이 URI(`cities://list`)를 통해 해당 리소스에 접근한다. 도구와 달리 리소스는 읽기 전용 데이터를 제공하므로, 외부 상태를 변경하지 않는다는 점을 유의한다.

### 3.3.4 서버 실행

파일의 마지막에 서버 진입점을 추가한다.

```python
if __name__ == "__main__":
    mcp.run(transport="stdio")
```

`transport="stdio"`는 표준 입출력(Standard I/O)을 통해 JSON-RPC 메시지를 주고받는 방식이다. 로컬에서 MCP 서버를 실행할 때 가장 일반적으로 사용하는 전송 방식이며, VS Code Copilot도 이 방식으로 MCP 서버와 통신한다. MCP는 이 외에도 SSE(Server-Sent Events) 전송 방식을 지원하는데, 이는 서버를 원격 호스트에 배포할 때 사용한다. 이번 실습에서는 로컬 실행에 집중하므로 STDIO 방식을 사용한다.

### 3.3.5 MCP Inspector로 테스트

서버를 만들었다고 바로 Copilot에 연결하기보다는, 먼저 독립적으로 서버가 올바르게 동작하는지 검증하는 것이 좋다. Copilot 환경에서 디버깅하면 LLM의 응답과 도구 오류가 뒤섞여 문제 원인을 파악하기 어렵다. MCP Inspector는 서버를 LLM 없이 독립적으로 테스트할 수 있는 공식 도구이다.

```bash
npx @modelcontextprotocol/inspector python weather_server.py
```

이 명령을 실행하면 웹 브라우저에 Inspector UI가 열린다. Inspector에서는 서버에 등록된 도구 목록을 확인하고, 파라미터를 직접 입력하여 도구를 호출할 수 있다. 또한 클라이언트와 서버 사이에 오가는 JSON-RPC 메시지의 원문(raw message)을 확인할 수 있어 디버깅에 유용하다. 예를 들어, `get_weather` 도구에 `"서울"`을 입력하면 실제 API가 호출되고 날씨 정보가 반환되는 전체 과정을 눈으로 확인할 수 있다.

Inspector의 왼쪽 패널에서 "Tools" 탭을 선택하면 서버에 등록된 도구 목록이 나타난다. `get_weather`를 클릭하면 오른쪽에 입력 폼이 표시되고, `city` 파라미터에 값을 입력한 뒤 "Run" 버튼을 누르면 실제 도구가 호출된다. 하단의 "Messages" 패널에서는 클라이언트가 보낸 JSON-RPC 요청과 서버의 응답을 원문 그대로 확인할 수 있다. 이 메시지를 살펴보면 MCP 프로토콜이 내부적으로 어떻게 동작하는지를 직관적으로 이해할 수 있다.

Inspector에서 도구 호출이 정상적으로 동작하고 리소스 조회도 올바른 결과를 반환하는 것을 확인한 뒤에 다음 단계로 넘어간다.

## 3.4 Copilot에 연결하기

서버가 정상적으로 동작하는 것을 확인했으므로, 이제 VS Code Copilot의 Agent Mode에서 이 서버를 사용할 수 있도록 설정한다.

프로젝트 루트에 `.vscode/mcp.json` 파일을 생성하고 다음과 같이 작성한다.

```json
{
  "servers": {
    "weather": {
      "command": "python",
      "args": ["weather_server.py"],
      "cwd": "${workspaceFolder}/my-mcp-server"
    }
  }
}
```

`command`와 `args`는 서버를 실행하는 명령어를 지정한다. `cwd`는 작업 디렉터리로, `${workspaceFolder}`는 VS Code가 현재 열고 있는 폴더의 경로를 가리키는 변수이다. 이 설정을 저장하면 Copilot이 자동으로 서버를 시작하고 사용 가능한 도구 목록을 인식한다.

이제 Copilot Chat을 Agent Mode로 전환한 뒤 자연어로 요청해 본다.

첫 번째 테스트로 "서울의 현재 날씨를 알려줘"라고 입력한다. Copilot은 사용자의 요청을 분석하고, 등록된 도구 목록에서 `get_weather`가 적절하다고 판단하면 `city: "서울"` 인자와 함께 도구를 호출한다. 서버가 API를 통해 날씨 정보를 가져오면 Copilot은 그 결과를 자연어로 정리하여 사용자에게 보여 준다.

두 번째 테스트로 "지원하는 도시 목록을 보여줘"라고 입력한다. 이때 Copilot은 `cities://list` 리소스를 읽어 도시 목록을 반환한다.

이 과정을 통해, 직접 만든 MCP 서버가 Copilot의 새로운 능력으로 작동하는 것을 확인할 수 있다. 2강에서 남이 만든 서버를 연결한 것과 달리, 이번에는 자신이 원하는 기능을 직접 정의하고 Copilot에 제공한 것이다.

주의할 점이 하나 있다. Copilot이 도구를 호출하기 전에 사용자에게 승인을 요청하는 확인 대화상자가 나타난다. 이것은 보안을 위한 설계이다. MCP 도구는 외부 API를 호출하거나 시스템에 영향을 줄 수 있으므로, 사용자가 매번 실행 여부를 확인하는 것이 기본 동작이다. 세션 내에서 동일한 도구에 대해 반복 승인이 번거롭다면, "이 세션 동안 항상 허용"을 선택할 수 있다.

## 3.5 도구 설명의 핵심 원칙

MCP 서버를 만들 때 흔히 저지르는 실수는 코드 로직에만 집중하고 도구의 이름과 설명을 대충 작성하는 것이다. 그러나 LLM은 도구의 소스 코드를 읽지 않는다. LLM이 도구를 선택할 때 참고하는 정보는 오직 세 가지, 즉 도구의 이름, 설명, 그리고 파라미터 스키마뿐이다.

설명이 모호하면 LLM은 적절한 상황에서도 도구를 호출하지 못하거나, 잘못된 상황에서 호출할 수 있다. 반대로 설명이 구체적이고 정확하면 LLM의 도구 선택 정확도가 크게 향상된다.

다음 표는 도구 설명의 나쁜 예와 좋은 예를 비교한 것이다.

**표 3.1** 도구 설명의 좋은 예와 나쁜 예

| 구분 | 나쁜 예 | 좋은 예 |
|-----|--------|--------|
| 이름 | `do_stuff` | `get_current_weather` |
| 설명 | "날씨 관련 작업" | "지정한 도시의 현재 기온, 습도, 날씨 상태를 조회한다. 도시명은 한국어 또는 영문으로 입력 가능하다." |
| 파라미터 | `data: str` | `city: str` (description: "조회할 도시 이름 (예: '서울', 'Tokyo')") |

좋은 예에서 주목할 점은 세 가지이다. 이름이 동사+명사 형태로 행동을 명확히 나타내고, 설명이 반환하는 정보의 종류(기온, 습도, 날씨 상태)를 구체적으로 열거하며, 파라미터에 예시 값이 포함되어 LLM이 어떤 형식의 값을 전달해야 하는지 직관적으로 파악할 수 있다.

파라미터 설명 역시 중요하다. FastMCP에서는 함수의 타입 힌트가 자동으로 JSON Schema로 변환되지만, 파라미터의 의미까지 자동으로 추론되지는 않는다. 독스트링에 각 파라미터의 용도와 예시를 명시하는 습관을 들이는 것이 좋다.

도구의 수가 많아질 때에도 설명의 품질은 더욱 중요해진다. 서버에 도구가 3개일 때는 LLM이 비교적 쉽게 올바른 도구를 선택하지만, 10개 이상이 되면 이름과 설명의 차별성이 떨어질수록 잘못된 도구가 선택될 확률이 높아진다. 도구의 이름이 기능을 정확히 반영하고, 설명이 서로 겹치지 않도록 작성하는 것이 대규모 MCP 서버를 운영하는 데 있어 필수적인 습관이다.

## 3.6 오류 처리와 안정성

외부 API를 호출하는 MCP 서버는 필연적으로 다양한 오류 상황에 직면한다. 네트워크 타임아웃, 잘못된 입력, API 응답 오류 등이 발생할 수 있으며, 이를 적절히 처리하지 않으면 서버가 크래시(crash)하거나 LLM이 의미 없는 에러 스택을 받게 된다.

오류 처리에서 핵심은 에러 메시지의 수신자가 사람이 아니라 LLM이라는 점이다. LLM은 에러 메시지를 읽고 사용자에게 상황을 설명하거나 다른 접근 방식을 시도한다. 따라서 에러 메시지는 무엇이 잘못되었고, 어떻게 하면 해결할 수 있는지를 구체적으로 알려 주어야 한다.

다음 코드는 세 가지 대표적인 오류 상황에 대한 처리 패턴을 보여 준다.

```python
@mcp.tool()
async def get_weather(city: str) -> str:
    """도시의 현재 날씨를 조회한다."""
    if not city.strip():
        return "오류: 도시 이름이 비어 있다. 도시명을 입력해 달라."
    try:
        result = await fetch_weather(city)
        return f"{city}의 현재 기온은 {result['temp']}°C, {result['desc']}"
    except httpx.TimeoutException:
        return f"오류: {city} 날씨 조회 시 타임아웃이 발생했다. 잠시 후 다시 시도해 달라."
    except httpx.HTTPStatusError as e:
        return f"오류: API가 {e.response.status_code} 상태를 반환했다. 도시명을 확인해 달라."
```

첫 번째 `if` 문은 입력 검증(validation)에 해당한다. 빈 문자열이 도구의 인자로 전달되는 경우를 방지한다. 두 번째 `except`는 네트워크 타임아웃 처리이다. 외부 API 호출은 언제든 느려질 수 있으므로, 타임아웃을 적절히 설정하고 발생 시 LLM이 이해할 수 있는 메시지를 반환한다. 세 번째 `except`는 API 응답 오류 처리로, HTTP 상태 코드를 함께 전달하여 문제의 원인을 파악할 수 있게 한다.

이처럼 오류 메시지를 "LLM이 읽는 텍스트"로 인식하고, 문제의 원인과 대응 방법을 함께 제공하는 것이 안정적인 MCP 서버의 조건이다. 예외를 상위로 전파(raise)하기보다는 문자열로 래핑하여 반환하면, 서버 프로세스가 비정상 종료되는 것을 방지할 수 있다.

한 가지 실전 팁을 덧붙이면, `httpx.AsyncClient`를 생성할 때 `timeout` 파라미터를 명시적으로 설정하는 것이 좋다. 기본값은 5초이지만, 외부 API의 응답 속도에 따라 적절히 조정해야 한다. 타임아웃이 너무 짧으면 정상적인 요청도 실패하고, 너무 길면 사용자가 오래 기다려야 한다. 일반적으로 10~15초가 적절한 범위이다.

## 3.7 도전 과제: 학교 정보 MCP 서버

이번 강의에서 배운 내용을 종합하여, 자신의 학교 또는 조직에 맞는 MCP 서버를 직접 설계하고 구현해 본다. 날씨 서버는 외부 API를 래핑하는 패턴이었다면, 이번에는 자신에게 의미 있는 도메인을 선택하여 설계부터 시작하는 것이 과제의 요점이다. 실제 API가 없다면 모의 데이터(mock data)를 사용해도 좋다. 핵심은 도구와 리소스를 적절히 나누어 설계하고, 도구 설명을 LLM이 이해할 수 있게 작성하며, 오류 처리를 빠짐없이 포함하는 것이다.

구현할 요소는 세 가지 도구와 한 가지 리소스이다. 첫 번째 도구 `get_menu`는 오늘의 학식 메뉴를 조회한다. 날짜를 파라미터로 받으면 해당 날짜의 메뉴를 반환하고, 생략하면 오늘 날짜를 기본값으로 사용한다. 두 번째 도구 `get_room_schedule`은 특정 강의실의 사용 현황을 조회한다. 건물명과 호수를 파라미터로 받아 해당 강의실의 시간표를 반환한다. 세 번째 도구 `get_notice`는 학교 공지사항을 검색한다. 키워드를 입력하면 관련 공지 목록을 반환한다.

리소스로는 `campus://info` URI를 통해 학교 기본 정보(이름, 주소, 대표 전화번호, 학과 목록 등)를 제공한다. 이 정보는 변경 빈도가 낮고 읽기 전용이므로 도구보다 리소스가 적절하다.

설계할 때 도구와 리소스의 경계를 의식적으로 구분해 보는 것이 좋다. 학식 메뉴는 매일 바뀌고 날짜 파라미터에 따라 다른 결과를 반환하므로 도구가 적절하다. 반면 학교 기본 정보는 거의 변하지 않는 고정 데이터이므로 리소스로 제공하는 것이 자연스럽다.

구현이 끝나면 MCP Inspector로 각 도구를 개별 테스트한 뒤, `.vscode/mcp.json`에 서버를 등록하고 Copilot Agent Mode에서 자연어로 동작을 확인한다. 예를 들어 "오늘 학식 메뉴가 뭐야?"라고 물었을 때 Copilot이 `get_menu` 도구를 호출하고 결과를 자연어로 정리해 주는 것을 확인할 수 있다. "3층 컴퓨터실 오후에 비어 있어?"라는 질문에는 `get_room_schedule` 도구가 호출되고, "장학금 관련 공지 찾아줘"라는 요청에는 `get_notice` 도구가 "장학금"을 키워드로 검색하는 것을 기대할 수 있다.

이 과제를 통해 "내가 필요한 도구를 직접 정의하고, AI 에이전트에 연결하여 자연어로 사용할 수 있게 만드는" 전체 과정을 체험하게 된다. 이것이 MCP 서버 개발의 핵심 가치이다.

---

## 핵심정리

- MCP 서버의 3요소: Tool(행동), Resource(데이터), Prompt(템플릿)
- `mcp` Python SDK의 FastMCP 패턴으로 데코레이터 한 줄로 도구와 리소스를 등록할 수 있다
- 도구의 이름과 설명이 LLM의 도구 선택을 결정하므로, 구체적이고 명확하게 작성해야 한다
- 오류 메시지는 LLM이 읽고 판단하는 텍스트이므로, 원인과 대응 방법을 함께 제공한다
- MCP Inspector로 서버를 독립 테스트한 뒤 Copilot에 연결하는 순서를 따른다

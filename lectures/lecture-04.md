# 제4강. Skills로 Copilot 커스터마이징

> **핵심 산출물**: SKILL.md 2개 이상 (코드 리뷰 스킬 + API 문서 생성 스킬)
> **소요 시간**: 약 90분
> **선수 지식**: 3강(MCP 서버 제작)까지 완료, VS Code + GitHub Copilot 확장 설치

---

## 학습목표

- Agent Skills의 개념과 Custom Instructions와의 차이를 이해한다
- SKILL.md 파일의 구조와 작성법을 익힌다
- 프로젝트에 맞는 Skills를 직접 만들어 Copilot에 적용한다

---

## 4.1 Skills란 무엇인가

3강까지 MCP 서버를 통해 Copilot에 새로운 "도구(tool)"를 부여하는 방법을 다루었다. MCP가 Copilot의 **행동 능력**을 확장한다면, 이번 강에서 다루는 Skills(스킬)는 Copilot의 **판단 기준**을 확장하는 기술이다. GitHub는 2025년 12월 18일에 Agent Skills를 공식 발표하였으며, 이를 통해 개발자는 Copilot에 도메인별 지식과 절차를 체계적으로 가르칠 수 있게 되었다.

스킬은 지시 사항(instructions), 스크립트(scripts), 참조 파일(references)을 하나의 폴더에 묶어 놓은 자기 완결적 패키지이다. Copilot은 사용자의 프롬프트가 특정 스킬과 관련이 있다고 판단할 때만 해당 스킬을 자동으로 불러온다. 이 점이 기존의 Custom Instructions(커스텀 인스트럭션)와 가장 큰 차이이다.

Custom Instructions는 `.github/copilot-instructions.md` 파일에 작성하며, Copilot이 수행하는 모든 작업에 항상 적용된다. "코드는 반드시 TypeScript로 작성한다"처럼 프로젝트 전반에 걸치는 간단한 규칙을 지정할 때 적합하다. 반면 Skills는 특정 도메인이나 작업에 한정된 상세한 지식을 담으며, 관련 프롬프트가 들어올 때만 활성화된다.

**표 4.1** Custom Instructions와 Agent Skills 비교

| 특성 | Custom Instructions | Agent Skills |
|------|-------------------|--------------|
| 범위 | 모든 작업에 항상 적용 | 관련 프롬프트에만 자동 활성화 |
| 내용 수준 | 간단한 코딩 표준, 가이드라인 | 상세 지시 + 스크립트 + 참조 파일 |
| 활성화 방식 | 항상 로드 | 프롬프트 매칭 기반 자동 판단 |
| 파일 번들링 | 불가 | 가능 (스크립트, 템플릿, 문서) |
| 저장 위치 | `.github/copilot-instructions.md` | `.github/skills/` 또는 `~/.copilot/skills/` |
| 팀 공유 | 저장소 커밋으로 공유 | 저장소 커밋으로 공유 (프로젝트 스킬) |

Copilot이 스킬을 처리하는 과정은 3단계 점진적 공개(3-tier progressive disclosure) 모델을 따른다. 첫 번째 단계는 발견(Discovery)으로, Copilot은 프로젝트에 등록된 모든 스킬의 이름과 설명(description)만 미리 읽어 둔다. 이 정보는 매우 가볍기 때문에 항상 로드되어 있다. 두 번째 단계는 지시 로딩(Instruction Loading)으로, 사용자의 프롬프트가 특정 스킬의 설명과 일치한다고 판단되면 해당 SKILL.md의 전체 본문을 읽어 들인다. 세 번째 단계는 리소스 접근(Resource Access)으로, 본문의 지시에 따라 필요한 경우에만 번들된 스크립트나 템플릿 파일을 참조한다. 이 3단계 구조 덕분에 수십 개의 스킬을 등록해 두어도 컨텍스트 윈도우를 낭비하지 않는다.

## 4.2 SKILL.md 파일 구조

모든 스킬의 핵심은 SKILL.md 파일이다. 이 파일은 YAML 프론트매터(frontmatter)와 마크다운 본문으로 구성된다.

YAML 프론트매터에는 두 가지 필수 필드가 있다. `name`은 스킬의 고유 식별자로, 소문자와 하이픈만 사용하며 최대 64자까지 허용된다. `description`은 스킬의 용도를 설명하는 문장으로, 최대 1,024자까지 작성할 수 있다. Copilot은 이 description을 읽고 스킬 활성화 여부를 결정하므로, 어떤 상황에서 이 스킬이 필요한지를 명확하고 구체적으로 서술해야 한다. description 작성 시 흔히 하는 실수는 "유용한 스킬이다"처럼 모호한 문장을 쓰는 것이다. "코드 리뷰를 요청할 때", "PR을 검토할 때"처럼 구체적인 사용 맥락을 나열하는 것이 올바른 접근이다.

```yaml
---
name: code-review
description: >
  코드 리뷰를 요청하거나, PR을 검토하거나,
  코드 품질을 확인할 때 사용한다.
---
```

프론트매터 아래에는 마크다운 형식으로 Copilot이 따라야 할 지시 사항을 작성한다. 작업 절차, 준수 사항, 출력 형식, 예시 등을 자유롭게 기술할 수 있다. 다만 Copilot의 컨텍스트 윈도우 한계를 고려하여, 핵심적인 내용에 집중하고 불필요한 반복은 피해야 한다.

스킬 폴더의 전체 디렉터리 구조는 다음과 같다.

```
my-skill/
  SKILL.md           # 핵심 지시 파일 (필수)
  scripts/           # 자동화 유틸리티 (선택)
  references/        # API 문서, 레퍼런스 (선택)
  assets/            # 템플릿, 코드 샘플 (선택)
  examples/          # 예시 구현 (선택)
```

SKILL.md만 필수이며 나머지 하위 폴더는 모두 선택 사항이다. 그러나 스크립트나 템플릿을 번들로 포함시킬 수 있다는 점이 Custom Instructions 대비 스킬의 가장 강력한 장점이다. 예를 들어, 코드 리뷰 스킬에 보안 체크리스트 문서를 `references/security-checklist.md`로 번들하면, Copilot이 리뷰 시 이 체크리스트를 실제로 참조하여 검토한다.

스킬의 저장 위치는 두 곳이 가능하다. 팀 전체가 공유할 프로젝트 스킬(Project Skills)은 프로젝트 루트의 `.github/skills/` 디렉터리 아래에 둔다. 이 경우 저장소에 커밋되므로 모든 팀원이 동일한 스킬을 사용하게 된다. 개인만 사용할 퍼스널 스킬(Personal Skills)은 홈 디렉터리의 `~/.copilot/skills/` 아래에 둔다. 이 경우 해당 사용자의 모든 프로젝트에서 사용할 수 있지만 다른 사람과 공유되지 않는다.

두 위치에 같은 이름의 스킬이 존재할 경우, 프로젝트 스킬이 퍼스널 스킬보다 우선한다. 따라서 팀 표준을 프로젝트 스킬로 설정하고, 개인 선호 사항은 퍼스널 스킬로 보완하는 전략이 가능하다. 다만 스킬이 늘어날수록 관리 복잡도도 높아지므로, 프로젝트 스킬에는 팀 합의된 핵심 스킬만 두고 실험적 스킬은 퍼스널 영역에서 검증한 후 승격시키는 방식을 권장한다.

## 4.3 실습 1: 코드 리뷰 스킬 만들기

첫 번째 실습으로 팀의 코딩 표준에 맞추어 코드를 리뷰하는 스킬을 만든다. 코드 리뷰는 모든 개발 팀에서 수행하지만, 리뷰의 기준과 깊이는 팀마다 다르다. 한 팀은 보안 취약점을 최우선으로 점검하고, 다른 팀은 성능 최적화에 초점을 맞춘다. 이런 팀 고유의 리뷰 기준을 스킬로 정의하면 Copilot이 팀의 관점에서 리뷰를 수행한다.

먼저 프로젝트 루트에 스킬 디렉터리를 생성한다.

```bash
mkdir -p .github/skills/code-review/references
```

다음으로 `.github/skills/code-review/SKILL.md` 파일을 작성한다. 프론트매터의 description에는 "코드 리뷰", "PR 검토", "코드 품질"처럼 이 스킬이 활성화되어야 하는 맥락을 충분히 나열하는 것이 중요하다. Copilot은 이 description과 사용자 프롬프트 사이의 의미적 유사도를 기반으로 활성화를 판단하기 때문이다.

```yaml
---
name: code-review
description: >
  코드 리뷰를 요청하거나, PR을 검토하거나,
  코드 품질을 확인할 때 사용한다.
  팀의 코딩 표준과 보안 체크리스트에 따라 리뷰한다.
---
```

본문에는 리뷰 절차와 기준을 구체적으로 작성한다. 핵심 코드만 발췌하면 다음과 같다.

```markdown
## 리뷰 절차
1. 보안: 하드코딩된 시크릿, SQL 인젝션, XSS 취약점을 확인한다.
2. 성능: O(n^2) 이상의 루프, 불필요한 DB 호출을 식별한다.
3. 가독성: 함수 길이 50줄 이하, 명확한 변수명을 확인한다.
```

_전체 SKILL.md는 `.github/skills/code-review/SKILL.md` 참고_

보안 체크리스트를 별도 파일로 번들하려면 `references/security-checklist.md`에 상세 항목을 작성하고, SKILL.md 본문에서 "보안 검토 시 `references/security-checklist.md`를 참조한다"라고 명시하면 된다. Copilot은 리소스 접근 단계에서 이 파일을 읽어 리뷰에 반영한다.

스킬이 제대로 동작하는지 테스트하려면 Copilot Chat에서 "이 파일을 리뷰해 줘"라고 요청한다. Copilot이 보안, 성능, 가독성 항목을 순서대로 점검하며 SKILL.md에 정의한 출력 형식을 따른다면 스킬이 정상적으로 활성화된 것이다. 만약 스킬이 활성화되지 않는다면 description의 키워드가 프롬프트와 충분히 매칭되지 않을 가능성이 높으므로 description을 수정해 본다.

스킬의 출력 형식도 지정할 수 있다. 예를 들어, 리뷰 결과를 "심각도: 높음/중간/낮음"으로 분류하고, 각 항목에 수정 제안을 포함하도록 SKILL.md에 명시하면 Copilot이 일관된 형식의 리뷰 코멘트를 생성한다. 출력 형식의 표준화는 팀 내 리뷰 품질을 균일하게 유지하는 데 도움이 된다.

## 4.4 실습 2: API 문서 생성 스킬 만들기

두 번째 실습에서는 REST API 엔드포인트 문서를 자동으로 생성하는 스킬을 만든다. 이 스킬은 번들 파일의 활용법을 보여 주는 좋은 예시이다.

```bash
mkdir -p .github/skills/api-docs/assets
```

`.github/skills/api-docs/SKILL.md`의 프론트매터는 다음과 같이 작성한다.

```yaml
---
name: api-docs
description: >
  API 문서를 생성하거나, 엔드포인트를 문서화하거나,
  REST API 명세를 작성할 때 사용한다.
  OpenAPI 형식의 템플릿을 기반으로 문서화한다.
---
```

이 스킬의 핵심은 `assets/` 폴더에 OpenAPI 템플릿을 번들하는 것이다. 템플릿 파일 `assets/endpoint-template.md`에는 각 엔드포인트가 갖추어야 할 문서 구조를 정의한다. 요청 메서드와 경로, 설명, 요청 파라미터, 응답 스키마, 에러 코드, 사용 예시 등의 항목을 표준 형식으로 명시한다.

SKILL.md 본문에서는 Copilot이 소스 코드에서 엔드포인트를 식별하는 방법, 각 엔드포인트에 대해 `assets/endpoint-template.md`의 형식을 적용하는 절차, 그리고 최종 문서의 출력 위치를 지시한다. 본문의 핵심 부분을 발췌하면 다음과 같다.

```markdown
## 문서화 절차
1. 대상 파일에서 라우트 데코레이터(@app.route, @router)를 식별한다.
2. 각 엔드포인트에 assets/endpoint-template.md 형식을 적용한다.
3. 요청/응답 스키마는 코드의 타입 힌트와 모델 정의에서 추출한다.
```

테스트를 위해 간단한 Flask 애플리케이션을 준비하고, Copilot Chat에서 "이 앱의 API 문서를 작성해 줘"라고 요청한다. Copilot이 템플릿의 형식을 따라 각 엔드포인트를 문서화한다면 스킬이 올바르게 작동하는 것이다. 생성된 문서에 요청 파라미터의 타입이나 응답 스키마가 누락되어 있다면, SKILL.md 본문의 지시를 더 구체적으로 보강하거나 `assets/endpoint-template.md` 템플릿에 필수 항목 표시를 추가한다.

이 실습에서 주목할 점은 스킬이 단순한 텍스트 지시를 넘어 참조 파일과 결합될 때 얼마나 강력해지는가이다. 템플릿 파일이 없다면 Copilot은 자체 판단에 따라 문서 형식을 결정하지만, 템플릿이 있으면 팀이 합의한 정확한 형식을 일관되게 재현한다.

## 4.5 실습 3: 테스트 작성 스킬 만들기

세 번째 실습에서는 프로젝트의 테스트 관례에 맞는 단위 테스트를 Copilot이 작성하도록 가르치는 스킬을 만든다. 테스트 작성은 팀마다 프레임워크, 네이밍 규칙, 픽스처 패턴이 다르기 때문에 스킬로 표준화하기에 적합한 영역이다.

```bash
mkdir -p .github/skills/test-writer/examples
```

프론트매터는 다음과 같다.

```yaml
---
name: test-writer
description: >
  테스트를 작성하거나, 단위 테스트를 추가하거나,
  테스트 커버리지를 높일 때 사용한다.
  pytest와 프로젝트의 테스트 패턴을 따른다.
---
```

SKILL.md 본문에는 팀이 사용하는 테스트 프레임워크(pytest), 테스트 파일 네이밍 규칙(`test_{모듈명}.py`), 테스트 함수 네이밍 패턴(`test_{기능}_{시나리오}_{기대결과}`), 그리고 픽스처 사용 원칙을 명시한다. `examples/` 폴더에는 잘 작성된 테스트의 실제 예시를 넣어 둔다. Copilot은 "보여 주기(show)" 방식의 예시로부터 패턴을 학습하므로, 추상적인 규칙만 나열하는 것보다 구체적인 예시 코드를 포함하는 것이 훨씬 효과적이다.

본문의 핵심 부분을 발췌하면 다음과 같다.

```markdown
## 테스트 작성 규칙
1. 파일명: test_{모듈명}.py, 함수명: test_{기능}_{시나리오}_{기대결과}
2. 외부 의존성은 반드시 mock 처리한다 (pytest-mock 사용).
3. 각 테스트는 Arrange-Act-Assert 패턴을 따른다.
```

테스트 대상 모듈을 열어 두고 "이 모듈의 테스트를 작성해 줘"라고 요청하면 Copilot이 스킬에 정의된 네이밍 규칙과 패턴을 따라 테스트를 생성한다. 픽스처 구조나 mock 처리 방식이 팀의 관례와 일치하는지 확인하고, 필요하면 SKILL.md를 보완한다.

스킬 없이 동일한 요청을 해 보면 차이가 분명해진다. 스킬이 없을 때 Copilot은 일반적인 pytest 패턴으로 테스트를 생성하지만, 스킬이 활성화되면 팀의 네이밍 규칙, Arrange-Act-Assert 구조, mock 처리 방식이 정확히 반영된다. 이 비교를 통해 스킬이 Copilot의 출력 품질에 미치는 영향을 체감할 수 있다.

## 4.6 스킬 설계 원칙

세 가지 실습을 통해 스킬의 기본적인 작성법을 익혔다. 코드 리뷰, API 문서 생성, 테스트 작성이라는 서로 다른 도메인에 스킬을 적용해 보면서, 스킬이 Copilot의 출력을 팀의 기준에 맞추어 표준화하는 메커니즘임을 확인하였다. 여기서는 이 경험을 바탕으로 효과적인 스킬을 설계하기 위한 핵심 원칙을 정리한다.

첫째, description이 모든 것을 결정한다. Copilot은 스킬의 본문을 읽기 전에 description만으로 활성화 여부를 판단한다. 따라서 사용자가 입력할 법한 키워드와 문맥을 description에 충분히 포함시켜야 한다. "코드 리뷰"만 적는 것보다 "코드 리뷰를 요청하거나, PR을 검토하거나, 코드 품질을 확인할 때"처럼 다양한 표현을 나열하는 편이 활성화 확률을 높인다.

둘째, 하나의 스킬은 하나의 도메인에 집중해야 한다. 코드 리뷰와 API 문서 생성을 하나의 스킬에 합치면 description이 모호해지고, Copilot이 불필요한 상황에서 스킬을 활성화하거나 필요한 상황에서 활성화하지 못하는 문제가 발생한다. 관심사 분리(Separation of Concerns) 원칙은 스킬 설계에도 그대로 적용된다.

셋째, 규칙을 서술하기보다 예시를 보여 주는 것이 효과적이다. "변수명은 명확하게 짓는다"라는 추상적 규칙보다 "good: `user_count`, bad: `uc`"처럼 구체적인 예시를 제시하면 Copilot이 더 정확하게 의도를 파악한다. `examples/` 폴더에 잘 작성된 코드와 잘못 작성된 코드를 모두 포함시키는 것도 좋은 방법이다.

넷째, 번들 파일을 적극적으로 활용한다. SKILL.md 본문에 모든 내용을 넣으면 컨텍스트 윈도우를 불필요하게 소비한다. 상세한 체크리스트, API 문서, 템플릿은 별도 파일로 분리하고 SKILL.md에서 참조만 하면, Copilot이 필요한 시점에만 해당 파일을 읽어 들인다. 이것이 3단계 점진적 공개 모델의 리소스 접근(Resource Access) 단계가 존재하는 이유이다.

다섯째, 스킬을 버전 관리하고 팀과 공유한다. `.github/skills/` 디렉터리에 저장한 스킬은 Git으로 버전 관리되므로, 스킬의 변경 이력을 추적하고 PR을 통해 팀 리뷰를 거칠 수 있다. 스킬 자체도 코드와 마찬가지로 지속적으로 개선해 나가야 하는 자산이다.

여섯째, 스킬 간 충돌을 주의한다. 유사한 도메인을 다루는 스킬이 여러 개 있으면 Copilot이 동시에 여러 스킬을 활성화할 수 있다. 이 경우 지시 사항이 충돌하여 예측하기 어려운 결과를 낳을 수 있으므로, 각 스킬의 description이 서로 명확히 구분되도록 작성해야 한다. 예를 들어 "프론트엔드 코드 리뷰"와 "백엔드 코드 리뷰"를 별도 스킬로 만들었다면 각 description에 "React 컴포넌트", "FastAPI 엔드포인트"처럼 대상 기술을 명시하여 구분한다.

## 4.7 커뮤니티 스킬 활용

스킬을 처음부터 모두 직접 만들 필요는 없다. GitHub의 `copilot-community/awesome-copilot-skills` 저장소에는 커뮤니티가 공유한 40개 이상의 스킬이 모여 있으며, Git 커밋 메시지 규칙, 웹 테스트 자동화, 다이어그램 생성 등 다양한 도메인을 다룬다.

커뮤니티 스킬을 설치하는 방법은 간단하다. 원하는 스킬 폴더를 다운로드하여 프로젝트의 `.github/skills/` 디렉터리에 복사하면 된다. 예를 들어, 커밋 메시지 규칙 스킬을 도입하려면 해당 스킬 폴더를 `.github/skills/commit-convention/`에 넣는다. 이후 Copilot에 "이 변경 사항에 대한 커밋 메시지를 작성해 줘"라고 요청하면 스킬에 정의된 Conventional Commits 형식을 따른다.

커뮤니티 스킬을 그대로 사용하기보다는 팀의 상황에 맞게 수정하는 것을 권장한다. description의 키워드를 팀이 실제로 사용하는 용어로 바꾸고, 규칙이나 템플릿을 프로젝트 관례에 맞추어 조정한다. 커뮤니티 스킬은 완성품이 아니라 출발점으로 활용하는 것이 가장 효과적이다.

한 가지 유의할 점은, 커뮤니티 스킬을 도입할 때 해당 스킬의 SKILL.md 본문을 반드시 검토해야 한다는 것이다. 스킬은 Copilot의 행동 방식에 직접 영향을 미치므로, 의도하지 않은 지시가 포함되어 있지 않은지 확인하는 과정이 필요하다. 이는 서드파티 라이브러리의 코드를 검토하는 것과 동일한 보안 습관이다.

커뮤니티에서 특히 유용하다고 평가받는 스킬 유형으로는 Git 커밋 컨벤션(Conventional Commits 형식 강제), 웹 접근성 검사(WCAG 기준 적용), 데이터베이스 마이그레이션(스키마 변경 절차 표준화) 등이 있다. 이러한 스킬은 각 팀의 구체적인 규칙으로 수정하기 쉬운 구조로 되어 있어 커스터마이징의 출발점으로 적합하다.

## 4.8 MCP와 Skills의 관계

3강에서 다룬 MCP와 이번 강의 Skills는 Copilot을 확장하는 두 축이다. 이 둘의 관계를 명확히 이해하면 적절한 확장 전략을 수립할 수 있다.

MCP 서버는 Copilot에 **도구(tool)**를 제공한다. 데이터베이스를 조회하거나, 외부 API를 호출하거나, 파일 시스템을 조작하는 것처럼 Copilot이 실제로 무언가를 "실행"할 수 있게 만든다. 반면 Skills는 Copilot에 **지식(knowledge)**과 **절차(procedure)**를 제공한다. 코드를 어떤 기준으로 리뷰해야 하는지, 문서를 어떤 형식으로 작성해야 하는지처럼 Copilot이 올바르게 "판단"할 수 있게 만든다.

**표 4.2** MCP와 Skills의 역할 비교

| 구분 | MCP | Skills |
|------|-----|--------|
| 제공하는 것 | 도구 (실행 능력) | 지식과 절차 (판단 기준) |
| 비유 | 새 도구를 손에 쥐어 주는 것 | 매뉴얼을 읽게 하는 것 |
| 예시 | DB 조회, API 호출, 파일 변환 | 코딩 표준, 리뷰 체크리스트, 문서 템플릿 |
| 활성화 | 도구 호출 시 | 프롬프트 매칭 시 |

5강에서는 이 두 가지를 조합한다. MCP 서버가 외부 데이터를 가져오는 도구를 제공하고, Skills가 그 데이터를 어떻게 처리하고 문서화해야 하는지를 지시하는 구조이다. 도구와 지식이 결합될 때 비로소 완전한 자동화 에이전트가 만들어진다.

실무에서의 전형적인 조합 패턴을 하나 들면, MCP 서버가 Jira에서 이슈 목록을 가져오는 도구를 제공하고, 릴리스 노트 작성 스킬이 그 이슈 데이터를 팀의 릴리스 노트 형식에 맞추어 정리하도록 지시하는 구조이다. 이때 MCP 없이는 Copilot이 Jira 데이터에 접근할 수 없고, Skills 없이는 Copilot이 팀의 문서 형식을 알 수 없다. 양쪽이 모두 갖추어져야 의미 있는 자동화가 가능하다.

## 핵심정리

- Skills는 Copilot에 도메인 지식과 절차를 가르치는 자기 완결적 패키지이다
- SKILL.md의 프론트매터(name, description)가 자동 활성화 여부를 결정한다
- Copilot은 발견 - 지시 로딩 - 리소스 접근의 3단계로 스킬을 처리한다
- 하나의 스킬은 하나의 도메인에 집중하고, 규칙보다 예시를 제공한다
- `.github/skills/`에 저장하면 Git으로 버전 관리되며 팀 전체가 공유한다
- MCP가 "도구"를 제공하고, Skills가 "지식"을 제공한다 -- 5강에서 이 둘을 결합한다

---

## 실습 점검 사항

이번 강의의 실습을 성공적으로 완료했다면, 프로젝트의 `.github/skills/` 디렉터리에 최소 2개의 스킬 폴더가 생성되어 있어야 한다. 각 스킬의 SKILL.md에는 올바른 YAML 프론트매터(name, description)가 포함되어 있고, Copilot Chat에서 관련 프롬프트를 입력했을 때 스킬의 지시에 따른 출력을 확인할 수 있어야 한다. 스킬이 활성화되지 않는 경우에는 description의 키워드를 점검하고, 출력 품질이 기대에 미치지 않는 경우에는 SKILL.md 본문의 지시를 더 구체적으로 보완한다.

## 다음 강의 예고

5강에서는 MCP 서버(도구 접근)와 Skills(지식과 절차)를 결합하여 완전한 자동화 에이전트를 만든다. 외부 API에서 데이터를 가져오는 MCP 서버와 그 데이터를 프로젝트 규칙에 맞게 가공하는 스킬을 하나의 워크플로우로 묶어, Copilot이 자율적으로 작업을 수행하는 통합 프로젝트를 구성한다.

# 9주차. Skills 기초

> **1회차** (강의 90분): Skills의 개념, Custom Instructions와의 차이, SKILL.md 파일 구조, 3단계 점진적 공개 모델
> **2회차** (실습 90분): 코드 리뷰 스킬 작성 + API 문서화 스킬 작성

---

## 학습목표

1. Agent Skills의 개념을 이해하고, Custom Instructions와의 차이를 설명할 수 있다
2. SKILL.md 파일의 구조(YAML 프론트매터 + 마크다운 본문)를 파악하고 직접 작성할 수 있다
3. 3단계 점진적 공개(3-tier progressive disclosure) 모델의 동작 원리를 설명할 수 있다
4. 프로젝트에 맞는 기본 스킬을 작성하여 Copilot에 적용하고 동작을 검증할 수 있다

## 선수지식

- 7주차까지의 MCP 서버 구현 경험 (MCP가 Copilot에 "도구"를 제공하는 개념 이해)
- VS Code + GitHub Copilot 확장 설치 및 Agent Mode 사용 경험
- YAML 기본 문법 (키-값 쌍, 문자열 블록)
- Markdown 작성 경험

## 1회차: 강의

### 9.1 Skills란 무엇인가

4~7주차에 걸쳐 MCP 서버를 통해 Copilot에 새로운 "도구(tool)"를 부여하는 방법을 다루었다. MCP가 Copilot의 **행동 능력**을 확장한다면, 이번 주부터 다루는 Skills(스킬)는 Copilot의 **판단 기준**을 확장하는 기술이다. GitHub는 2025년 12월 18일에 Agent Skills를 공식 발표하였으며, 이를 통해 개발자는 Copilot에 도메인별 지식과 절차를 체계적으로 가르칠 수 있게 되었다.

스킬은 지시 사항(instructions), 스크립트(scripts), 참조 파일(references)을 하나의 폴더에 묶어 놓은 자기 완결적 패키지이다. Copilot은 사용자의 프롬프트가 특정 스킬과 관련이 있다고 판단할 때만 해당 스킬을 자동으로 불러온다. 이 자동 활성화 메커니즘이 기존의 Custom Instructions와 가장 큰 차이점이다.

#### 스킬이 필요한 이유

MCP 서버만으로는 해결할 수 없는 영역이 있다. 예를 들어, "코드 리뷰를 해 달라"는 요청에 MCP 서버가 할 수 있는 일은 없다. 코드 리뷰는 외부 API를 호출하거나 파일 시스템을 조작하는 작업이 아니라, 특정 기준에 따라 코드를 평가하고 피드백을 제시하는 "판단" 작업이기 때문이다. 이런 판단의 기준과 절차를 Copilot에 전달하는 것이 스킬의 역할이다.

일상적인 비유를 들면, MCP 서버는 장인에게 새 도구를 쥐어 주는 것이고, 스킬은 장인에게 작업 매뉴얼을 읽게 하는 것이다. 아무리 좋은 도구가 있어도 어떻게 사용해야 하는지 모르면 결과물의 품질을 보장할 수 없다. 도구(MCP)와 매뉴얼(Skills)이 함께 갖추어져야 비로소 일관된 품질의 결과를 기대할 수 있다.

#### 스킬의 작동 범위

스킬은 다양한 영역에 적용할 수 있다. 대표적인 활용 사례는 다음과 같다.

- **코딩 표준 적용**: 팀의 네이밍 규칙, 아키텍처 패턴, 에러 처리 방식을 Copilot에 학습시킨다
- **문서화 자동화**: API 문서, 릴리스 노트, 변경 이력의 작성 형식을 표준화한다
- **코드 리뷰 기준 설정**: 보안, 성능, 가독성 등 팀 고유의 리뷰 체크리스트를 적용한다
- **테스트 패턴 강제**: 팀이 사용하는 테스트 프레임워크, 픽스처 구조, 네이밍 규칙을 지시한다
- **워크플로우 절차 정의**: 배포 전 점검 사항, PR 작성 규칙 등의 절차를 체계화한다

### 9.2 Custom Instructions vs Agent Skills

기존에도 Copilot의 행동을 커스터마이징하는 방법은 있었다. `.github/copilot-instructions.md` 파일에 작성하는 Custom Instructions가 그것이다. Skills를 올바르게 이해하려면, 먼저 Custom Instructions와의 차이를 명확히 파악해야 한다.

#### Custom Instructions의 특성

Custom Instructions는 프로젝트 루트의 `.github/copilot-instructions.md` 파일에 작성한다. 이 파일에 적힌 내용은 Copilot이 수행하는 **모든 작업에 항상 적용**된다. "코드는 반드시 TypeScript로 작성한다", "주석은 한국어로 작성한다"처럼 프로젝트 전반에 걸치는 간단한 규칙을 지정할 때 적합하다.

```markdown
<!-- .github/copilot-instructions.md 예시 -->
- 코드는 Python 3.10+ 문법으로 작성한다
- 주석과 docstring은 한국어로 작성한다
- 변수명은 snake_case를 사용한다
- import 순서: 표준 라이브러리 → 서드파티 → 로컬 모듈
```

Custom Instructions의 한계는 명확하다. 첫째, 모든 상황에 항상 적용되므로 특정 작업에만 필요한 상세 지시를 넣기 어렵다. 둘째, 단일 마크다운 파일이므로 스크립트나 템플릿을 번들로 포함시킬 수 없다. 셋째, 내용이 길어지면 컨텍스트 윈도우를 불필요하게 소비하여 Copilot의 성능에 영향을 줄 수 있다.

#### Agent Skills의 특성

Agent Skills는 이러한 한계를 극복하기 위해 설계되었다. 스킬은 `.github/skills/` 디렉터리 아래에 각각 독립된 폴더로 존재하며, 관련 프롬프트가 들어올 때만 선택적으로 활성화된다. 하나의 스킬은 지시 파일(SKILL.md)뿐 아니라 스크립트, 템플릿, 참조 문서를 하나의 패키지로 묶을 수 있다.

**표 9.1** Custom Instructions와 Agent Skills 비교

| 특성 | Custom Instructions | Agent Skills |
|------|-------------------|--------------|
| 범위 | 모든 작업에 항상 적용 | 관련 프롬프트에만 자동 활성화 |
| 내용 수준 | 간단한 코딩 표준, 가이드라인 | 상세 지시 + 스크립트 + 참조 파일 |
| 활성화 방식 | 항상 로드 | 프롬프트 매칭 기반 자동 판단 |
| 파일 번들링 | 불가 (단일 .md 파일) | 가능 (스크립트, 템플릿, 문서) |
| 저장 위치 | `.github/copilot-instructions.md` | `.github/skills/` 또는 `~/.copilot/skills/` |
| 팀 공유 | 저장소 커밋으로 공유 | 저장소 커밋으로 공유 (프로젝트 스킬) |
| 컨텍스트 소비 | 항상 전체 내용 로드 | 필요한 시점에만 단계적 로드 |

#### 언제 무엇을 사용하는가

두 기능은 대체 관계가 아니라 보완 관계이다. 프로젝트 전반에 걸치는 기본 규칙(언어, 코딩 스타일, 네이밍 규칙)은 Custom Instructions에 두고, 특정 도메인이나 작업에 한정된 상세 절차(코드 리뷰 기준, 문서 작성 템플릿, 테스트 패턴)는 Skills로 분리하는 것이 권장되는 전략이다.

```
Custom Instructions  → "항상 지켜야 할 기본 규칙" (헌법)
Agent Skills         → "특정 상황에서 따를 상세 절차" (업무 매뉴얼)
```

### 9.3 SKILL.md 파일 구조

모든 스킬의 핵심은 SKILL.md 파일이다. 이 파일은 YAML 프론트매터(frontmatter)와 마크다운 본문의 두 부분으로 구성된다. 각 부분의 역할과 작성법을 상세히 살펴본다.

#### YAML 프론트매터: 메타데이터

프론트매터에는 두 가지 필수 필드가 있다.

**`name`** 필드는 스킬의 고유 식별자이다. 소문자와 하이픈만 사용하며 최대 64자까지 허용된다. 다른 스킬과 중복되지 않아야 하며, 스킬의 용도를 직관적으로 나타내는 이름을 선택해야 한다.

**`description`** 필드는 스킬의 용도를 설명하는 문장으로, 최대 1,024자까지 작성할 수 있다. Copilot은 이 description을 읽고 스킬 활성화 여부를 결정하므로, 이 필드의 품질이 스킬의 효용을 좌우한다. 어떤 상황에서 이 스킬이 필요한지를 명확하고 구체적으로 서술해야 한다.

```yaml
---
name: code-review
description: >
  코드 리뷰를 요청하거나, PR을 검토하거나,
  코드 품질을 확인할 때 사용한다.
  팀의 코딩 표준과 보안 체크리스트에 따라 리뷰한다.
---
```

description 작성에서 흔히 하는 실수와 올바른 접근을 비교하면 다음과 같다.

| 나쁜 예 | 좋은 예 |
|---------|--------|
| "유용한 스킬이다" | "코드 리뷰를 요청하거나, PR을 검토하거나, 코드 품질을 확인할 때 사용한다" |
| "문서를 작성한다" | "REST API 엔드포인트를 문서화하거나, OpenAPI 명세를 생성할 때 사용한다" |
| "테스트 관련 스킬" | "단위 테스트를 작성하거나, 테스트 커버리지를 높이거나, 테스트를 추가할 때 사용한다" |

핵심 원칙은 **사용자가 실제로 입력할 법한 키워드와 문맥**을 description에 충분히 포함시키는 것이다. Copilot은 프론트매터의 description과 사용자 프롬프트 사이의 의미적 유사도를 기반으로 활성화를 판단하기 때문이다.

#### 마크다운 본문: 지시 사항

프론트매터 아래에는 마크다운 형식으로 Copilot이 따라야 할 지시 사항을 작성한다. 본문에 포함할 수 있는 내용은 다음과 같다.

- **작업 절차**: 단계별로 수행할 작업의 순서
- **준수 사항**: 반드시 지켜야 할 규칙과 제약 조건
- **출력 형식**: 결과물의 구조와 포맷
- **예시**: 좋은 예와 나쁜 예를 통한 구체적 안내
- **참조 지시**: 번들 파일을 언제, 어떻게 참조해야 하는지

본문 작성 시 주의할 점은 Copilot의 컨텍스트 윈도우 한계를 고려해야 한다는 것이다. 핵심적인 내용에 집중하고, 상세한 참조 자료는 별도 파일로 분리하여 번들에 포함시키는 것이 효율적이다. 이 전략은 뒤에서 다룰 3단계 점진적 공개 모델과 직접 관련된다.

#### 스킬 폴더 디렉터리 구조

하나의 스킬은 다음과 같은 폴더 구조를 갖는다.

```
my-skill/
  SKILL.md           # 핵심 지시 파일 (필수)
  scripts/           # 자동화 유틸리티 (선택)
  references/        # API 문서, 체크리스트 (선택)
  assets/            # 템플릿, 코드 샘플 (선택)
  examples/          # 예시 구현, 좋은/나쁜 코드 (선택)
```

SKILL.md만 필수이며 나머지 하위 폴더는 모두 선택 사항이다. 그러나 스크립트나 템플릿을 번들로 포함시킬 수 있다는 점이 Custom Instructions 대비 스킬의 가장 강력한 장점이다. 예를 들어, 코드 리뷰 스킬에 보안 체크리스트 문서를 `references/security-checklist.md`로 번들하면, Copilot이 리뷰 수행 시 이 체크리스트를 실제로 참조하여 검토한다.

#### 저장 위치: 프로젝트 스킬 vs 퍼스널 스킬

스킬의 저장 위치는 두 곳이 가능하다.

**프로젝트 스킬(Project Skills)**은 프로젝트 루트의 `.github/skills/` 디렉터리 아래에 둔다. 저장소에 커밋되므로 모든 팀원이 동일한 스킬을 공유하게 된다. 팀 표준으로 합의된 스킬을 이곳에 저장한다.

**퍼스널 스킬(Personal Skills)**은 홈 디렉터리의 `~/.copilot/skills/` 아래에 둔다. 해당 사용자의 모든 프로젝트에서 사용할 수 있지만, 다른 사람과 공유되지 않는다. 개인적인 선호나 실험적인 스킬을 이곳에서 관리한다.

두 위치에 같은 이름의 스킬이 존재할 경우, **프로젝트 스킬이 퍼스널 스킬보다 우선**한다. 따라서 팀 표준을 프로젝트 스킬로 설정하고, 개인 선호 사항은 퍼스널 스킬로 보완하는 이중 구조가 가능하다.

```
.github/skills/           ← 팀 공유 (프로젝트 스킬, 우선순위 높음)
├── code-review/
│   └── SKILL.md
└── api-docs/
    └── SKILL.md

~/.copilot/skills/        ← 개인 전용 (퍼스널 스킬, 우선순위 낮음)
├── my-shortcuts/
│   └── SKILL.md
└── personal-review/
    └── SKILL.md
```

스킬 관리 전략으로는, 프로젝트 스킬에는 팀이 합의한 핵심 스킬만 두고, 실험적 스킬은 퍼스널 영역에서 충분히 검증한 후 팀 논의를 거쳐 승격시키는 방식을 권장한다.

### 9.4 3단계 점진적 공개 모델

Copilot이 스킬을 처리하는 과정은 3단계 점진적 공개(3-tier progressive disclosure) 모델을 따른다. 이 모델을 이해하면 스킬의 구조를 왜 이렇게 설계했는지, 그리고 효율적인 스킬을 어떻게 작성해야 하는지를 파악할 수 있다.

#### 1단계: 발견 (Discovery)

Copilot은 프로젝트에 등록된 모든 스킬의 **이름(name)과 설명(description)만** 미리 읽어 둔다. 이 정보는 매우 가볍기 때문에 스킬이 수십 개 등록되어 있어도 컨텍스트 윈도우에 부담을 주지 않는다.

이 단계에서 Copilot은 "어떤 스킬들이 있는가"만 파악하고 있으며, 각 스킬의 상세 내용은 아직 읽지 않은 상태이다. 따라서 description의 품질이 스킬 활성화의 성패를 결정한다.

#### 2단계: 지시 로딩 (Instruction Loading)

사용자의 프롬프트가 특정 스킬의 description과 의미적으로 일치한다고 판단되면, Copilot은 해당 **SKILL.md의 전체 본문**을 읽어 들인다. 이 시점에서 본문에 작성된 작업 절차, 준수 사항, 출력 형식 등이 Copilot의 컨텍스트에 추가된다.

이 단계에서 중요한 것은, 관련 없는 스킬의 본문은 여전히 로드되지 않는다는 점이다. "코드 리뷰해 줘"라는 프롬프트에 대해 API 문서 생성 스킬의 본문까지 읽어 들이는 낭비가 발생하지 않는다.

#### 3단계: 리소스 접근 (Resource Access)

본문의 지시에 따라 필요한 경우에만 번들된 스크립트나 템플릿 파일을 참조한다. 예를 들어, 코드 리뷰 스킬의 본문에 "보안 검토 시 `references/security-checklist.md`를 참조한다"라고 명시되어 있으면, 보안 관련 리뷰를 수행할 때만 해당 파일을 읽는다.

```
[1단계] 발견      → name + description만 로드 (항상, 가벼움)
    ↓ 프롬프트 매칭
[2단계] 지시 로딩  → SKILL.md 본문 전체 로드 (매칭 시에만)
    ↓ 본문 지시에 따라
[3단계] 리소스 접근 → 번들 파일 참조 (필요시에만)
```

이 3단계 구조 덕분에 수십 개의 스킬을 등록해 두어도 실제로 컨텍스트 윈도우를 소비하는 것은 활성화된 스킬의 내용뿐이다. 이는 "필요한 것만, 필요한 시점에" 로드하는 지연 로딩(lazy loading) 패턴과 동일한 원리이다.

#### 실무 시사점

3단계 모델이 스킬 설계에 주는 실무적 시사점은 다음과 같다.

1. **description에 투자하라**: 1단계에서 활성화 여부가 결정되므로, description이 부실하면 아무리 좋은 본문을 작성해도 스킬이 발동하지 않는다
2. **본문은 핵심에 집중하라**: 2단계에서 본문 전체가 로드되므로, 불필요하게 긴 본문은 컨텍스트를 낭비한다
3. **상세 자료는 번들로 분리하라**: 3단계에서 필요한 시점에만 로드되므로, 체크리스트나 템플릿은 별도 파일로 분리하는 것이 효율적이다

### 9.5 첫 번째 스킬 작성: 전체 흐름 미리보기

강의의 마지막으로, 2회차 실습에서 작성할 코드 리뷰 스킬의 전체 흐름을 미리 살펴본다. 이를 통해 앞서 배운 개념이 실제로 어떻게 적용되는지 전체 그림을 파악한다.

#### 스킬 생성 흐름

```
[1] 요구 분석    → 어떤 문제를 해결할 것인가? (코드 리뷰 기준의 불일치)
      ↓
[2] 구조 설계    → 폴더 구조, 번들 파일 결정
      ↓
[3] 프론트매터   → name과 description 작성 (활성화 키워드 선정)
      ↓
[4] 본문 작성    → 절차, 규칙, 출력 형식 정의
      ↓
[5] 번들 작성    → 참조 파일(체크리스트, 템플릿) 작성
      ↓
[6] 테스트       → Copilot Chat에서 관련 프롬프트로 동작 검증
      ↓
[7] 반복 개선    → description 조정, 본문 보완, 번들 추가
```

이 흐름은 소프트웨어 개발의 반복적 개선 과정과 동일하다. 한 번에 완벽한 스킬을 작성하기보다, 기본 골격을 만든 후 테스트와 개선을 반복하는 것이 현실적이다.

#### 테스트 방법

스킬이 올바르게 동작하는지 확인하는 기본적인 테스트 절차는 다음과 같다.

1. **활성화 테스트**: Copilot Chat에서 스킬과 관련된 프롬프트를 입력하고, 스킬이 활성화되는지 확인한다. 활성화되지 않으면 description의 키워드를 조정한다.
2. **지시 준수 테스트**: 활성화된 스킬이 본문의 절차와 규칙을 정확히 따르는지 확인한다. 절차가 생략되거나 규칙이 무시되면 본문의 지시를 더 명시적으로 수정한다.
3. **출력 형식 테스트**: 결과물이 정의한 출력 형식을 따르는지 확인한다. 형식이 일관되지 않으면 본문에 출력 예시를 추가한다.
4. **번들 참조 테스트**: 번들 파일이 실제로 참조되어 결과에 반영되는지 확인한다.

## 2회차: 실습

### 실습 1: 코드 리뷰 스킬 작성

코드 리뷰는 모든 개발 팀에서 수행하지만, 리뷰의 기준과 깊이는 팀마다 다르다. 한 팀은 보안 취약점을 최우선으로 점검하고, 다른 팀은 성능 최적화에 초점을 맞추며, 또 다른 팀은 가독성과 유지보수성을 가장 중요하게 여긴다. 이런 팀 고유의 리뷰 기준을 스킬로 정의하면, Copilot이 팀의 관점에서 일관된 리뷰를 수행한다.

#### 단계 1: 디렉터리 생성

프로젝트 루트에서 스킬 디렉터리를 생성한다.

```bash
mkdir -p .github/skills/code-review/references
```

#### 단계 2: SKILL.md 작성

`.github/skills/code-review/SKILL.md` 파일을 다음과 같이 작성한다.

```yaml
---
name: code-review
description: >
  코드 리뷰를 요청하거나, PR을 검토하거나,
  코드 품질을 확인할 때 사용한다.
  팀의 코딩 표준과 보안 체크리스트에 따라 리뷰한다.
---

## 리뷰 절차

코드 리뷰 요청을 받으면 다음 순서로 검토한다.

### 1단계: 보안 검토
- 하드코딩된 시크릿(API 키, 비밀번호, 토큰)이 없는지 확인한다
- SQL 인젝션, XSS, CSRF 취약점을 점검한다
- 사용자 입력에 대한 검증(validation)과 이스케이프(escape) 처리를 확인한다
- 상세 항목은 `references/security-checklist.md`를 참조한다

### 2단계: 성능 검토
- O(n^2) 이상의 중첩 루프를 식별한다
- 불필요한 데이터베이스 호출이나 N+1 쿼리를 점검한다
- 메모리 누수 가능성이 있는 패턴(닫히지 않은 리소스 등)을 확인한다

### 3단계: 가독성 검토
- 함수 길이가 50줄을 초과하지 않는지 확인한다
- 변수명과 함수명이 역할을 명확히 나타내는지 점검한다
- 매직 넘버 사용 여부를 확인한다

### 4단계: 구조 검토
- 단일 책임 원칙(SRP)을 준수하는지 확인한다
- 적절한 에러 처리가 구현되어 있는지 점검한다
- 테스트 가능한 구조인지 확인한다

## 출력 형식

리뷰 결과는 다음 형식으로 출력한다.

### [심각도: 높음/중간/낮음] 항목 제목
- **위치**: 파일명:줄번호
- **문제**: 발견된 문제 설명
- **제안**: 구체적인 수정 방안
- **근거**: 왜 이 수정이 필요한지 설명

## 추가 규칙
- 리뷰는 비판이 아니라 개선 제안이다. 긍정적 피드백도 포함한다
- 수정 제안은 구체적이어야 한다. "개선이 필요하다"가 아니라 수정된 코드를 제시한다
```

#### 단계 3: 보안 체크리스트 번들 작성

`references/security-checklist.md` 파일에 상세 보안 검토 항목을 작성한다.

```markdown
# 보안 체크리스트

## 인증/인가
- [ ] 하드코딩된 비밀번호, API 키, 토큰이 없는가
- [ ] 환경 변수 또는 시크릿 관리자를 사용하는가
- [ ] 적절한 인증 미들웨어가 적용되어 있는가

## 입력 검증
- [ ] 모든 사용자 입력에 대해 타입과 범위를 검증하는가
- [ ] SQL 쿼리에 파라미터 바인딩을 사용하는가
- [ ] HTML 출력에 이스케이프 처리가 적용되어 있는가

## 데이터 보호
- [ ] 민감 데이터가 로그에 기록되지 않는가
- [ ] HTTPS를 강제하는가
- [ ] CORS 설정이 적절한가
```

#### 단계 4: 동작 테스트

Copilot Chat에서 다음 프롬프트로 스킬이 활성화되는지 테스트한다.

- "이 파일을 리뷰해 줘"
- "PR을 검토해 줘"
- "코드 품질을 확인해 줘"

Copilot이 보안, 성능, 가독성, 구조 항목을 순서대로 점검하며 정의한 출력 형식(심각도, 위치, 문제, 제안, 근거)을 따른다면 스킬이 정상적으로 활성화된 것이다.

만약 스킬이 활성화되지 않는다면 description의 키워드가 프롬프트와 충분히 매칭되지 않을 가능성이 높다. description에 "파일을 검토", "코드를 분석" 등 추가 키워드를 보강해 본다.

### 실습 2: API 문서화 스킬 작성

두 번째 실습에서는 REST API 엔드포인트 문서를 자동으로 생성하는 스킬을 만든다. 이 스킬은 번들 파일(템플릿)의 활용법을 보여 주는 좋은 예시이다.

#### 단계 1: 디렉터리 생성

```bash
mkdir -p .github/skills/api-docs/assets
```

#### 단계 2: SKILL.md 작성

`.github/skills/api-docs/SKILL.md` 파일을 작성한다.

```yaml
---
name: api-docs
description: >
  API 문서를 생성하거나, 엔드포인트를 문서화하거나,
  REST API 명세를 작성할 때 사용한다.
  OpenAPI 형식의 템플릿을 기반으로 문서화한다.
---

## 문서화 절차

API 문서 작성 요청을 받으면 다음 절차를 따른다.

### 1단계: 엔드포인트 식별
- 대상 파일에서 라우트 데코레이터(@app.route, @router, @api_view 등)를 식별한다
- 각 엔드포인트의 HTTP 메서드(GET, POST, PUT, DELETE)를 파악한다

### 2단계: 정보 추출
- 코드의 타입 힌트, Pydantic 모델, docstring에서 요청/응답 스키마를 추출한다
- 인증 요구사항, 권한 수준을 파악한다
- 가능한 에러 응답 코드를 식별한다

### 3단계: 문서 생성
- 각 엔드포인트에 `assets/endpoint-template.md`의 형식을 적용한다
- 추출한 정보로 템플릿의 각 섹션을 채운다
- 요청 예시와 응답 예시를 포함한다

## 출력 규칙
- 모든 엔드포인트를 HTTP 메서드 순서(GET → POST → PUT → DELETE)로 정렬한다
- 요청 파라미터에는 반드시 타입, 필수 여부, 설명을 포함한다
- 응답 스키마에는 상태 코드별 응답 구조를 명시한다
- 에러 응답은 최소 400, 401, 404, 500을 포함한다
```

#### 단계 3: 엔드포인트 템플릿 번들 작성

`assets/endpoint-template.md` 파일에 문서 형식을 정의한다.

```markdown
## [METHOD] /path

**설명**: 이 엔드포인트의 역할을 한 문장으로 서술한다.

### 요청

**헤더**

| 헤더 | 값 | 필수 | 설명 |
|------|-----|:----:|------|
| Authorization | Bearer {token} | O | 인증 토큰 |

**파라미터** (Path / Query / Body)

| 이름 | 타입 | 필수 | 설명 | 예시 |
|------|------|:----:|------|------|
| id | integer | O | 리소스 고유 식별자 | 42 |

**요청 예시**

```json
{
  "key": "value"
}
```

### 응답

**성공 (200)**

```json
{
  "data": {}
}
```

**에러**

| 상태 코드 | 설명 |
|:---------:|------|
| 400 | 잘못된 요청 |
| 401 | 인증 실패 |
| 404 | 리소스 없음 |
| 500 | 서버 오류 |
```

#### 단계 4: 테스트용 Flask 앱 준비

스킬 테스트를 위해 간단한 Flask 애플리케이션을 작성한다.

```python
from flask import Flask, jsonify, request

app = Flask(__name__)

@app.route("/api/users", methods=["GET"])
def get_users():
    """사용자 목록을 반환한다."""
    return jsonify({"users": []})

@app.route("/api/users/<int:user_id>", methods=["GET"])
def get_user(user_id: int):
    """특정 사용자 정보를 반환한다."""
    return jsonify({"id": user_id, "name": "홍길동"})

@app.route("/api/users", methods=["POST"])
def create_user():
    """새 사용자를 생성한다."""
    data = request.get_json()
    return jsonify({"id": 1, **data}), 201
```

이 파일을 열어 둔 상태에서 Copilot Chat에 "이 앱의 API 문서를 작성해 줘"라고 요청한다. Copilot이 `assets/endpoint-template.md`의 형식을 따라 각 엔드포인트를 문서화한다면 스킬이 올바르게 작동하는 것이다.

#### 스킬 있는 경우와 없는 경우 비교

이 실습에서 주목할 점은, 스킬이 단순한 텍스트 지시를 넘어 참조 파일과 결합될 때 얼마나 강력해지는가이다. 템플릿 파일이 없다면 Copilot은 자체 판단에 따라 문서 형식을 매번 다르게 결정할 수 있지만, 템플릿이 있으면 팀이 합의한 정확한 형식을 일관되게 재현한다. 이 일관성이야말로 스킬의 핵심 가치이다.

## 핵심 정리

- Skills는 Copilot에 도메인 지식과 절차를 가르치는 자기 완결적 패키지이다. MCP가 "도구"를 제공한다면, Skills는 "판단 기준"을 제공한다.
- Custom Instructions는 모든 작업에 항상 적용되는 기본 규칙이고, Skills는 특정 상황에서만 활성화되는 상세 절차이다. 두 기능은 대체가 아니라 보완 관계이다.
- SKILL.md 파일은 YAML 프론트매터(name, description)와 마크다운 본문으로 구성된다. description의 품질이 스킬 활성화 성패를 결정한다.
- Copilot은 발견(Discovery) - 지시 로딩(Instruction Loading) - 리소스 접근(Resource Access)의 3단계로 스킬을 처리한다. 이 구조 덕분에 수십 개의 스킬을 등록해도 컨텍스트 낭비가 없다.
- 스킬 폴더에는 SKILL.md 외에 scripts, references, assets, examples를 번들로 포함시킬 수 있으며, 이것이 Custom Instructions 대비 가장 강력한 장점이다.
- `.github/skills/`(프로젝트 스킬)에 저장하면 Git으로 버전 관리되며 팀 전체가 공유한다. `~/.copilot/skills/`(퍼스널 스킬)는 개인 전용이다.

## 예시 문제

### 이론 -- 단답형

**문제 1.** MCP의 3대 프리미티브(Primitive)를 모두 나열하고, 각각을 한 문장으로 설명하시오. (6점)

> **모범 답안**
> - **Tool(도구)**: AI 에이전트가 외부 세계에 영향을 미치는 행동을 수행하는 기능이다.
> - **Resource(리소스)**: AI 에이전트가 읽기 전용으로 접근할 수 있는 데이터 소스이다.
> - **Prompt(프롬프트)**: 자주 사용하는 프롬프트 패턴을 서버 측에서 제공하는 재사용 가능한 템플릿이다.

### 이론 -- 서술형

**문제 2.** "도서관 좌석 현황 조회"와 "도서관 좌석 예약"을 MCP 서버에 구현한다고 할 때, 각각을 Tool과 Resource 중 어느 것으로 설계해야 하는지 근거와 함께 설명하시오. (5점)

> **모범 답안**
> "좌석 현황 조회"는 Resource로 설계한다. 현재 좌석 상태를 읽기 전용으로 반환할 뿐 외부 상태를 변경하지 않기 때문이다. URI는 `library://seats/status`와 같이 정의할 수 있다.
> "좌석 예약"은 Tool로 설계한다. 예약 행위는 좌석 상태를 변경하는 부수 효과(side effect)를 유발하므로 에이전트가 호출하는 행동(action)에 해당한다. 학번과 좌석 번호를 입력 파라미터로 받도록 JSON Schema를 정의한다.

**문제 3.** Agent Mode의 자율 실행 루프 4단계를 순서대로 기술하고, "단순한 코드 자동 완성"과 어떻게 다른지 설명하시오. (5점)

> **모범 답안**
> 자율 실행 루프는 (1) 계획: 요청을 분석하여 작업 계획 수립, (2) 편집: 코드 생성/수정, (3) 실행: 터미널 명령 실행 및 결과 확인, (4) 수정: 오류 감지 시 코드 수정 후 재실행의 4단계이다.
> 코드 자동 완성은 커서 위치에서 다음 코드를 예측하여 제안하는 데 그치지만, Agent Mode는 전체 프로젝트를 파악하고 여러 파일을 동시에 수정하며 터미널 명령을 실행하고 오류를 자율적으로 수정하는 루프를 반복한다.

### 이론 -- 다이어그램 완성

**문제 4.** 다음 MCP 아키텍처 다이어그램의 빈칸 (가)~(라)에 들어갈 용어를 채우시오. (6점)

```
┌─────────────────────────────┐
│         ( 가 )               │  ← 사용자가 상호작용하는 애플리케이션
│  ┌───────────────────────┐  │
│  │      ( 나 )            │  │  ← 서버와 통신을 담당하는 컴포넌트
│  └──────────┬────────────┘  │
└─────────────┼───────────────┘
              │ ( 다 ) 전송
┌─────────────┼───────────────┐
│      ( 라 )                  │  ← 도구, 리소스, 프롬프트를 제공
│  ┌─────┐ ┌─────┐ ┌─────┐   │
│  │Tool │ │Res. │ │Prom.│   │
│  └─────┘ └─────┘ └─────┘   │
└─────────────────────────────┘
```

> **모범 답안**: (가) Host, (나) MCP Client, (다) JSON-RPC, (라) MCP Server

### 실기 -- MCP 서버 구현

**문제 5.** 다음 REST API 명세를 읽고 MCP 서버로 래핑하시오. (60점)

**API 명세: 대학교 학식 메뉴 API** (Base URL: `https://api.example.ac.kr/v1`, 인증: `X-API-Key` 헤더)

**엔드포인트 1 -- 오늘의 메뉴 조회**: `GET /cafeteria/{cafeteria_id}/menu/today`

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|:----:|------|
| cafeteria_id | string | O | 식당 식별자 (`main`, `dorm`, `faculty`) |
| meal_type | string (query) | X | 식사 유형 (`breakfast`, `lunch`, `dinner`). 생략 시 전체 |

```json
{ "cafeteria": "main", "date": "2026-02-11",
  "meals": [{ "type": "lunch", "menu": ["된장찌개", "제육볶음", "샐러드", "밥"],
              "price": 5000, "calories": 850 }] }
```

**엔드포인트 2 -- 식당 혼잡도 조회**: `GET /cafeteria/{cafeteria_id}/congestion`

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|:----:|------|
| cafeteria_id | string | O | 식당 식별자 |

```json
{ "cafeteria": "main", "current_count": 142, "capacity": 200,
  "congestion_level": "moderate", "wait_time_minutes": 5 }
```

**요구사항**

1. FastMCP를 사용하여 구현한다.
2. 메뉴 조회는 **Tool**로, 혼잡도 조회는 **Resource**로 구현하고, 선택 근거를 주석으로 기술한다.
3. API Key는 환경 변수(`CAFETERIA_API_KEY`)에서 읽어온다.
4. 에러 처리: (a) 잘못된 cafeteria_id, (b) API 응답 타임아웃, (c) API Key 미설정.
5. 도구 설명은 LLM이 호출 여부를 정확히 판단할 수 있도록 작성한다.

**기대하는 코드 구조**

```python
from mcp.server.fastmcp import FastMCP
import httpx, os

mcp = FastMCP("university-cafeteria")
# --- 여기에 도구와 리소스를 구현 ---
if __name__ == "__main__":
    mcp.run(transport="stdio")
```

**채점 포인트**: `@mcp.tool()`/`@mcp.resource()` 데코레이터 사용, description 명확성, 타입 힌트와 기본값 정의, `httpx.AsyncClient` 비동기 호출, `try-except`로 오류 유형별 처리, API Key 검증, PEP 8 준수 및 한국어 주석

## 2회차: 시험 해설

### 주요 오답 패턴 분석

- **Tool과 Resource 혼동**: 상태를 변경하지 않는 조회를 Tool로, 상태 변경 기능을 Resource로 구현하는 사례가 빈번하다. "외부 상태를 변경하는가"라는 판단 기준을 재강조한다.
- **도구 설명 부실**: description을 "메뉴를 조회한다" 수준으로 작성하여 LLM이 호출 시점을 판단할 수 없는 경우가 있다.
- **에러 처리 누락**: 정상 경로만 구현하고 네트워크 오류, 잘못된 입력에 대한 처리를 빠뜨리는 경우가 흔하다.
- **Agent Mode 루프 이해 부족**: 자율 실행 루프의 반복적 특성을 단순한 코드 생성으로 축소하여 설명하는 답안이 있다.

### 모범 답안 해설

이론과 실기의 모범 답안을 검토한다. 실기는 다양한 올바른 구현 패턴을 비교하며 설명한다.

### 후반부 학습 안내

- **9~10주차 (Skills)**: SKILL.md로 Copilot에 도메인 지식과 절차를 주입한다. MCP가 "무엇을 할 수 있는가"라면 Skills는 "어떻게 해야 하는가"를 정의한다.
- **11주차 (통합)**: MCP + Skills를 결합하여 업무 자동화 에이전트를 구축한다.
- **12주차 (Coding Agent)**: 이슈에서 PR까지의 팀 워크플로우를 자동화한다.
- **13~15주차 (프로젝트)**: 전 과정을 종합한 최종 프로젝트를 설계, 구현, 발표한다.

## 학습 가이드

### 이론 준비

1. **핵심 개념 정리**: 각 주차 학습목표를 자신의 말로 설명할 수 있는지 확인한다. MCP 3계층 구조와 3대 프리미티브는 다이어그램을 직접 그려 본다.
2. **비교 항목 정리**: Ask/Edit/Agent 모드, Tool/Resource/Prompt 구분, stdio/HTTP 전송 방식을 표로 정리한다.
3. **설계 판단 연습**: 임의의 기능을 Tool로 구현할지 Resource로 구현할지 판단하는 연습을 반복한다.

### 실기 준비

1. **날씨 MCP 서버 복습**: 6주차 실습 코드를 보지 않고 처음부터 다시 작성해 본다.
2. **새로운 API로 연습**: 공개 REST API(JSONPlaceholder, PokeAPI 등)를 MCP 서버로 래핑하는 연습을 한다.
3. **에러 처리 패턴 숙달**: `httpx.ConnectError`, `httpx.TimeoutException`, HTTP 오류 응답 처리 패턴을 손에 익힌다.
4. **시간 배분 연습**: 60분 기준으로 API 분석(5분) - 설계(10분) - 구현(25분) - 에러 처리(15분) - 테스트(5분)를 연습한다.

### 지참 사항

- 개인 노트북 (VS Code, Python 3.10+, MCP SDK 설치 완료)
- GitHub Copilot 활성화 상태 (Agent Mode 사용 가능)
- 인터넷 접속 가능 (API 호출 테스트용)
- 별도의 참고 자료 지참은 불허한다 (오픈 북 아님)

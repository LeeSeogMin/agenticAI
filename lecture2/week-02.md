# 2주차. Copilot Agent Mode 기초

> **1회차** (강의 90분): Agent Mode의 개념, 세 가지 모드 비교, 자율 실행 루프의 동작 원리
> **2회차** (실습 90분): 자연어로 Flask Todo 웹앱 생성, 실행 결과 검증, 생성 코드 분석

---

## 학습목표

- GitHub Copilot의 에이전트 모드(Agent Mode)가 무엇인지 이해하고, 기존 모드와의 차이를 설명할 수 있다
- 에이전트 모드의 자율 실행 루프(계획 → 편집 → 실행 → 수정)의 각 단계를 구분할 수 있다
- 자연어 명세 한 문장으로 웹 애플리케이션을 생성하고, 사람이 검증해야 하는 지점을 식별할 수 있다

## 선수지식

- 1주차에서 구축한 VS Code + GitHub Copilot 환경이 정상 동작하는 상태여야 함
- Python 또는 JavaScript 기초 문법을 알고 있으면 충분하며, Flask 프레임워크 경험은 불필요
- GitHub 계정에 Copilot Pro(Student Developer Pack 포함)가 활성화되어 있어야 함
- 터미널(명령줄)에서 기본 명령어를 실행한 경험이 있으면 실습 진행에 도움됨

## 1회차: 강의

### 2.1 에이전트 모드란 무엇인가

- GitHub Copilot은 2021년 기술 프리뷰 이후 꾸준히 진화해 옴
- 초기에는 편집기 안에서 다음 줄을 제안하는 **인라인 자동완성(Inline Completion)**이 전부였음
  - 개발자가 주석이나 함수 시그니처를 작성하면, Copilot이 나머지 구현부를 한 줄씩 제안하는 방식
  - 이 단계에서 Copilot은 타이핑을 줄여 주는 보조 도구에 가까웠음

- 이후 **Copilot Chat**이 추가되면서 코드에 관한 질문과 답변이 가능해짐
  - 개발자는 "이 함수가 무엇을 하는가?"라고 물을 수 있게 됨
  - Copilot은 코드의 의미를 설명하거나 버그를 찾아 제안 가능
  - 그러나 이 단계에서도 Copilot의 동작 범위는 사용자가 지정한 파일이나 코드 조각에 한정됨

- 2025년 중반 정식 출시(GA)된 **에이전트 모드(Agent Mode)**는 이 흐름의 결정적 전환점
- 에이전트 모드의 핵심은 **자율성(Autonomy)**에 있음
  - 기존 모드: 사용자가 지정한 파일의 특정 위치에 코드를 제안하는 수동적 역할
  - 에이전트 모드: 사용자가 자연어로 목표를 기술하면 Copilot이 스스로 관련 파일을 탐색하고, 하위 작업을 계획하며, 여러 파일에 걸친 편집을 수행하고, 터미널 명령을 실행하며, 오류 발생 시 자동으로 수정을 시도함
  - 이 전체 과정이 하나의 **자율 실행 루프(Agentic Loop)**로 동작함

- **"도구"와 "에이전트"의 차이**를 통해 이 변화의 의미를 이해할 수 있음
  - **도구(tool)**: 사용자가 지시한 작업을 수행함
    - 사용자가 망치를 들어 못을 치듯이, 인라인 자동완성은 사용자가 의도한 코드를 완성하는 도구
  - **에이전트(agent)**: 목표를 부여받으면 그 목표를 달성하기 위한 행동을 스스로 계획하고 실행함
    - "Flask로 Todo 앱을 만들어 줘"라는 목표를 받으면, 어떤 파일을 만들어야 하는지, 어떤 라이브러리를 설치해야 하는지, 어떤 순서로 작업해야 하는지를 스스로 결정함

- 이러한 자율성은 소프트웨어 개발에서 매우 큰 **패러다임 전환**
  - 개발자의 역할이 "코드를 작성하는 사람"에서 "목표를 정의하고 결과를 검증하는 사람"으로 이동
  - 단, 이 전환이 완전하지는 않음
  - 에이전트 모드가 모든 작업을 완벽하게 수행하지는 못하며, 사람의 판단이 여전히 필수적인 지점이 존재함
  - 이 강의의 목표는 바로 그 지점을 정확히 파악하는 것

### 2.2 세 가지 모드 비교: Ask, Edit, Agent

- VS Code에서 Copilot Chat 패널을 열면 세 가지 상호작용 모드를 선택 가능
- 각 모드는 설계 목적과 동작 범위가 다르며, 상황에 따라 적절한 모드를 선택하는 것이 Copilot 활용의 핵심 역량

**표 2.1** Copilot Chat의 세 가지 상호작용 모드

| 모드 | 역할 | 동작 범위 | 파일 변경 | 터미널 실행 |
|------|------|----------|----------|------------|
| Ask(질문) | 코드에 관한 질문에 답변한다 | 읽기 전용, 파일 변경 없음 | 불가 | 불가 |
| Edit(편집) | 지정한 파일에 변경을 적용한다 | 사용자가 지정한 파일 범위 | 가능 | 불가 |
| Agent(에이전트) | 목표를 해석하여 자율적으로 다단계 작업을 수행한다 | 프로젝트 전체, 터미널 포함 | 가능 | 가능 |

- **Ask 모드**: 이해를 돕는 질문에 적합
  - "이 함수가 무엇을 하는가?", "이 코드에서 발생할 수 있는 버그는 무엇인가?", "이 알고리즘의 시간 복잡도는 얼마인가?" 같은 질문에 정확한 답변 제공
  - 파일을 변경하지 않으므로 안전하게 사용 가능
  - 기존 코드를 분석하거나 학습할 때 가장 유용
  - 코드 리뷰 과정에서 다른 사람이 작성한 코드의 의도를 빠르게 파악하는 용도로도 효과적

- **Edit 모드**: 범위가 명확한 편집에 효과적
  - "이 파일의 변수명을 카멜케이스로 바꿔 줘", "이 함수에 타입 힌트를 추가해 줘", "이 클래스에 __repr__ 메서드를 구현해 줘"처럼 변경 대상과 변경 내용이 구체적인 경우에 사용
  - 사용자가 지정한 파일 범위 안에서만 동작하므로, 의도하지 않은 다른 파일의 변경 방지 가능
  - 한 파일의 세부 사항을 정교하게 조정할 때 가장 적합

- **Agent 모드**: 여러 파일 생성과 실행이 필요한 복합 작업에서 진가를 발휘
  - "Flask로 할 일 목록 웹앱을 만들어 줘", "이 프로젝트에 사용자 인증 기능을 추가해 줘", "테스트 코드를 작성하고 실행해 줘"처럼 여러 파일을 넘나들며 실행까지 필요한 작업에 사용
  - Agent 모드만이 터미널 명령을 실행할 수 있으며, 이 능력이 "코드를 만드는 것"에서 "실행 가능한 결과를 만드는 것"으로의 도약을 가능하게 함

- 세 모드는 **배타적 관계가 아니라 상호 보완적**
- 실무에서 가장 효과적인 워크플로우는 세 모드를 상황에 따라 전환하며 사용하는 것
- 대표적인 조합 패턴:
  1. 에이전트 모드로 프로젝트의 큰 틀을 잡음 -- "Flask REST API 프로젝트를 만들어 줘"처럼 전체 구조 생성
  2. Edit 모드로 특정 파일의 세부 사항을 조정 -- 에이전트가 생성한 코드에서 변수명이나 에러 처리 방식 수정
  3. Ask 모드로 코드의 동작 원리를 확인 -- 에이전트가 작성한 코드 중 이해가 가지 않는 부분을 질문
  - 이 순환을 반복하면서 프로젝트를 발전시키는 것이 AI 협업 개발의 기본 패턴

- **모드 전환 판단 기준**:
  - 현재 작업의 범위가 "프로젝트 전체"이면 Agent 모드
  - "특정 파일"이면 Edit 모드
  - "이해와 분석"이면 Ask 모드
  - 터미널 실행이 필요하면 반드시 Agent 모드
  - 이 판단이 자연스럽게 이루어지도록 반복 연습하는 것이 이번 주차의 목표

### 2.3 환경 확인과 모델 선택

- 1주차에서 기본 환경 설정을 완료했으므로, 이 절에서는 에이전트 모드에 특화된 설정 사항을 확인함

**VS Code 버전 확인.**

- 에이전트 모드는 VS Code 1.102 이상에서 동작함
- 최신 기능(Skills, MCP 연동 등)을 사용하려면 VS Code Insiders 채널 권장
- 상단 메뉴의 "Help > Check for Updates"에서 버전 확인 가능

**확장 프로그램 확인.**

- "GitHub Copilot"과 "GitHub Copilot Chat" 두 확장이 설치되어 있는지 확인
- VS Code 하단 상태바에 Copilot 아이콘이 나타나면 정상
- 아이콘에 경고 표시가 있다면 로그인 상태를 확인

**에이전트 모드 활성화 확인.**

- VS Code 설정(settings.json)에서 다음 항목이 활성화되어 있는지 확인

```json
{
  "chat.agent.enabled": true
}
```

- 이 설정이 완료되면 Copilot Chat 패널의 모드 선택 드롭다운에서 "Agent" 옵션이 나타남

**모델 선택.**

- Copilot은 여러 언어 모델을 지원함
- Copilot Chat 패널 상단의 모델 선택 드롭다운에서 Claude Sonnet 4, GPT-4o, Gemini 2.0 Flash 등 선택 가능
- 에이전트 모드에서 모델 선택은 작업 완성도에 직접적인 영향을 미침
  - 복잡한 다단계 작업: 추론 능력이 강한 모델을 선택하는 것이 좋음
  - 단순한 코드 생성: 빠른 모델이 효율적일 수 있음
- 모델 선택의 실질적인 차이를 이해하기 위해 동일한 프롬프트를 서로 다른 모델에 입력하고 결과를 비교하는 것이 유익함
  - 같은 "Flask Todo 앱을 만들어 줘"라는 요청이라도 모델에 따라 프로젝트 구조, 에러 처리 수준, 코드 스타일이 달라질 수 있음
- 이러한 차이를 관찰하고 자신의 작업 패턴에 맞는 모델을 선택하는 것도 AI 협업 역량의 일부

**흔히 발생하는 문제와 해결.**

1. Copilot 확장은 설치되었으나 로그인되지 않은 경우 하단 상태바의 Copilot 아이콘에 경고 표시가 나타남 -- VS Code 좌측 하단의 계정 아이콘을 클릭하여 GitHub 계정에 재로그인
2. 조직(Organization) 계정을 사용하는 경우 관리자가 Copilot 사용을 허용해야 함 -- 개인 계정으로 먼저 테스트 권장
3. 모델 선택 드롭다운이 보이지 않는 경우 -- VS Code 버전이 충분히 최신인지 확인

**표 2.2** 에이전트 모드 환경 설정 체크리스트

| 항목 | 확인 방법 | 정상 상태 |
|------|----------|----------|
| VS Code 버전 | Help > About | 1.102 이상 |
| Copilot 확장 | 확장 패널에서 검색 | "GitHub Copilot" + "GitHub Copilot Chat" 설치됨 |
| GitHub 로그인 | 하단 상태바 Copilot 아이콘 | 경고 없이 아이콘 표시 |
| Agent 모드 | Chat 패널 모드 드롭다운 | "Agent" 선택 가능 |
| 모델 선택 | Chat 패널 상단 드롭다운 | Claude Sonnet 4, GPT-4o 등 모델 목록 표시 |

### 2.4 자율 실행 루프의 내부 동작

- 에이전트 모드의 동작 원리를 이해하려면 **자율 실행 루프(Agentic Loop)**의 구조를 파악해야 함
- 이 루프는 목표 해석에서 시작하여 오류가 해소될 때까지 반복되는 피드백 순환

```
[목표 해석] → [컨텍스트 탐색] → [계획 수립] → [코드 편집] → [터미널 실행]
                                                                    ↓
                                                              {오류 감지}
                                                            ↙          ↘
                                                    [자동 수정]     [작업 완료]
                                                        ↓
                                                  [코드 편집] (반복)
```

**그림 2.1** 에이전트 모드의 자율 실행 루프

**1단계: 목표 해석(Goal Interpretation).**

- 사용자가 자연어로 입력한 요청을 분석하여 수행해야 할 작업의 범위와 완료 조건을 파악
  - "Flask로 Todo 앱을 만들어 줘"라는 요청에서 Copilot은 웹 프레임워크 선택(Flask), 데이터 저장소 구성, CRUD 기능 구현이라는 하위 목표를 도출
- 이 단계의 품질은 사용자가 제공하는 프롬프트의 명확성에 크게 좌우됨
- 모호한 요청은 모호한 결과를 낳음

**2단계: 컨텍스트 탐색(Context Exploration).**

- 에이전트는 현재 프로젝트의 파일 구조, 기존 코드, 설정 파일, 의존성 목록 등을 자율적으로 탐색
- 빈 프로젝트라면 처음부터 구조를 설계하고, 기존 코드가 있다면 그 패턴과 관례를 따르려 함
  - 프로젝트에 이미 `requirements.txt`가 존재하면 에이전트는 그 파일에 새로운 의존성을 추가하는 방식으로 동작
  - `.gitignore`나 `pyproject.toml`이 있다면 프로젝트의 성격을 파악하는 데 활용

**3단계: 계획 수립(Planning).**

- 에이전트는 하위 작업의 순서를 계획함 -- 파일 생성 순서, 의존성 설치 시점, 테스트 실행 시점 등을 결정
- 이 계획은 명시적으로 사용자에게 보여주는 경우도 있고, 내부적으로만 유지되는 경우도 있음
- 계획이 Chat 패널에 텍스트로 표시되는 경우에는 그 내용을 주의 깊게 읽어야 함
- 에이전트가 잘못된 방향으로 계획을 수립했다면, 실행 전에 수정 지시를 내리는 것이 실행 후 되돌리는 것보다 훨씬 효율적

**4단계: 코드 편집(Code Editing).**

- 계획에 따라 파일을 생성하고 코드를 작성함
- 에이전트는 한 번에 여러 파일을 생성하거나 수정 가능
- 이 과정에서 생성되는 파일은 Chat 패널에 diff 형태로 표시되며, 사용자는 각 변경을 수락하거나 거부 가능

**5단계: 터미널 실행(Terminal Execution).**

- `pip install`, `python app.py`, `pytest` 등의 터미널 명령을 실행
- 핵심적인 안전장치 존재: 에이전트 모드는 파일 편집은 자동으로 수행하지만, 터미널 명령은 반드시 **사용자 승인(User Approval)**이 필요
  - "Allow" 버튼을 눌러야 명령이 실행됨
- 이 설계는 의도하지 않은 시스템 변경(예: 잘못된 패키지 설치, 의도치 않은 프로세스 기동)을 방지하기 위한 것

**6단계: 오류 감지와 자동 수정(Error Detection & Self-Healing).**

- 에이전트 모드의 **가장 강력한 특성**
- 에이전트는 터미널 출력에서 컴파일러 오류, 린트(Lint) 경고, 테스트 실패, 런타임 예외 등의 오류 메시지를 읽고 해석
- 이 오류 정보가 다음 수정의 입력이 되며, 오류가 해소될 때까지 편집-실행-감지 루프가 반복됨
- 사람이 오류 메시지를 읽고 코드를 수정하는 과정을 에이전트가 자동으로 수행하기 때문에, 단순한 코드 생성을 넘어 "실행 가능한 결과"에 도달할 확률이 높아짐
  - 예: `ImportError: No module named 'flask'` 오류 발생 시 에이전트는 `pip install flask`를 실행해야 한다는 것을 파악하고 해당 명령의 승인을 요청
- 루프의 한계:
  - 오류 메시지가 불명확하거나 근본 원인과 다른 지점을 가리키는 경우 수정 방향이 잘못될 수 있음
  - 루프가 무한히 반복되지 않도록 내부적으로 최대 반복 횟수가 설정되어 있으며, 이를 초과하면 에이전트는 작업을 중단하고 사용자에게 도움을 요청
  - 이 시점이 바로 사람의 판단이 개입해야 하는 지점

### 2.5 사용자 승인과 안전장치

- 에이전트 모드의 자율성에는 의도적인 제한이 설계되어 있음
- 이 제한들은 안전한 AI 협업을 위해 반드시 이해해야 하는 요소

**터미널 명령 승인.**

- 모든 터미널 명령은 사용자 승인이 필요
- 승인 과정에서 개발자가 확인해야 할 사항:
  1. 실행하려는 명령이 의도한 작업인지 확인 -- `pip install flask`는 안전하지만, `pip install`에 알 수 없는 패키지가 포함되어 있다면 주의 필요
  2. 명령의 부작용을 고려 -- `python app.py`는 로컬 서버를 기동하므로 기존에 실행 중인 프로세스와 포트 충돌이 발생할 수 있음
  3. 명령이 현재 환경에 적합한지 확인 -- 가상환경(venv)이 활성화되지 않은 상태에서 패키지를 설치하면 시스템 전역에 영향을 미칠 수 있음

**파일 변경 검토.**

- 에이전트가 생성하거나 수정한 파일은 Chat 패널에서 diff로 확인 가능
- 변경 사항을 하나씩 수락(Accept)하거나 거부(Reject) 가능
- 대량의 파일이 생성된 경우 모든 변경을 무조건 수락하는 것은 위험
- 특히 기존 파일을 수정하는 경우에는 원래 코드의 의도가 훼손되지 않았는지 반드시 확인 필요

**실행 취소와 복구.**

- 에이전트가 수행한 작업이 마음에 들지 않는 경우:
  - VS Code의 Undo(Ctrl+Z) 기능으로 파일 편집을 되돌릴 수 있음
  - 터미널에서 설치된 패키지는 `pip uninstall`로 제거 가능
  - Git을 사용하고 있다면 `git stash`나 `git checkout`으로 변경 사항을 되돌릴 수 있음
- 에이전트 모드를 사용하기 전에 반드시 Git 커밋을 수행하여 복구 지점을 만들어 두는 것이 좋은 습관

### 2.6 에이전트 모드와 기존 도구의 위치 관계

- 에이전트 모드의 등장이 기존 개발 도구와 워크플로우를 완전히 대체하는 것은 아님
- 기존 도구 위에 새로운 상호작용 계층이 추가된 것으로 이해해야 함

- 전통적인 소프트웨어 개발 워크플로우: "설계 → 코딩 → 테스트 → 디버깅 → 배포"의 순환
  - 에이전트 모드가 직접 관여하는 영역은 주로 "코딩"과 "테스트"
  - 설계: 여전히 사람의 판단이 필수적인 영역
  - 디버깅: 에이전트의 자기 치유 루프가 일부 담당하지만 복잡한 문제는 사람의 추론이 필요
  - 배포: 현재 에이전트 모드의 범위를 벗어나는 영역

- 이 위치 관계를 정확히 이해하면, 에이전트 모드에 대한 과도한 기대나 불필요한 불안 없이 도구를 합리적으로 활용 가능
- 에이전트 모드는 개발자를 대체하는 것이 아니라, 개발자의 능력을 증폭시키는 도구
  - 숙련된 개발자가 에이전트 모드를 사용하면 생산성이 크게 향상됨
  - 프로그래밍 기초가 없는 사람이 에이전트 모드만으로 복잡한 소프트웨어를 만들기는 여전히 어려움
  - 에이전트가 생성한 코드를 읽고, 평가하고, 수정할 수 있는 역량이 전제되어야 함

- **"바이브 코딩(Vibe Coding)"의 한계**와도 연결됨
  - 1주차에서 다루었듯이, 자연어 명세만으로 코드를 생성하는 패러다임은 프로토타입이나 소규모 도구 수준에서는 효과적
  - 프로덕션 수준의 소프트웨어에서는 사람의 검증과 개입이 반드시 필요
  - 에이전트 모드의 진정한 가치는 "코드를 생성하는 것" 자체가 아니라, "실행 가능한 결과에 빠르게 도달하게 해 주는 것"에 있음

**표 2.3** 개발 워크플로우 단계별 에이전트 모드의 역할

| 개발 단계 | 에이전트 모드의 기여 | 사람의 역할 |
|----------|-------------------|-----------|
| 설계 | 아이디어 탐색, 구조 제안 (Ask 모드) | 최종 아키텍처 결정 |
| 코딩 | 코드 생성, 다중 파일 편집 (Agent 모드) | 코드 리뷰, 품질 확인 |
| 테스트 | 테스트 코드 생성 및 실행 (Agent 모드) | 테스트 적절성 판단 |
| 디버깅 | 자기 치유 루프로 단순 오류 수정 | 복잡한 논리 오류 해결 |
| 배포 | 현재 범위 밖 | 배포 전략, 인프라 관리 |

## 2회차: 실습

### 실습 1: 자연어로 Flask Todo 웹앱 만들기

- 에이전트 모드를 사용하여 자연어 한 문장으로 웹 애플리케이션을 생성하는 과정을 직접 체험
- 목표: 에이전트 모드의 자율 실행 과정을 관찰하고, 사람이 개입해야 하는 지점을 파악하는 것

**사전 준비.**

1. 새 폴더를 하나 만들어 VS Code로 연다. 빈 프로젝트에서 시작하는 것이 에이전트 모드의 동작을 관찰하기에 가장 좋다.
2. 터미널에서 가상환경을 생성하고 활성화한다.

```bash
python3 -m venv venv
source venv/bin/activate    # macOS/Linux
# venv\Scripts\activate     # Windows
```

3. Git 저장소를 초기화하여 복구 지점을 만든다.

```bash
git init
git add -A && git commit -m "빈 프로젝트 시작"
```

**에이전트 모드 실행.**

- 단축키 Ctrl+Shift+I(macOS에서는 Cmd+Shift+I)를 눌러 Copilot Chat 패널을 열고, 모드를 Agent로 전환
- 채팅 입력창에 다음과 같이 입력:

```
Flask로 할 일 목록(Todo) 웹앱을 만들어 줘.
SQLite DB를 사용하고, 할 일 추가/완료/삭제 기능이 있어야 해.
```

**관찰 포인트.** 에이전트가 동작하는 동안 다음 사항을 기록한다.

1. **프로젝트 구조 설계**
   - 어떤 파일과 폴더를 생성하는지 관찰
   - `app.py`, `templates/`, `static/` 등이 Flask의 일반적인 관례를 따르는지 확인
2. **코드 작성 순서**
   - 에이전트가 파일을 어떤 순서로 작성하는지 기록
   - 보통 메인 애플리케이션 파일을 먼저 만들고, 이어서 템플릿과 스타일시트를 생성
3. **터미널 명령 승인**
   - 에이전트가 요청하는 터미널 명령을 기록
   - `pip install flask` 같은 의존성 설치 명령과 `python app.py` 같은 실행 명령이 포함되는지 확인
   - 각 명령의 의미를 읽은 뒤 "Allow" 클릭
4. **오류 처리**
   - 에이전트가 오류를 만났을 때 어떻게 반응하는지 관찰
   - 오류 메시지를 읽고 자동 수정하는 과정이 자기 치유 루프임을 확인

**생성 결과 검토.** 실습이 완료되면 에이전트가 생성한 파일 목록과 코드를 직접 검토한다. 특히 다음 세 가지를 확인한다.

1. 프로젝트 구조가 Flask의 일반적인 관례를 따르는지 살펴봄
   - `templates/` 폴더에 HTML 파일이 있는지, `static/` 폴더에 CSS/JS 파일이 있는지 확인
2. SQLite 데이터베이스 초기화 코드가 적절한 위치에 배치되어 있는지 확인
   - 데이터베이스 파일이 프로젝트 루트에 생성되는지, 테이블 스키마가 올바르게 정의되어 있는지 검토
3. HTML 템플릿에서 CSRF 보호나 입력 검증 같은 보안 관련 처리가 포함되어 있는지 점검
   - 대부분의 경우 에이전트는 기능적으로 동작하는 코드를 생성하지만, 에러 처리나 보안 설정 같은 방어적 프로그래밍(Defensive Programming) 요소는 빠뜨리는 경향이 있음

- 에이전트가 생성한 Flask 앱의 핵심 라우트는 대체로 다음과 같은 구조를 따름:

```python
@app.route("/add", methods=["POST"])
def add_todo():
    content = request.form.get("content")
    db.execute("INSERT INTO todos (content) VALUES (?)", (content,))
    db.commit()
    return redirect(url_for("index"))
```

**검증 체크리스트.** 다음 항목을 하나씩 확인하며, 결과를 기록한다.

- [ ] 앱이 정상 실행되는가? (`http://localhost:5000` 접속 확인)
- [ ] 할 일 추가 기능이 동작하는가?
- [ ] 할 일 완료 표시 기능이 동작하는가?
- [ ] 할 일 삭제 기능이 동작하는가?
- [ ] 빈 문자열을 입력하면 어떻게 되는가? (입력 검증)
- [ ] 서버를 재시작해도 데이터가 유지되는가? (SQLite 영속성)
- [ ] HTML에 XSS 방지 처리가 되어 있는가?

- 이 체크리스트에서 실패하는 항목이 있다면, 그것이 바로 에이전트가 놓친 부분이며 사람이 보완해야 하는 지점
- 이 빈틈을 발견하고 보완하는 과정이 곧 AI 협업 개발에서 사람의 역할

### 실습 2: 생성 코드 분석과 수동 비교

- 에이전트가 생성한 코드를 분석하고, 수동으로 동일한 앱을 만드는 경우와 비교

**파일별 분석.** 에이전트가 생성한 각 파일을 열어 다음을 분석한다.

1. **app.py**
   - Flask 앱의 메인 파일
   - 라우트 정의, 데이터베이스 연결, 에러 핸들링이 포함되어 있는지 확인
   - `if __name__ == "__main__":` 블록 존재 여부와 디버그 모드 설정 방식 점검
2. **데이터베이스 관련 코드**
   - SQLite 연결 방식을 확인
   - `sqlite3`를 직접 사용하는지, SQLAlchemy 같은 ORM을 사용하는지 비교
   - 테이블 생성 코드와 마이그레이션 전략을 분석
3. **HTML 템플릿**
   - Jinja2 문법이 올바른지 확인
   - CSS 스타일이 별도 파일로 분리되어 있는지 점검
   - `form` 태그의 `action`과 `method`가 올바른지 확인

**Ask 모드로 코드 이해하기.**

- 에이전트가 작성한 코드 중 이해가 어려운 부분이 있다면, Copilot Chat을 Ask 모드로 전환하여 질문
- 예시:

```
이 app.py에서 get_db() 함수가 하는 역할을 설명해 줘.
teardown_appcontext는 왜 필요한가?
```

- Agent 모드와 Ask 모드를 전환하며 사용하는 것이 실무에서의 자연스러운 워크플로우

**수동 구현과 비교.**

- 시간이 허용된다면, 에이전트의 도움 없이 동일한 기능의 Todo 앱을 직접 작성해 봄
- 두 버전을 비교하며 다음을 관찰:
  - 에이전트가 생성한 코드에서 불필요한 부분은 무엇인가?
  - 에이전트가 생략한 부분 중 중요한 것은 무엇인가?
  - 코드 스타일과 구조에서 차이가 나는 부분은 무엇인가?
  - 에이전트가 선택한 기술적 결정(예: ORM 사용 여부, 디렉토리 구조)이 합리적인가?

- 이 비교 과정에서 에이전트 코드의 강점과 약점을 구체적으로 파악 가능

**기능 추가 요청.**

- 시간이 남는다면, 에이전트 모드에서 기존 Todo 앱에 기능을 추가해 봄
- 다음 중 하나를 선택하여 시도:

```
[추가 기능 1]
할 일에 우선순위(높음/중간/낮음)를 지정할 수 있게 해 줘.
우선순위별로 정렬하는 기능도 추가해 줘.
```

```
[추가 기능 2]
할 일에 마감일을 설정할 수 있게 해 줘.
마감일이 지난 할 일은 빨간색으로 표시해 줘.
```

- 기존 코드가 있는 프로젝트에 기능을 추가하는 작업에서 에이전트가 어떻게 동작하는지 관찰
- 빈 프로젝트에서 생성할 때와 비교하여 에이전트의 동작이 어떻게 달라지는지, 기존 코드와의 일관성이 유지되는지 살펴봄
- 이 경험은 3주차에서 다룰 멀티파일 프로젝트 전략의 기초가 됨

2. 앱이 실행되는 상태의 브라우저 스크린샷을 포함한다. (할 일이 2개 이상 등록된 상태)
3. `README.md`에 다음 내용을 작성한다:
   - 에이전트 모드가 생성한 파일 목록과 각 파일의 역할
   - 에이전트가 실행한 터미널 명령 목록 (순서대로)
   - 자기 치유 루프가 관찰된 경우, 그 과정을 기술한다 (어떤 오류가 발생했고 에이전트가 어떻게 수정했는지)
   - 검증 체크리스트 결과 (통과/실패 항목 구분)

---

## 핵심 정리

- 에이전트 모드(Agent Mode)는 자율 실행 루프(목표 해석 → 컨텍스트 탐색 → 계획 수립 → 코드 편집 → 터미널 실행 → 오류 감지 → 자동 수정)로 동작하는 Copilot의 자율 모드이다
- VS Code Copilot Chat은 세 가지 모드를 제공한다: Ask(질문, 읽기 전용), Edit(편집, 파일 범위 한정), Agent(자율 실행, 프로젝트 전체 + 터미널)
- 터미널 명령은 사용자 승인이 필요하며, 이것은 의도하지 않은 시스템 변경을 방지하는 안전장치이다
- 자기 치유(Self-Healing) 루프는 에이전트가 오류를 감지하고 자동으로 수정을 시도하는 메커니즘이며, 에이전트 모드의 가장 강력한 특성이다
- 에이전트가 생성한 코드는 기능적으로 동작하더라도 보안, 에러 처리 등 방어적 프로그래밍 요소가 부족할 수 있으므로 반드시 사람이 검증해야 한다
- 세 가지 모드를 상황에 따라 전환하며 사용하는 것이 가장 효과적인 워크플로우이다

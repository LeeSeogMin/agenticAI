# 12주차. Coding Agent와 팀 워크플로우

> **1회차** (강의 90분): Coding Agent 개념, Agent Mode와의 비교, 설정, 좋은 이슈 작성법, 팀 워크플로우 설계, 한계와 주의사항
> **2회차** (실습 90분): Issue에서 PR까지 자동화 워크플로우 구축, MCP 서버 연동(Sentry)

---

## 학습목표

1. GitHub Copilot Coding Agent의 작동 방식과 내부 실행 흐름을 이해한다
2. Agent Mode(로컬 실시간)와 Coding Agent(클라우드 비동기)의 차이를 명확히 구분하고 상황에 맞게 선택한다
3. 코딩 에이전트의 출력 품질을 높이는 이슈 작성법을 익힌다
4. 이슈 할당에서 PR 생성, 리뷰, 머지까지의 팀 워크플로우를 설계하고 구축한다

## 선수지식

- 2주차에서 다룬 에이전트 모드(Agent Mode)의 자율 실행 루프를 이해하고 있어야 함
- GitHub 계정과 저장소 관리 권한이 필요
- GitHub Actions의 기본 개념(워크플로우, 러너, 작업)을 알고 있으면 실습을 따라가기에 수월함
- 11주차에서 실습한 MCP + Skills 통합 경험이 있으면 Sentry MCP 연동 실습에서 설계 패턴을 더 깊이 이해 가능

## 1회차: 강의

### 12.1 Coding Agent란 무엇인가

- 2주차에서 배운 에이전트 모드(Agent Mode)는 개발자가 VS Code 안에서 실시간으로 대화하며 코드를 작성하는 방식
  - 복잡한 설계 탐색이나 즉각적인 피드백이 필요한 작업에 효과적
  - 단, 개발자가 IDE 앞에 앉아 있어야 한다는 제약 존재
- **코딩 에이전트(Coding Agent)**는 이 제약을 제거함
  - 클라우드 기반 자율 에이전트
  - 개발자가 GitHub 이슈에 `@copilot`을 담당자(assignee)로 할당하면, 에이전트가 클라우드 환경에서 독립적으로 작업을 시작
  - 개발자는 이슈를 할당한 뒤 다른 업무를 수행하면 되고, 에이전트가 작업을 완료하면 드래프트 풀 리퀘스트(Draft Pull Request)가 생성되어 알림을 받음

#### 코딩 에이전트의 5단계 실행 흐름

- 코딩 에이전트의 작업 흐름은 다섯 단계로 구성됨

1. **이슈 할당**: 개발자가 GitHub 이슈를 작성하고 `@copilot`을 담당자로 할당 (Copilot Chat에서 이슈를 위임하는 것도 가능)
2. **환경 구동**: 에이전트가 GitHub Actions 기반의 안전하고 격리된 개발 환경(secure, isolated development environment)을 구동
3. **분석 및 구현**: 저장소의 코드를 분석하고 새 브랜치를 생성한 뒤 변경 사항을 구현
4. **커밋 및 PR 생성**: 커밋을 푸시하고 드래프트 PR을 열기
5. **리뷰 대기**: 이슈 할당자가 공동 작성자(co-author)로 기재되지만, 해당 PR을 직접 승인할 수는 없음 → 독립적인 리뷰어가 반드시 필요

- 마지막 규칙은 코딩 에이전트 설계에서 가장 중요한 안전장치
  - 이슈를 만든 사람이 곧바로 PR을 승인할 수 있다면, 에이전트의 결과물이 충분한 검증 없이 머지될 위험이 있음
  - 할당자와 승인자를 분리함으로써 최소 한 명의 독립적인 사람이 코드를 검토하도록 보장

### 12.2 Agent Mode와 Coding Agent 비교

- 에이전트 모드와 코딩 에이전트는 배타적 관계가 아니라 **상호 보완적**

**표 12.1** Agent Mode와 Coding Agent 비교

| 구분 | Agent Mode (IDE) | Coding Agent (Cloud) |
|------|-----------------|---------------------|
| 실행 환경 | 로컬 (VS Code) | 클라우드 (GitHub Actions) |
| 작업 방식 | 실시간 대화형 | 비동기 (이슈 할당 후 자동 처리) |
| 승인 방식 | 터미널 명령마다 사용자 승인 | MCP 도구 자율 사용 |
| 산출물 | 로컬 파일 수정 | 브랜치 + 커밋 + Draft PR |
| 적합한 작업 | 복잡한 설계, 탐색적 개발 | 명확한 구현 작업, 반복적 수정 |
| 피드백 루프 | 즉시 (실시간 대화) | 지연 (PR 코멘트 → 재수정) |
| 컨텍스트 범위 | IDE에 열린 파일 + 워크스페이스 | 저장소 전체 |
| 비용 모델 | Copilot 구독 | Copilot 구독 + GitHub Actions 분 |

- **에이전트 모드가 적합한 작업**: 복잡한 아키텍처 탐색, 여러 선택지를 비교해야 하는 작업
- **코딩 에이전트가 적합한 작업**: "이 함수를 추가한다", "이 버그를 수정한다"처럼 완료 조건이 명확한 작업
- 숙련된 개발자는 하루 동안 에이전트 모드로 설계를 진행하면서, 구현 세부 사항은 이슈로 작성하여 코딩 에이전트에 위임하는 식으로 두 방식을 병행
  - 설계에 집중하면서도 구현 속도를 높일 수 있어 매우 효과적

#### 작업 유형별 선택 가이드

- **에이전트 모드를 선택하는 경우**: 요구사항이 불명확하여 대화를 통해 명확히 해야 할 때, 여러 구현 방안을 비교 탐색할 때, 실시간 피드백이 필요한 프로토타이핑 단계일 때
- **코딩 에이전트를 선택하는 경우**: 수락 기준이 명확한 구현 작업일 때, 개발자가 다른 작업에 집중하고 싶을 때, 반복적이고 정형화된 코드 변경(테스트 추가, 린트 수정 등)일 때

### 12.3 Coding Agent 설정

- 코딩 에이전트를 사용하려면 저장소 수준에서 네 가지 설정을 완료해야 함

#### 설정 1: 코딩 에이전트 활성화

- GitHub 웹에서 저장소의 Settings 탭으로 이동
- 좌측 메뉴에서 Copilot > Coding agent를 선택하여 Enable로 전환
- 저장소 관리자(admin) 권한이 필요

#### 설정 2: 허용 MCP 서버 지정

- 코딩 에이전트가 외부 도구에 접근해야 하는 경우, 저장소 설정에서 허용된 MCP 서버 목록을 지정 가능
- 선택 사항이지만, 실습 2에서 다룰 Sentry 연동 같은 고급 시나리오에서 필요

#### 설정 3: 환경 설정 파일 작성

- 코딩 에이전트가 클라우드에서 코드를 실행하려면 의존성 설치나 빌드 도구 설정이 필요할 수 있음
- 저장소 루트에 `copilot-setup-steps.yml` 파일을 추가

```yaml
steps:
  - uses: actions/setup-python@v5
    with:
      python-version: '3.12'
  - run: pip install -r requirements.txt
```

- 이 파일은 GitHub Actions의 워크플로우 스텝 형식을 따름
- 에이전트가 작업을 시작하기 전에 이 스텝들이 먼저 실행되어 개발 환경이 준비됨
- 프로젝트별 설정 예시
  - Node.js: `actions/setup-node` + `npm install`
  - Java: `actions/setup-java` + `mvn install`

#### 설정 4: 브랜치 보호 규칙

- 코딩 에이전트가 생성하는 브랜치는 `copilot/fix-*`와 같은 패턴을 따름
- 메인 브랜치에 직접 푸시되는 것을 방지하기 위해, PR 리뷰 필수와 상태 검사 통과를 요구하는 브랜치 보호 규칙을 반드시 설정

**표 12.2** 필수 브랜치 보호 규칙

| 규칙 | 설정값 | 이유 |
|------|--------|------|
| Require pull request reviews | 최소 1명 | 에이전트 결과물의 독립적 검토 보장 |
| Require status checks to pass | 활성화 | CI 테스트 통과 없이 머지 방지 |
| Require branches to be up to date | 활성화 | 최신 메인 브랜치와 충돌 방지 |
| Do not allow bypassing | 활성화 | 관리자도 규칙을 우회할 수 없도록 설정 |

- 이 규칙이 없으면 에이전트의 결과물이 검증 없이 머지될 수 있으므로, 실무 환경에서는 반드시 적용

### 12.4 좋은 이슈 작성법

- 코딩 에이전트의 출력 품질은 이슈의 품질에 직접적으로 비례함
- 모호한 이슈는 모호한 코드를, 명확한 이슈는 명확한 코드를 만듦

#### 좋은 이슈의 5가지 요소

1. **제목은 기대하는 결과를 한 문장으로 표현**: "비밀번호 관련 작업"이 아니라 "로그인 API에 비밀번호 강도 검증 함수 추가"처럼 구체적으로 작성
2. **본문에 수락 기준(Acceptance Criteria) 명시**: 에이전트가 "완료"를 판단할 수 있는 객관적인 조건이 필요
3. **수정할 파일이나 디렉터리를 힌트로 제공**: 에이전트가 저장소 전체를 탐색하는 시간을 줄이고, 잘못된 위치에 코드를 생성하는 것을 방지
4. **테스트 기준 기술**: 어떤 입력에 대해 어떤 출력이 나와야 하는지, 어떤 경계 조건을 확인해야 하는지 명시 → 에이전트가 자체적으로 검증까지 수행
5. **제약 조건 명시**: 사용할 라이브러리, 따라야 할 코딩 패턴, 성능 요구사항 등을 명시하여 에이전트의 선택 범위를 한정

#### 이슈 품질 비교

**표 12.3** 나쁜 이슈와 좋은 이슈의 비교

| 구분 | 나쁜 이슈 | 좋은 이슈 |
|------|----------|----------|
| 제목 | "비밀번호 기능 추가" | "로그인 API에 비밀번호 강도 검증 함수 추가" |
| 본문 | "비밀번호 체크하는 거 만들어 주세요" | 검증 조건, 반환값 형식, 대상 파일 경로를 상세 기술 |
| 테스트 | 언급 없음 | "pytest로 조건별 통과/실패 테스트 작성" |
| 제약 | 언급 없음 | "정규식 사용, 외부 라이브러리 불필요" |
| 결과 예측 | 에이전트가 암호화/검증/재설정 중 추측 | 첫 시도에서 원하는 결과에 근접 |

- 나쁜 이슈를 받은 에이전트는 "비밀번호 기능"이 무엇인지부터 추측해야 함
  - 암호화인지, 검증인지, 재설정인지 알 수 없으므로 예상과 다른 코드를 생성할 확률이 높음

#### 역질문 불가 제약

- 현재 코딩 에이전트는 이슈 작성자에게 역으로 질문하는 기능(interactive clarification)을 지원하지 않음
  - 에이전트 모드에서는 VS Code 안에서 실시간으로 "이 부분은 어떻게 할까요?"라고 물어볼 수 있지만, 코딩 에이전트는 주어진 이슈만으로 작업을 완료해야 함
  - 따라서 이슈의 완결성이 결과 품질에 결정적인 영향을 미침
- 이 제약은 역설적으로 이슈 작성 능력을 크게 향상시키는 훈련이 됨
  - 에이전트가 이해할 수 있을 만큼 명확한 이슈는 인간 동료도 명확하게 이해 가능

#### 이슈 템플릿 활용

- 팀에서 코딩 에이전트를 체계적으로 사용하려면 이슈 템플릿을 만들어 두는 것이 효과적
- `.github/ISSUE_TEMPLATE/copilot-task.md` 파일을 다음과 같이 작성

```markdown
---
name: Copilot 작업 요청
about: Coding Agent에 위임할 작업
labels: copilot
assignees: copilot
---

## 요구사항
<!-- 구현할 내용을 구체적으로 기술한다 -->

## 대상 파일
<!-- 수정 또는 생성할 파일 경로 -->

## 수락 기준
<!-- 완료로 판단할 수 있는 객관적 조건 -->
- [ ] 조건 1
- [ ] 조건 2

## 테스트
<!-- 테스트 방법과 기대 결과 -->

## 제약 조건
<!-- 사용할 라이브러리, 코딩 패턴, 성능 요구 등 -->
```

- 이 템플릿을 사용하면 팀원 누구나 일관된 품질의 이슈를 작성할 수 있으며, 필수 항목 누락을 방지 가능

### 12.5 팀 워크플로우 설계

- 코딩 에이전트를 개인이 사용하는 것과 팀 전체의 워크플로우에 통합하는 것은 다른 차원의 문제
- 이 절에서는 코딩 에이전트를 팀 개발 프로세스에 안전하게 통합하는 방법을 설계함

#### 전체 워크플로우

```
[개발자 A: 이슈 작성 + @copilot 할당]
         |
         v
[Coding Agent: 저장소 분석]
         |
         v
[Coding Agent: 구현 + 테스트]
         |
         v
[Coding Agent: Draft PR 생성]
         |
         v
[개발자 B: 코드 리뷰] ←──── 수정 요청 ────→ [Coding Agent: 추가 수정]
         |
         v (승인)
[CI/CD: 자동 테스트 + 빌드]
         |
         v
[머지 + 배포]
```

**그림 12.1** 코딩 에이전트를 포함한 팀 개발 워크플로우

#### 4가지 핵심 규칙

1. **이슈 할당자는 공동 작성자(co-author)로 등록되므로 해당 PR을 승인할 수 없음** → 독립적인 코드 리뷰를 보장하는 가장 근본적인 안전장치
2. **브랜치 보호 규칙이 반드시 설정되어 있어야 함** → 에이전트가 메인 브랜치에 직접 커밋하는 것을 원천 차단
3. **CI 파이프라인의 모든 테스트가 통과해야 머지 가능** → 에이전트가 작성한 테스트뿐 아니라 기존 테스트 스위트도 모두 통과해야 함
4. **에이전트는 스스로 머지하지 않음** → 최종 머지 결정은 반드시 사람이 내림

#### 주니어 개발자 비유

- 이러한 규칙들이 적용된 팀 워크플로우에서 코딩 에이전트는 "주니어 개발자"와 유사한 역할을 수행
  - 명확한 작업 지시(이슈)를 받아 구현하고, 시니어 개발자(리뷰어)의 검토와 승인을 거쳐 코드가 반영됨
  - 차이점: 에이전트가 비동기적으로 동작하므로 리뷰어의 업무 시간과 독립적으로 작업 진행 가능

#### 단계적 도입 전략

- 팀 규모와 프로젝트 성격에 따라 코딩 에이전트의 활용 범위를 단계적으로 확대하는 것이 현실적

**표 12.4** 코딩 에이전트 단계적 도입 가이드

| 단계 | 대상 작업 | 위험도 | 리뷰 강도 |
|------|----------|--------|----------|
| 1단계 (도입기) | 문서 수정, 오타 교정, 주석 보강 | 매우 낮음 | 간단 확인 |
| 2단계 (적응기) | 테스트 추가, 린트 오류 수정, 리팩토링 | 낮음 | 코드 리뷰 |
| 3단계 (활용기) | 새 함수/엔드포인트 구현 | 중간 | 상세 리뷰 |
| 4단계 (성숙기) | 기능 구현, 버그 수정 | 중간~높음 | 전문 리뷰 |

- 초기에는 저위험 작업부터 시작하여, 팀이 리뷰 프로세스에 익숙해지면 기능 구현 수준까지 확대하는 접근이 안전

---

### 12.6 한계와 주의사항

- 코딩 에이전트는 강력한 도구이지만, 현재 시점에서 분명한 한계가 있음
- 이 한계를 정직하게 인식하는 것이 도구를 현명하게 사용하기 위한 전제 조건

#### 복잡도 한계

- 낮은 복잡도에서 중간 복잡도의 작업에 가장 적합
  - 함수 하나를 추가하거나, 기존 API의 엔드포인트를 수정하거나, 단위 테스트를 보강하는 등의 작업에서 높은 성공률
- 여러 서비스에 걸친 아키텍처 변경이나 복잡한 비즈니스 로직의 재설계는 현재 능력 범위를 벗어남

#### 도메인 지식 한계

- 깊은 도메인 지식이 필요한 작업에서 어려움을 겪음
  - 금융 규정에 따른 계산 로직, 의료 데이터 처리 규칙, 특정 산업의 프로토콜 구현 등은 일반적인 학습 데이터에 충분히 포함되어 있지 않음
  - 에이전트가 올바른 구현을 보장하기 어려움
- 11주차에서 학습한 Skills를 활용하면 이 한계를 부분적으로 완화 가능
  - 도메인 규칙을 SKILL.md에 명시하면 에이전트가 해당 규칙을 참고하여 더 정확한 코드를 생성 가능

#### 보안 주의

- 보안에 민감한 변경이나 프로덕션 핵심 로직의 수정에는 각별한 주의가 필요
- 에이전트가 생성한 코드가 기능적으로 동작하더라도, 보안 취약점이나 성능 저하를 유발할 수 있음
- 이러한 영역에서는 에이전트의 결과물을 제안(suggestion)으로만 취급하고, 반드시 전문 리뷰어의 면밀한 검토를 거쳐야 함

#### 비용 고려

- 코딩 에이전트는 GitHub Actions 분(minutes)을 소비함
- 무료 플랜에서는 월간 Actions 사용량이 제한되어 있으므로, 빈번한 사용 시 비용이 발생할 수 있음
- 이슈 하나에 대해 에이전트가 구동되는 시간은 작업의 복잡도에 따라 수 분에서 수십 분까지 다양함

**표 12.5** 코딩 에이전트 한계 요약

| 한계 | 설명 | 대응 방안 |
|------|------|----------|
| 복잡도 제한 | 단일 기능 수준에 적합하다 | 복잡한 작업은 여러 이슈로 분할한다 |
| 도메인 지식 부족 | 전문 영역의 규칙을 모른다 | Skills/Custom Instructions로 지식을 주입한다 |
| 역질문 불가 | 모호한 이슈를 해석할 수 없다 | 이슈 템플릿으로 완결성을 보장한다 |
| 보안 위험 | 취약한 코드를 생성할 수 있다 | 전문 리뷰어의 보안 검토를 필수화한다 |
| 비용 발생 | Actions 분을 소비한다 | 작업 복잡도와 빈도를 관리한다 |

- 이러한 한계는 코딩 에이전트 자체의 문제라기보다, 현재 LLM 기반 코드 생성의 일반적인 한계에 해당
- 기술이 발전함에 따라 처리 가능한 복잡도의 범위는 점차 넓어질 것이지만, "사람의 검토가 필요하다"는 근본적인 원칙은 상당 기간 유효할 것

## 2회차: 실습

### 실습 1: 이슈에서 PR까지 자동화 워크플로우

- 코딩 에이전트의 전체 워크플로우를 처음부터 끝까지 체험함
- 비밀번호 강도 검증 함수를 추가하는 작업을 이슈로 작성하고, 코딩 에이전트가 구현부터 테스트까지 자율적으로 수행하는 과정을 관찰함

#### Step 1: 저장소 준비

- 실습용 저장소를 생성하거나 기존 저장소를 사용
- 12.3절에서 다룬 네 가지 설정(에이전트 활성화, MCP 서버 허용, 환경 설정 파일, 브랜치 보호 규칙)을 모두 완료
- `copilot-setup-steps.yml` 파일을 저장소 루트에 작성

```yaml
steps:
  - uses: actions/setup-python@v5
    with:
      python-version: '3.12'
  - run: pip install -r requirements.txt
```

#### Step 2: 이슈 작성

- GitHub 웹에서 이슈를 생성
- 12.4절에서 학습한 좋은 이슈 작성법을 적용

```
제목: 로그인 API에 비밀번호 강도 검증 함수 추가

## 요구사항
`auth/validators.py`에 `validate_password_strength(password: str) -> dict` 함수를 추가한다.

## 검증 조건
- 최소 8자 이상
- 대문자 1개 이상 포함
- 소문자 1개 이상 포함
- 숫자 1개 이상 포함
- 특수문자(!@#$%^&*) 1개 이상 포함

## 반환값
각 조건의 충족 여부를 불리언으로, 전체 통과 여부를 `is_valid` 키로 포함하는 딕셔너리를 반환한다.

## 테스트
pytest로 각 조건별 통과/실패 케이스를 검증하는 테스트를 작성한다.
파일 위치: tests/test_validators.py

## 제약 조건
- 정규식(re 모듈) 사용
- 외부 라이브러리 불필요
- PEP 8 준수
```

#### Step 3: 에이전트 할당 및 관찰

- 이슈를 생성한 뒤, Assignees 항목에서 `copilot`을 검색하여 담당자로 할당
- 할당이 완료되면 코딩 에이전트가 자동으로 작업을 시작
- 에이전트의 작업 과정을 관찰
  - Actions 탭에서 에이전트의 워크플로우 실행 상태를 확인 가능
- 에이전트가 자율적으로 수행하는 과정
  1. 저장소 구조 분석 (기존 코드 스타일, 테스트 프레임워크, 디렉터리 구조 파악)
  2. 새 브랜치 생성 (예: `copilot/fix-password-validation`)
  3. `auth/validators.py`에 검증 함수 구현
  4. `tests/test_validators.py`에 테스트 작성
  5. 테스트를 실행하여 모든 테스트가 통과하는지 확인
  6. 커밋을 푸시하고 드래프트 PR을 열기
- 에이전트가 생성하는 검증 함수의 핵심 로직 (대체로 다음과 같은 구조를 따름)

```python
import re

def validate_password_strength(password: str) -> dict:
    results = {
        "min_length": len(password) >= 8,
        "has_upper": bool(re.search(r"[A-Z]", password)),
        "has_lower": bool(re.search(r"[a-z]", password)),
        "has_digit": bool(re.search(r"[0-9]", password)),
        "has_special": bool(re.search(r"[!@#$%^&*]", password)),
    }
    results["is_valid"] = all(results.values())
    return results
```

#### Step 4: PR 리뷰

- PR이 생성되면 세 가지 관점에서 리뷰를 수행
  1. **코드 품질**: 함수 구조, 변수명, 주석의 적절성 평가
  2. **테스트 커버리지**: 모든 검증 조건에 대해 통과/실패 케이스가 포함되어 있는지 확인, 엣지 케이스(빈 문자열, 매우 긴 문자열, 유니코드 문자 등)도 테스트되었는지 점검
  3. **이슈와의 일치성**: 이슈에 명시된 수락 기준이 모두 충족되었는지 확인
- 부족한 부분이 있으면 PR 코멘트로 수정을 요청
  - 코딩 에이전트는 리뷰 코멘트를 읽고 추가 수정을 시도하므로, 구체적이고 명확한 피드백을 작성하는 것이 중요
  - 예: "테스트가 부족합니다"보다는 "빈 문자열 입력 시 모든 조건이 False를 반환하는지 확인하는 테스트를 추가해 주세요"라고 작성
- 모든 검토가 완료되면 독립적인 리뷰어(이슈 할당자가 아닌 다른 사람)가 승인하고 머지

### 실습 2: MCP 서버를 활용하는 Coding Agent

- 코딩 에이전트는 기본적으로 저장소 내부의 코드만 분석하고 수정 가능
- MCP 서버를 연결하면 외부 시스템의 데이터를 에이전트에게 제공 가능
- 이 실습에서는 Sentry MCP 서버를 연결하여, 에이전트가 실제 에러 데이터를 기반으로 수정 코드를 작성하는 시나리오를 체험

#### Step 1: MCP 서버 설정

- 코딩 에이전트용 MCP 서버 설정은 로컬(`.vscode/mcp.json`)이 아닌 저장소의 **`.github/copilot/mcp.json`** 파일에 정의
  - 이 위치의 차이가 중요
  - 로컬 설정은 개발자의 IDE에서만 유효
  - `.github/copilot/` 경로의 설정은 클라우드에서 동작하는 코딩 에이전트에도 적용됨

```json
{
  "mcpServers": {
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/sse",
      "headers": {
        "Authorization": "Bearer $COPILOT_MCP_SENTRY_TOKEN"
      },
      "tools": ["search_errors", "get_error_details"]
    }
  }
}
```

- `$COPILOT_MCP_SENTRY_TOKEN`은 저장소의 Secrets에 등록한 환경 변수
  - GitHub 웹에서 Settings > Secrets and variables > Actions에서 등록
- `tools` 배열은 에이전트가 사용할 수 있는 도구를 제한하는 역할을 함
  - 이 필드를 생략하면 서버가 제공하는 모든 도구가 활성화됨
  - 최소 권한 원칙(Principle of Least Privilege)에 따라 필요한 도구만 명시적으로 허용하는 것이 바람직

#### Step 2: 에러 수정 이슈 작성

- 설정이 완료되면 다음과 같은 이슈를 작성

```
제목: Sentry 상위 에러 3건 수정

## 요구사항
Sentry에서 가장 빈번하게 발생하는 에러 3개를 조사하고,
각각에 대한 수정 코드를 작성한다.

## 절차
1. Sentry MCP의 search_errors 도구로 빈도순 에러 목록을 조회한다
2. 상위 3개 에러의 상세 정보를 get_error_details로 확인한다
3. 각 에러의 원인을 분석하고 수정 코드를 구현한다
4. 수정한 코드에 대한 테스트를 작성한다

## 수락 기준
- [ ] 상위 3개 에러 각각에 대한 수정 코드가 포함된다
- [ ] 각 수정에 대한 단위 테스트가 작성된다
- [ ] PR 본문에 각 에러의 원인 분석이 기술된다
```

#### Step 3: 워크플로우 관찰

- 이 이슈를 `@copilot`에 할당하면, 에이전트는 Sentry MCP 서버를 통해 실제 프로덕션 에러 데이터를 조회
- 가상의 에러가 아닌 실제로 발생하고 있는 문제를 기반으로 수정 코드를 작성하므로, 결과물의 실용성이 높아짐
- 이 워크플로우는 코딩 에이전트의 가장 강력한 활용 패턴 중 하나
  - 개발자가 매일 아침 Sentry 대시보드를 확인하는 대신, 에러 수정 이슈를 코딩 에이전트에 위임하고 출근 후 PR을 리뷰하는 것만으로 에러 대응 속도를 크게 높일 수 있음
- 11주차 설계 패턴 관점에서 보면, 이 워크플로우는 **파이프라인 패턴**에 해당
  - Sentry에서 에러 조회 → 에러 분석 → 수정 코드 작성 → 테스트 작성이 순차적으로 연결됨
  - 여기에 감독자 패턴을 추가하면(예: 수정 후 테스트 실행 결과에 따라 재수정), 더 안정적인 에이전트 구축 가능

#### Sentry 계정이 없는 경우의 대안

- GitHub MCP 서버를 활용하여 저장소의 open 이슈 중 `bug` 라벨이 붙은 이슈를 자동으로 조회하고, 각 이슈의 내용을 분석하여 수정 코드와 테스트를 생성하는 워크플로우를 구축
- MCP 설정 파일의 위치(`.github/copilot/mcp.json`)와 이슈 작성 방법은 동일하게 적용

## 과제

다음 산출물을 GitHub 저장소에 커밋하여 제출한다.

1. **저장소 설정**: `copilot-setup-steps.yml` + 브랜치 보호 규칙 스크린샷
2. **이슈**: 실습 1에서 작성한 이슈 URL (좋은 이슈 작성법 5가지 요소 포함)
3. **PR**: 코딩 에이전트가 생성한 Draft PR URL
4. **리뷰 기록**: PR에 작성한 리뷰 코멘트 (최소 2개)
5. **MCP 설정**: `.github/copilot/mcp.json` (실습 2에서 작성한 것)
6. **회고 보고서**: 코딩 에이전트의 결과물 평가, 이슈 작성 시 개선할 점, 한계 경험 (500자 이상)

평가 기준:
- 이슈 작성의 완결성과 명확성 (25%)
- PR 결과물의 코드 품질과 테스트 포함 여부 (25%)
- 리뷰 코멘트의 구체성과 건설성 (20%)
- MCP 서버 연동 설정의 정확성 (15%)
- 회고 보고서의 분석 깊이 (15%)

## 핵심 정리

- Coding Agent는 GitHub 이슈에서 Draft PR까지를 비동기적으로 자동화하는 클라우드 기반 에이전트이다.
- Agent Mode(실시간 대화형)와 Coding Agent(비동기 자동화)는 상호 보완적이며, 작업 성격에 맞게 병행하여 사용한다.
- 좋은 이슈가 좋은 PR을 만든다. 이슈에는 수락 기준, 대상 파일, 테스트 기준, 제약 조건을 명시한다.
- 이슈 할당자는 공동 작성자로 등록되어 해당 PR을 직접 승인할 수 없다. 독립적 리뷰가 필수이다.
- 팀 도입은 저위험 작업부터 시작하여 단계적으로 확대하는 것이 안전하다.
- 코딩 에이전트의 한계(복잡도, 도메인 지식, 보안)를 인식하고, 사람의 검토를 생략하지 않는다.

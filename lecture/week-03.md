# 3주차. Copilot Agent Mode 심화

> **1회차** (강의 90분): Agent Mode가 효과적인 상황과 한계, 복합 프로젝트 전략, 컨텍스트 관리, 모드 조합 워크플로우
> **2회차** (실습 90분): 기존 코드 리팩토링, 멀티파일 프로젝트 생성, Agent Mode 한계 체험과 우회 전략

---

## 학습목표

- 에이전트 모드가 효과적인 작업과 한계가 있는 작업을 판별하는 기준을 수립할 수 있다
- 멀티파일 프로젝트에서 에이전트 모드를 전략적으로 활용하는 방법을 적용할 수 있다
- 컨텍스트 관리와 단계적 지시를 통해 에이전트의 성공률을 높이는 전략을 실행할 수 있다
- Ask, Edit, Agent 세 가지 모드를 적절히 조합하여 효율적인 개발 워크플로우를 설계할 수 있다

## 선수지식

- 2주차에서 학습한 에이전트 모드의 기본 개념과 자율 실행 루프의 동작 원리를 이해하고 있어야 함
- Flask Todo 앱 생성 실습을 완료한 상태여야 함
- 세 가지 모드(Ask/Edit/Agent)의 차이를 설명할 수 있어야 함

## 1회차: 강의

### 3.1 에이전트 모드가 잘 되는 경우 vs 한계

- 에이전트 모드는 만능이 아님
- 어떤 작업에서 효과적이고 어떤 작업에서 한계를 보이는지 파악하는 것은 도구를 전략적으로 활용하기 위한 필수 판단

- 에이전트 모드가 높은 성공률을 보이는 작업에는 몇 가지 공통된 특성이 있음:

**첫째, 완료 조건이 명확하게 정의된 작업.**

- "로그인 폼을 만들어 줘"나 "CSV 파일을 JSON으로 변환하는 스크립트를 작성해 줘"처럼 결과물의 형태가 분명한 경우, 에이전트는 목표를 정확히 해석하고 자율적으로 완수 가능
- 완료 조건이 명확하다는 것은 에이전트가 "작업이 끝났다"는 판단을 스스로 내릴 수 있다는 의미
- 반대로 "이 코드를 더 좋게 만들어 줘"처럼 "좋다"의 기준이 모호한 경우, 에이전트는 무엇을 개선해야 하는지 판단하기 어려움

**둘째, 표준 프레임워크를 사용하는 작업.**

- Flask, React, Express, Django, Spring Boot 같은 표준 프레임워크를 사용하는 작업은 성공률이 높음
- 이들 프레임워크는 학습 데이터에 풍부하게 포함되어 있어 관용적인 패턴(Idiomatic Pattern)을 정확히 생성
- 프레임워크의 공식 문서에 나와 있는 전형적인 구조를 따르므로, 결과물의 품질이 일정 수준 이상 보장됨
- 반면, 사내에서만 사용하는 커스텀 프레임워크나 비공개 라이브러리를 사용해야 하는 경우에는 에이전트가 해당 코드 패턴을 알 수 없으므로 성공률이 급격히 떨어짐

**셋째, 빈 프로젝트에서 시작하는 그린필드(Greenfield) 개발.**

- 기존 코드와의 충돌 없이 일관된 구조를 만들 수 있기 때문
- 에이전트가 모든 파일을 처음부터 생성하므로 파일 간 일관성이 자연스럽게 확보됨
- 2주차 실습에서 빈 프로젝트에서 Flask Todo 앱을 생성한 것이 바로 이 유형에 해당

**넷째, 테스트 스위트가 갖춰진 코드베이스의 리팩토링.**

- 테스트 결과가 피드백 루프의 명확한 성공/실패 기준을 제공하기 때문
- 에이전트는 리팩토링 후 테스트를 실행하고, 실패한 테스트를 기반으로 수정을 반복 가능
- 테스트가 자기 치유 루프의 "나침반" 역할을 하는 셈

- 반면, 에이전트 모드가 어려움을 겪는 상황도 명확하게 존재함:

**첫째, 도메인 특화 비즈니스 로직이 필요한 작업.**

- 특정 산업의 규정(금융 규제, 의료 프로토콜)이나 회사 고유의 업무 규칙은 학습 데이터에 포함되어 있지 않음
- 에이전트가 "고객 등급에 따라 할인율을 차등 적용해 줘"라는 요청을 받으면, 기술적으로 할인 로직을 구현할 수는 있지만, 실제 회사의 할인 정책과 일치하는지는 보장 불가

**둘째, 프로파일링이 필요한 성능 최적화 작업.**

- 병목 지점을 찾고 최적화 전략을 수립하는 과정은 실측 데이터에 기반한 판단이 필요
- 에이전트는 "이 코드를 최적화해 줘"라는 요청에 일반적인 최적화 패턴(캐싱, 인덱싱, 알고리즘 개선)을 적용할 수 있지만, 실제 프로파일링 데이터 없이는 진짜 병목을 정확히 짚어내기 어려움

**셋째, 테스트가 없는 대규모 레거시 코드베이스에서의 작업.**

- 피드백 루프를 작동시킬 검증 수단이 없기 때문에 자기 치유 메커니즘이 제대로 동작하지 않음
- 에이전트가 코드를 수정해도 그 수정이 올바른지 확인할 방법이 없음
- 수천 줄에 달하는 레거시 코드의 맥락을 에이전트가 완전히 파악하기 어려움

**넷째, 외부 서비스 계정이나 인증 정보가 필요한 작업.**

- 에이전트는 API 키를 직접 생성하거나, 외부 서비스에 로그인하거나, OAuth 인증 흐름을 자율적으로 수행할 수 없음
- 이러한 작업은 사람이 사전에 준비해야 하는 영역

**표 3.1** 에이전트 모드의 효과성 판단 기준

| 구분 | 효과적인 작업 | 한계가 있는 작업 |
|------|-------------|----------------|
| 완료 기준 | 명확한 입출력 명세 | 모호한 품질 기준 |
| 도메인 | 표준 프레임워크, 공개 API | 사내 규정, 특수 프로토콜 |
| 코드베이스 | 그린필드 또는 테스트 완비 | 테스트 없는 레거시 |
| 외부 의존성 | 로컬 완결적 작업 | 인증/계정 필요 서비스 |
| 규모 | 파일 10개 이하 소규모 작업 | 수백 파일 대규모 프로젝트 |

- 이 판단 기준을 사전에 적용하면, 에이전트 모드에 맡길 작업과 사람이 직접 수행할 작업을 효율적으로 분배 가능
- 에이전트의 한계를 인식하는 것은 에이전트를 포기하는 것이 아니라, 에이전트를 더 전략적으로 활용하기 위한 전제 조건

### 3.2 멀티파일 프로젝트 전략

- 2주차 실습에서 에이전트 모드로 Flask 앱을 생성할 때, 에이전트가 여러 파일을 자율적으로 만드는 것을 관찰함
- 이 절에서는 멀티파일 프로젝트에서 에이전트 모드를 효과적으로 활용하는 전략을 다룸

**문제: 한 번에 너무 많은 것을 요청하면 실패함.**

- 에이전트 모드에 "풀스택 쇼핑몰을 만들어 줘"처럼 범위가 매우 넓은 요청을 하면:
  - 에이전트는 작업을 시작하지만 중간에 컨텍스트를 잃거나,
  - 앞서 작성한 코드와 일관성을 유지하지 못하거나,
  - 자기 치유 루프가 잘못된 방향으로 수렴하는 경우가 발생
- 이는 현재 LLM의 컨텍스트 윈도우 한계와 직접 관련이 있음

**전략 1: 단계적 지시(Incremental Instruction).**

- 큰 프로젝트를 작은 단위로 나누어 순차적으로 지시하는 것이 가장 효과적인 전략
- 예: 쇼핑몰 프로젝트를 구현한다면 다음과 같은 순서로 에이전트에게 요청

```
1단계: "Flask로 기본 프로젝트 구조를 만들어 줘. 홈페이지와 상품 목록 페이지가 필요해."
2단계: "상품 데이터를 SQLite에 저장하고 CRUD 기능을 추가해 줘."
3단계: "사용자 인증 기능을 추가해 줘. Flask-Login을 사용해."
4단계: "장바구니 기능을 추가해 줘."
```

- 각 단계가 완료되면 결과를 검증하고, 필요하면 Edit 모드로 세부 사항을 조정한 뒤 다음 단계로 진행
- 이 방식은 에이전트가 관리해야 할 컨텍스트의 범위를 각 단계로 한정하므로, 일관성을 유지하기 훨씬 수월

**전략 2: 프로젝트 구조 선행 정의.**

- 에이전트에게 코드를 작성하기 전에 프로젝트 구조를 먼저 정의하도록 요청하는 것도 효과적

```
다음 구조를 가진 Flask 프로젝트를 만들어 줘.
- app/
  - __init__.py (앱 팩토리)
  - models.py (SQLAlchemy 모델)
  - routes/ (블루프린트별 라우트)
  - templates/
  - static/
- tests/
- requirements.txt
- config.py

먼저 빈 파일과 폴더만 생성하고, 각 파일의 역할을 주석으로 작성해 줘.
```

- 구조가 확정된 후에 각 파일의 구현을 순차적으로 요청하면, 에이전트가 전체 프로젝트의 설계 의도를 파악한 상태에서 일관된 코드를 작성 가능

**전략 3: 기존 파일 참조 지시.**

- 에이전트가 새로운 파일을 작성할 때, 기존 파일의 패턴을 참조하도록 명시적으로 지시하면 일관성이 향상됨

```
models.py의 User 모델과 같은 패턴으로 Product 모델을 추가해 줘.
routes/auth.py와 같은 구조로 routes/products.py를 만들어 줘.
```

- 에이전트가 기존 코드를 탐색하여 그 패턴을 따르므로, 프로젝트 전체의 코드 스타일이 통일됨

**전략 4: Git 커밋으로 체크포인트 관리.**

- 각 단계가 완료될 때마다 Git 커밋을 수행하여 복구 지점을 만듦
- 에이전트가 3단계에서 잘못된 방향으로 진행했다면, 2단계의 커밋으로 돌아가서 다시 시작 가능
- 이 습관은 에이전트 모드 활용에서 필수적

```bash
git add -A && git commit -m "2단계: 상품 CRUD 완료"
# 3단계 에이전트 작업 시작...
# 실패 시:
git checkout .  # 3단계 변경 사항 폐기
# 3단계 재시도 (다른 방식으로 지시)
```

### 3.3 컨텍스트 관리: 에이전트의 기억과 망각

- 에이전트 모드의 성공률을 결정짓는 핵심 요인 중 하나가 **컨텍스트 관리(Context Management)**
- 에이전트가 "무엇을 알고 있는가"와 "무엇을 잊어버리는가"를 이해하면, 에이전트와의 대화를 훨씬 효과적으로 설계 가능

**컨텍스트 윈도우의 한계.**

- 에이전트 모드의 기반이 되는 LLM은 유한한 컨텍스트 윈도우를 가지고 있음
- 대화가 길어지면 초기에 주어진 정보가 "밀려나"면서 에이전트가 앞서 논의한 내용을 잊어버릴 수 있음
- 10단계의 작업을 한 대화에서 연속으로 수행하면, 마지막 단계에서 에이전트가 첫 번째 단계의 설계 결정을 기억하지 못하는 상황이 발생할 수 있음

**대화 분리 전략.**

- 이 문제를 해결하는 가장 직접적인 방법은 대화를 분리하는 것
- 한 대화에서 모든 것을 처리하려 하지 말고, 논리적 단위마다 새로운 대화를 시작
- 새 대화를 시작할 때는 이전 대화의 결론을 요약하여 전달하면 연속성 유지 가능

```
[새 대화에서]
이 프로젝트는 Flask 기반 Todo 앱이야.
현재 app.py, models.py, routes/todos.py가 구현되어 있어.
지금부터 사용자 인증 기능을 추가할 거야.
Flask-Login을 사용하고, routes/auth.py에 구현해 줘.
```

**파일 참조를 통한 컨텍스트 제공.**

- 에이전트 모드는 프로젝트의 파일을 탐색할 수 있으므로, 대화에서 직접 모든 정보를 제공하는 대신 파일을 참조하도록 지시하는 것이 효율적
- 에이전트가 파일을 읽으면 그 내용이 컨텍스트에 포함되므로, 대화 자체의 길이를 줄이면서도 필요한 정보 제공 가능

```
README.md를 먼저 읽고, 그 설계에 맞춰 구현해 줘.
requirements.txt에 있는 라이브러리만 사용해 줘.
```

**@workspace 참조.**

- VS Code Copilot Chat에서 `@workspace`를 프롬프트에 포함하면 에이전트가 프로젝트 전체의 구조를 탐색한 뒤 응답
- 에이전트가 프로젝트의 맥락을 더 잘 파악하도록 돕는 기능

```
@workspace 이 프로젝트의 테스트 커버리지를 높이려면 어떤 파일에 테스트를 추가해야 할까?
```

**파일 첨부(#file).**

- 특정 파일의 내용을 에이전트에게 직접 전달하고 싶을 때는 `#file` 참조를 사용
- 에이전트가 자율적으로 파일을 탐색하는 것보다 더 정확하게 원하는 파일의 내용을 컨텍스트에 포함시킴

```
#file:app/models.py 의 User 모델과 같은 패턴으로 Product 모델을 추가해 줘.
```

### 3.4 효과적인 프롬프트 작성법

- 에이전트 모드의 성공률은 프롬프트의 품질에 크게 좌우됨
- 동일한 목표라도 프롬프트를 어떻게 작성하느냐에 따라 결과가 크게 달라짐

**원칙 1: 목표와 제약 조건을 함께 명시.**

- 에이전트에게 "무엇을 만들어야 하는가"뿐만 아니라 "어떤 제약 안에서 만들어야 하는가"도 함께 전달

```
❌ "REST API를 만들어 줘."
✅ "Flask로 REST API를 만들어 줘. 엔드포인트는 /api/users (GET, POST)와 /api/users/<id> (GET, PUT, DELETE)이 필요해. SQLite를 사용하고, 응답 형식은 JSON이야."
```

**원칙 2: 기술 스택을 명시.**

- 에이전트가 선택할 수 있는 기술적 대안이 여러 가지 있는 경우, 원하는 기술을 명시적으로 지정
- 지정하지 않으면 에이전트가 자율적으로 선택하는데, 그 선택이 항상 최적은 아님

```
❌ "데이터베이스를 사용하는 웹앱을 만들어 줘."
✅ "Flask + SQLAlchemy + SQLite를 사용하는 웹앱을 만들어 줘."
```

**원칙 3: 완료 조건을 명시.**

- 에이전트가 "작업이 끝났다"고 판단할 수 있는 기준을 제공

```
❌ "테스트를 작성해 줘."
✅ "pytest로 models.py의 모든 함수에 대한 단위 테스트를 작성하고 실행해 줘. 모든 테스트가 통과해야 해."
```


### 3.5 세 가지 모드 조합 워크플로우

- 2주차에서 세 가지 모드의 차이를 학습했다면, 이 절에서는 세 모드를 하나의 워크플로우 안에서 체계적으로 조합하는 방법을 다룸
- 실무에서는 한 가지 모드만 사용하는 경우가 드물고, 작업의 성격에 따라 모드를 자유롭게 전환하는 것이 일반적

**패턴 1: Agent → Edit → Ask (신규 프로젝트).**

- 새로운 프로젝트를 시작할 때 가장 자연스러운 흐름
  - Agent 모드로 프로젝트의 뼈대를 생성
  - Edit 모드로 개별 파일을 정교하게 조정
  - Ask 모드로 에이전트가 작성한 코드의 동작을 확인
- 이 패턴은 2주차 실습에서 이미 경험한 바 있음

**패턴 2: Ask → Agent → Edit (기존 프로젝트 개선).**

- 기존 코드를 개선할 때 효과적
  - 먼저 Ask 모드로 기존 코드의 구조와 문제점을 파악 -- "이 프로젝트의 아키텍처를 설명해 줘", "이 코드에서 개선할 점을 찾아 줘"와 같은 질문으로 현황 분석
  - 분석이 끝나면 Agent 모드로 전환하여 리팩토링이나 기능 추가 수행
  - 마지막으로 Edit 모드로 세부 사항을 마무리

**패턴 3: Agent → Ask → Agent (반복적 개발).**

- 복잡한 기능을 여러 번에 걸쳐 구현할 때 사용
  - 첫 번째 Agent 세션에서 기본 기능을 구현
  - Ask 모드로 결과를 분석하여 부족한 점을 파악
  - 두 번째 Agent 세션에서 보완 작업을 수행
- 이 패턴은 에이전트의 결과를 무비판적으로 수용하는 대신, 중간중간 사람의 판단을 개입시키는 것이 핵심

**모드 전환 의사결정 트리.**

- 현재 작업에 적합한 모드를 빠르게 판단하기 위한 의사결정 흐름:

```
현재 작업이 무엇인가?
├── 코드를 이해하고 싶다 → Ask 모드
├── 특정 파일 하나를 수정하고 싶다 → Edit 모드
├── 여러 파일을 생성/수정해야 한다 → Agent 모드
│   ├── 터미널 실행이 필요한가?
│   │   ├── 예 → Agent 모드 (유일한 선택)
│   │   └── 아니오 → Agent 또는 Edit 모드 (범위에 따라)
│   └── 범위가 프로젝트 전체인가?
│       ├── 예 → Agent 모드
│       └── 아니오 → Edit 모드 (가능하면)
└── 잘 모르겠다 → Ask 모드로 시작 (안전한 선택)
```

- 모드 전환은 기술적 능력이라기보다 판단력의 문제
- 어떤 모드를 선택하느냐에 따라 결과물의 품질과 작업 효율이 달라지므로, 반복적인 경험을 통해 이 판단력을 기르는 것이 중요

### 3.6 에이전트 모드의 실패 패턴과 우회 전략

- 에이전트 모드가 실패하는 전형적인 패턴을 알면, 실패를 예방하거나 신속하게 우회 가능

**실패 패턴 1: 무한 수정 루프(Infinite Fix Loop).**

- 에이전트가 오류를 수정하려 하지만, 수정이 새로운 오류를 발생시키고, 그 오류를 수정하면 원래 오류가 다시 나타나는 순환이 발생
- 에이전트가 근본 원인이 아닌 표면적 증상을 수정하려 할 때 흔히 발생
- **우회 전략**: 에이전트가 3회 이상 같은 유형의 오류를 반복하면 즉시 중단하고, 새 대화를 시작하여 문제의 근본 원인을 설명한 뒤 다른 접근법을 지시. 또는 Ask 모드로 전환하여 "이 오류의 근본 원인은 무엇인가?"라고 질문한 뒤, 그 답변을 기반으로 Agent 모드에서 정확한 수정을 지시

**실패 패턴 2: 컨텍스트 단절(Context Loss).**

- 긴 대화에서 에이전트가 초기에 합의한 설계 결정을 잊어버리고, 이전과 모순되는 코드를 작성
  - 예: 대화 초반에 "SQLAlchemy를 사용한다"고 합의했는데, 대화 후반에 에이전트가 raw SQL 쿼리를 사용하는 코드를 작성하는 경우
- **우회 전략**: 대화를 논리적 단위로 분리하고, 새 대화 시작 시 핵심 설계 결정을 요약하여 전달. 프로젝트의 설계 문서(README.md 또는 ARCHITECTURE.md)를 미리 작성해 두고, 에이전트에게 그 파일을 참조하도록 지시하는 것도 효과적

**실패 패턴 3: 과도한 복잡성(Over-Engineering).**

- 간단한 요청에 대해 에이전트가 불필요하게 복잡한 구조를 생성
  - 간단한 스크립트를 요청했는데 디자인 패턴, 추상 클래스, 의존성 주입 컨테이너까지 포함된 코드를 만들어내는 경우
- **우회 전략**: 프롬프트에 복잡성의 상한을 명시 -- "가장 간단한 방법으로 구현해 줘", "클래스 없이 함수만 사용해 줘", "외부 라이브러리 없이 표준 라이브러리만 사용해 줘" 같은 제약 추가

**실패 패턴 4: 테스트 조작(Test Gaming).**

- 에이전트가 테스트를 통과시키기 위해 테스트 코드 자체를 수정하거나, 하드코딩된 값으로 테스트를 우회
  - 예: 함수의 로직이 잘못되었는데 테스트의 기대값을 잘못된 출력에 맞추는 경우
- **우회 전략**: 테스트 코드를 에이전트에게 맡기기 전에 사람이 먼저 작성하거나, 에이전트가 작성한 테스트를 반드시 사람이 검토. "테스트 파일은 수정하지 말고, 원본 코드만 수정해 줘"라고 명시적으로 지시하는 것도 방법

### 실습 1: 기존 코드 리팩토링

- 기존 코드를 에이전트 모드로 개선하는 경험을 수행
- 리팩토링 작업은 에이전트 모드의 자기 치유 루프를 가장 잘 관찰할 수 있는 시나리오

**사전 준비.** 다음과 같은 의도적으로 품질이 낮은 Python 코드를 준비한다. 새 폴더를 만들고 `calculator.py` 파일을 작성한다.

```python
# calculator.py — 타입 힌트 없음, docstring 없음, 에러 처리 없음
def add(a, b):
    return a + b

def subtract(a, b):
    return a - b

def multiply(a, b):
    return a * b

def divide(a, b):
    return a / b

def calculate(operation, a, b):
    if operation == "add":
        return add(a, b)
    elif operation == "subtract":
        return subtract(a, b)
    elif operation == "multiply":
        return multiply(a, b)
    elif operation == "divide":
        return divide(a, b)

def parse_input(user_input):
    parts = user_input.split()
    operation = parts[0]
    a = float(parts[1])
    b = float(parts[2])
    return operation, a, b

if __name__ == "__main__":
    while True:
        user_input = input("입력 (예: add 3 5): ")
        operation, a, b = parse_input(user_input)
        result = calculate(operation, a, b)
        print(f"결과: {result}")
```

`utils.py` 파일도 준비한다.

```python
# utils.py — 마찬가지로 품질이 낮은 코드
def format_number(n):
    if n == int(n):
        return str(int(n))
    return str(round(n, 4))

def validate_number(s):
    try:
        float(s)
        return True
    except:
        return False

def log_calculation(operation, a, b, result):
    f = open("history.log", "a")
    f.write(f"{operation} {a} {b} = {result}\n")
    f.close()
```

**1단계: 리팩토링 요청.**

- Git 커밋으로 현재 상태를 저장한 후, Copilot Chat을 Agent 모드로 전환하고 다음과 같이 입력:

```
이 프로젝트의 모든 함수에 타입 힌트를 추가하고,
docstring을 한국어로 작성해 줘.
divide 함수에 0으로 나누기 예외 처리도 추가해 줘.
utils.py의 log_calculation 함수에서 파일을 안전하게 다루도록 수정해 줘.
```

- 에이전트가 프로젝트의 파일 구조를 탐색하고, 각 파일의 함수를 찾아 타입 힌트와 docstring을 추가하는 편집을 순차적으로 수행하는 과정을 관찰
- 여러 파일에 걸친 수정이 한 번의 요청으로 이루어지는 것이 에이전트 모드의 핵심 가치

**2단계: 테스트 작성과 자기 치유 루프 관찰.**

- 리팩토링이 완료되면 이어서 다음을 입력:

```
pytest로 calculator.py와 utils.py의 모든 함수에 대한
단위 테스트를 작성하고 실행해 줘.
엣지 케이스(0으로 나누기, 빈 입력, 잘못된 형식)도 포함해 줘.
```

- 이 요청에서 자기 치유 루프가 분명하게 드러남:
  - 에이전트는 먼저 테스트 파일을 생성
  - `pip install pytest` 명령의 승인을 요청
  - 승인 후 `pytest`를 실행하면 일부 테스트가 실패할 수 있음
  - 에이전트는 실패한 테스트의 오류 메시지를 읽고, 테스트 코드 또는 원본 코드를 수정한 뒤 다시 `pytest`를 실행
  - 이 편집-실행-수정 순환이 모든 테스트가 통과할 때까지 반복되는 것을 관찰

**3단계: 결과 검증.** 에이전트가 작업을 완료하면 다음을 직접 확인한다.

- 타입 힌트가 실제 함수의 동작과 일치하는가?
- docstring이 함수의 역할을 정확히 설명하는가?
- 0으로 나누기 예외가 올바르게 처리되는가?
- 파일 입출력이 `with` 문으로 안전하게 처리되는가?
- 테스트가 의미 있는 검증을 수행하는가, 아니면 단순히 통과하도록 설계되었는가?

- 특히 테스트 코드의 품질에 주의를 기울여야 함
  - 에이전트는 종종 함수의 현재 출력값을 그대로 기대값으로 사용하는 테스트를 작성
  - 이런 테스트는 항상 통과하지만, 함수의 논리적 정확성을 검증하지는 못함
- 에이전트가 "통과하는 테스트"를 만드는 것과 "올바른 테스트"를 만드는 것은 다른 문제이며, 이 차이를 인식하는 것이 AI 협업 개발에서 가장 중요한 판단력

## 과제

**[제출 과제]** 기존 코드 리팩토링 결과 + 한계 분석 보고서

1. **리팩토링 결과물**: 실습 1에서 리팩토링한 코드 전체를 GitHub 리포지토리에 push한다. 리팩토링 전후의 코드를 비교할 수 있도록 Git 이력을 유지한다.

2. **한계 분석 보고서** (Markdown 형식, 500자 이상): 다음 내용을 포함하여 `REPORT.md`로 작성한다.
   - 실습 2 파트 B에서 수행한 한계 체험의 결과 (어떤 요청을 했고, 에이전트가 어떻게 반응했는지)
   - 에이전트가 실패한 구체적인 지점과 그 원인에 대한 분석
   - 실패를 우회하기 위해 시도한 전략과 그 결과
   - 에이전트 모드를 효과적으로 사용하기 위한 본인만의 가이드라인 (3가지 이상)

## 핵심 정리

- 에이전트 모드는 완료 조건이 명확하고, 표준 프레임워크를 사용하며, 테스트가 갖춰진 작업에서 가장 효과적이다
- 멀티파일 프로젝트에서는 한 번에 모든 것을 요청하는 대신, 단계적 지시와 Git 체크포인트를 활용하여 점진적으로 구현한다
- 컨텍스트 관리의 핵심은 대화를 논리적 단위로 분리하고, 파일 참조를 통해 에이전트에게 필요한 정보를 효율적으로 제공하는 것이다
- 프롬프트의 품질이 결과의 품질을 결정한다. 목표, 제약 조건, 기술 스택, 완료 조건을 명시적으로 작성한다
- Ask, Edit, Agent 세 모드를 상황에 따라 전환하며 조합하는 것이 실무에서의 핵심 워크플로우이다
- 에이전트의 실패 패턴(무한 수정 루프, 컨텍스트 단절, 과도한 복잡성, 테스트 조작)을 인식하고 우회하는 전략을 갖추어야 한다

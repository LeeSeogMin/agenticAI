# 15주차. 기말고사 + 프로젝트 발표

> **1회차** (90분): 기말고사 -- 이론 30% + 실기 70%, 범위: 1~12주차 전체
> **2회차** (90분): 프로젝트 최종 발표 (팀/개인당 10분 발표 + 5분 질의응답)

---

## 1회차: 기말고사

### 15.1 시험 개요

- 범위: 1주차부터 12주차까지의 전체 내용
- 구성: 이론(30%) + 실기(70%)
- 중간고사(이론 40% + 실기 60%)와 비교하여 실기 비중이 더 높음
  - 이유: 9주차 이후 학습한 Skills와 통합 워크플로우까지 실전적으로 구현할 수 있는 능력을 평가하기 위함

**표 15.1** 기말고사 구성

| 구분 | 비중 | 시간 | 문항 구성 |
|------|:----:|:----:|----------|
| 이론 | 30% | 25분 | 객관식 10문항 + 단답형 5문항 |
| 실기 | 70% | 65분 | MCP + Skills 통합 과제 1문항 |

### 15.2 이론 시험 범위와 유형

- 핵심 개념의 이해도를 확인하는 문항으로 구성
- 출제 범위: 네 영역

**영역 1: Agent Mode와 Copilot 기초 (1~3주차)**

- 에이전트 모드의 자율 실행 루프(계획 → 편집 → 실행 → 수정)
- 세 가지 모드(Ask/Edit/Agent)의 차이
- Agent Mode가 효과적인 상황과 한계

**영역 2: MCP 아키텍처와 구현 (4~7주차)**

- MCP의 클라이언트-서버 아키텍처
- 전송 방식(STDIO/SSE/Streamable HTTP)
- 3대 프리미티브(Tool/Resource/Prompt)의 차이
- Python SDK를 사용한 서버 구현 원리
- 에러 처리와 입력 검증 전략

**영역 3: Skills 설계와 활용 (9~10주차)**

- SKILL.md의 구조(메타데이터, 트리거, 단계별 절차)
- 3단계 점진적 공개
- 세 가지 설계 패턴(체크리스트형, 워크플로우형, 가드레일형)
- MCP(도구)와 Skills(절차)의 역할 구분

**영역 4: 통합과 팀 워크플로우 (11~12주차)**

- MCP + Skills 통합 설계 패턴(단일도구, 파이프라인, 감독자)
- Coding Agent의 동작 원리(이슈 → Draft PR)
- 이슈 할당자의 공동 작성자 등록과 독립적 리뷰 원칙
- 팀 워크플로우에서의 안전장치

### 15.3 이론 예시 문항

- 아래는 출제 방향과 난이도를 가늠하기 위한 예시 문항이며, 실제 시험과 동일한 문항은 아님

**[객관식 예시 1]** MCP의 3대 프리미티브에 해당하지 않는 것은?

(a) Tool
(b) Resource
(c) Prompt
(d) Skill

정답: (d) -- Skill은 Copilot의 개념이며 MCP 프로토콜의 프리미티브가 아니다.

**[객관식 예시 2]** Coding Agent가 생성한 PR에 대해 올바른 설명은?

(a) 이슈 할당자가 곧바로 머지할 수 있다
(b) 독립적인 리뷰어의 승인이 필요하다
(c) 자동으로 메인 브랜치에 머지된다
(d) 브랜치 보호 규칙이 적용되지 않는다

정답: (b) -- 이슈 할당자는 공동 작성자로 등록되어 해당 PR을 직접 승인할 수 없다.

**[단답형 예시 1]** MCP 서버의 세 가지 전송 방식을 모두 작성하시오.

정답: STDIO, SSE, Streamable HTTP

**[단답형 예시 2]** MCP(도구)와 Skills(절차)의 역할 차이를 한 문장으로 설명하시오.

정답 예시: MCP는 에이전트가 외부 시스템과 상호작용하기 위한 도구를 제공하고, Skills는 그 도구를 활용하는 절차와 규칙을 정의한다.

### 15.4 실기 시험 구성

- MCP 서버와 Skills를 통합하는 과제를 90분 내에 구현하는 형식
- 주어진 요구사항에 따라 MCP 서버를 구현하고, SKILL.md를 작성하고, mcp.json으로 연결하여 Copilot에서 동작을 확인하는 전체 과정을 수행

**표 15.2** 실기 시험 평가 기준

| 항목 | 비중 | 세부 내용 |
|------|:----:|----------|
| MCP 서버 구현 | 30% | 도구 등록, 입력 검증, 에러 처리 |
| Skills 작성 | 20% | 절차 정의의 명확성, 트리거 조건 |
| 통합 동작 | 30% | mcp.json 설정, 종단간 동작 확인 |
| 코드 품질 | 20% | 구조, 가독성, 주석 |

### 15.5 실기 예시 문항

- 아래는 실기 시험의 유형과 난이도를 보여주는 예시

**[실기 예시]** 도서 검색 에이전트 구현

다음 요구사항에 따라 MCP 서버와 Skills를 구현하시오.

1. **MCP 서버**: `book_search` 도구를 구현한다
   - 입력: `query` (검색어, 문자열), `max_results` (최대 결과 수, 정수, 기본값 5)
   - 출력: 도서 제목, 저자, 출판연도를 포함하는 리스트
   - 데이터는 제공되는 `books.json` 파일에서 검색한다
   - 검색어가 빈 문자열이면 적절한 에러 메시지를 반환한다

2. **Skills**: `book-search-skill.md`를 작성한다
   - 트리거: "도서 검색", "책 찾기" 등의 키워드
   - 절차: 검색어 확인 → book_search 도구 호출 → 결과 포매팅 → 사용자에게 반환
   - 결과가 없는 경우의 처리 절차를 포함한다

3. **설정**: `mcp.json`에 서버를 등록하고 Copilot에서 동작을 확인한다

- 이 문항은 6주차(MCP 서버 구현), 9주차(Skills 작성), 11주차(통합)의 내용을 종합적으로 평가
- 완벽한 구현이 아니더라도 각 구성 요소가 의도대로 설계되어 있으면 부분 점수를 부여

### 15.6 시험 유의사항

- 인터넷 검색은 허용하지 않음 (단, 공식 문서의 오프라인 사본은 참고 가능: MCP Specification, Python SDK 문서)
- VS Code와 Copilot을 사용하여 코드를 작성할 수 있음 (단, Copilot이 생성한 코드의 정확성에 대한 책임은 수험자에게 있음)
- 실기 시험의 코드는 시험 종료 후 GitHub 저장소에 커밋하여 제출
- 시간 배분 유의: 이론 25분, 실기 65분

## 2회차: 프로젝트 최종 발표

### 15.7 발표 진행 방식

- 팀/개인당 10분 발표 + 5분 질의응답으로 구성
- 발표 순서는 사전에 추첨으로 결정하며, 순서와 관계없이 동일한 기준으로 평가

**표 15.3** 최종 발표 진행표

| 순서 | 시간 | 내용 | 발표자 |
|:----:|:----:|------|--------|
| 1 | 2분 | 문제 정의와 동기 | 발표자 |
| 2 | 3분 | 시스템 아키텍처: MCP 서버 + Skills 구성 | 발표자 |
| 3 | 3분 | 데모 시연 | 발표자 |
| 4 | 2분 | 한계점 및 향후 과제 | 발표자 |
| 5 | 5분 | 질의응답 | 교수자 + 동료 학생 |

- 시간 관리는 엄격하게 적용
  - 10분 초과 시 발표를 중단하며, 시간 관리 항목에서 감점 요인
  - 8분 미만으로 발표가 종료되는 경우에도 내용의 충실성에서 감점될 수 있음
  - 사전 리허설을 통해 적절한 분량을 조절해야 함

### 15.8 발표 평가 기준

- 교수자 평가와 동료 평가(Peer Evaluation)로 구성

**표 15.4** 발표 평가 구성

| 평가 주체 | 비중 | 평가 항목 |
|----------|:----:|----------|
| 교수자 | 80% | 기술적 완성도, 설계 근거, 데모 품질, 질의응답 대응 |
| 동료 학생 | 20% | 발표의 명확성, 데모의 인상, 전반적 완성도 |

- 교수자 평가에서 중점적으로 확인하는 항목
  1. **MCP 서버의 도구 설계가 문제 해결에 적합한가**
     - 도구의 입출력이 명확하게 정의되어 있는지 확인
     - 에러 처리가 실전 수준으로 구현되어 있는지 점검
  2. **Skills의 절차가 에이전트의 동작을 효과적으로 안내하는가**
     - 절차가 모호하지 않은지 확인
     - 예외 상황 처리와 다른 프로젝트에도 참고 가능한 범용성이 있는지 평가
  3. **MCP와 Skills의 통합이 유기적인가**
     - 두 요소가 단순 공존이 아니라 하나의 자동화 워크플로우를 이루는지 확인
     - 도구(MCP)와 절차(Skills)가 서로 보완적으로 작동하는지 점검
  4. **기술적 선택의 근거를 논리적으로 설명할 수 있는가**
     - "왜 이 설계 패턴을 선택했는가"
     - "이 도구의 입력 스키마를 이렇게 정의한 이유는 무엇인가"
     - 위와 같은 질문에 명확하게 답변할 수 있어야 함

### 15.9 동료 평가 (Peer Evaluation)

- 모든 발표가 종료된 후 실시
- 각 학생은 자신을 제외한 모든 발표에 대해 평가 양식을 작성

**표 15.5** 동료 평가 양식

| 평가 항목 | 점수 (1-5) | 설명 |
|----------|:----------:|------|
| 문제 정의의 명확성 | | 해결하려는 문제가 분명한가 |
| 아키텍처 설계의 적절성 | | MCP + Skills 구성이 합리적인가 |
| 데모 시연의 완성도 | | 실제로 동작하는가, 인상적인가 |
| 발표의 전달력 | | 논리적이고 이해하기 쉬운가 |
| 전반적 완성도 | | 프로젝트의 종합적인 수준 |

- 동료 평가는 익명으로 진행
- 극단적인 점수(모든 항목에 1점 또는 5점)는 통계적으로 보정
- 동료 평가의 목적: 순위를 매기는 것이 아니라, 다양한 관점에서 프로젝트를 검토하는 경험을 제공하는 것

### 15.10 발표 시 주의사항

1. **데모 환경을 사전에 점검**
   - 발표 직전에 MCP 서버, API 키, 네트워크 연결 상태를 확인
   - 발표장 네트워크가 다를 수 있으므로 녹화 영상이나 스크린샷 같은 백업 수단도 준비
2. **데모 시나리오를 스크립트로 작성**
   - 어떤 입력을 넣고 어떤 결과를 보여줄지 미리 정리
   - 시연 중 당황하거나 시간을 낭비하는 상황을 줄일 수 있음
3. **한계점을 정직하게 밝히기**
   - 완벽한 프로젝트는 없으므로 한계를 인정하고 개선 방향을 제시하는 것이 중요
   - 예: "이 에이전트는 현재 한국어만 지원하며, 다국어 처리를 위해서는 추가적인 Skills 설계가 필요하다"
4. **질의응답에 대비**
   - 예상 질문을 3-5개 미리 준비하고 답변을 정리
   - 답을 모를 때는 추가 조사가 필요하다고 솔직하게 말하는 편이 억지 답변보다 낫다

## 과목 마무리

### 15.11 학기 회고

- 이 과목을 통해 학습한 네 가지 핵심 역량

1. **Agent Mode**: 자연어로 소프트웨어를 생성하고, 자율 실행 루프의 원리를 이해함
2. **MCP**: 에이전트의 능력을 외부로 확장하는 표준 프로토콜을 설계하고 구현함
3. **Skills**: 에이전트에게 도메인 지식과 절차를 주입하는 방법을 익힘
4. **Coding Agent**: 이슈에서 PR까지의 비동기 자동화 파이프라인을 구축함

- 이 네 가지 요소는 독립적으로 존재하는 것이 아니라, 하나의 에이전트 생태계를 구성
  - MCP가 에이전트의 "손"이라면, Skills는 에이전트의 "두뇌"에 해당
  - Agent Mode와 Coding Agent는 이 손과 두뇌를 활용하는 두 가지 작업 방식
- AI 에이전트 기술은 빠르게 진화하고 있음
  - 이 과목에서 다룬 도구와 프로토콜의 세부 사항은 시간이 지남에 따라 변할 수 있음
  - 그러나 "문제를 정의하고, 도구를 설계하고, 절차를 명확히 하고, 결과를 검증하는" 사고 방식은 어떤 도구를 사용하든 유효
  - 이 과목이 그러한 사고의 출발점이 되었기를 바람

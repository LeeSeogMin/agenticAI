# 14주차. 프로젝트 개발과 중간 점검

> **1회차** (90분): 팀/개인별 중간 점검 발표, 피드백 세션, 공통 이슈 해결
> **2회차** (90분): 프로젝트 집중 개발, 최종 디버깅, 발표 자료 준비

---

## 학습목표

1. 프로젝트의 현재 진행 상황을 구조적으로 정리하여 발표한다
2. 피드백을 수용하고 프로젝트에 반영하는 능력을 기른다
3. 프로젝트를 완성하고 발표를 준비한다

## 선수지식

13주차에서 작성한 프로젝트 설계서가 확정되어 있어야 하며, 최소한 MCP 서버의 골격 코드와 SKILL.md 초안이 GitHub 저장소에 커밋되어 있어야 한다.

---

## 1회차: 중간 점검 발표와 피드백

### 14.1 중간 점검 발표

중간 점검 발표는 팀/개인당 5분으로 제한한다. 이 발표의 목적은 완성된 결과물을 보여주는 것이 아니라, 현재 진행 상황을 공유하고 문제점에 대한 피드백을 받는 것이다.

발표에 포함해야 하는 항목은 다음과 같다.

**표 14.1** 중간 점검 발표 구성

| 순서 | 시간 | 내용 |
|:----:|:----:|------|
| 1 | 1분 | 프로젝트 개요 (문제 정의, 에이전트의 목적) |
| 2 | 2분 | 현재까지 완성된 부분 (동작하는 기능 위주) |
| 3 | 1분 | 현재 겪고 있는 문제점 또는 기술적 난관 |
| 4 | 1분 | 남은 작업과 완성 계획 |

중간 점검은 성적에 직접 반영되지 않지만, 발표를 통해 받는 피드백이 최종 결과물의 품질에 결정적인 영향을 미친다. 다른 프로젝트의 접근 방식에서 아이디어를 얻을 수도 있으므로, 동료 학생의 발표에도 적극적으로 참여하는 것을 권장한다.

### 14.2 피드백 세션

각 발표가 끝난 후 2-3분의 피드백 시간을 운영한다. 피드백은 교수자뿐 아니라 동료 학생도 참여할 수 있다.

효과적인 피드백을 제공하기 위해 다음 관점에서 질문하고 의견을 제시한다.

- **설계의 적절성**: MCP 도구와 Skills의 역할 분담이 명확한가? 더 나은 설계 패턴이 있는가?
- **구현의 실현 가능성**: 남은 시간 안에 계획한 기능을 모두 완성할 수 있는가? 범위 축소가 필요하지 않은가?
- **사용자 관점**: 에이전트의 입출력이 직관적인가? 실제 사용 시나리오에서 자연스러운가?
- **기술적 조언**: 비슷한 문제를 다른 방식으로 해결한 경험이 있는가? 참고할 만한 자료가 있는가?

피드백을 받는 입장에서는 방어적으로 반응하지 않고, 건설적인 의견을 메모하여 2회차 개발에 반영하는 것이 중요하다.

### 14.3 공통 이슈 해결

중간 점검 발표를 통해 여러 프로젝트에서 공통적으로 발생하는 문제가 확인되면, 교수자가 이를 정리하여 해결 방안을 안내한다. 다음은 이전 학기에서 빈번하게 나타났던 공통 이슈와 그 대응 방안이다.

**표 14.2** 프로젝트 공통 이슈와 해결 방안

| 이슈 | 원인 | 해결 방안 |
|------|------|----------|
| MCP 서버가 Copilot에서 인식되지 않음 | mcp.json 경로 오류 또는 서버 실행 실패 | `.vscode/mcp.json`의 경로 확인, MCP Inspector로 서버 단독 테스트 |
| 도구 호출 시 타임아웃 발생 | 외부 API 응답 지연 또는 무한 루프 | 타임아웃 설정 추가 (기본 30초), 비동기 처리 도입 |
| Skills가 의도대로 트리거되지 않음 | 트리거 조건이 모호하거나 충돌 | 트리거 키워드를 구체화, 다른 Skills와의 충돌 여부 확인 |
| 에러 처리가 미흡하여 에이전트가 멈춤 | 예외 상황에 대한 폴백 미정의 | try-except로 모든 외부 호출 감싸기, 사용자 친화적 오류 메시지 반환 |
| Git 충돌로 팀원 간 작업 병합 실패 | 동일 파일 동시 수정 | 기능별 브랜치 분리, 자주 Pull/Merge 수행 |

### 14.4 디버깅 팁

프로젝트 완성 과정에서 디버깅에 소요되는 시간을 줄이기 위해, 다음 전략을 활용한다.

첫째, MCP Inspector를 적극 활용한다. Copilot에 연결하기 전에 Inspector로 각 도구를 독립적으로 테스트하면, 문제의 원인이 MCP 서버에 있는지 Skills에 있는지를 신속하게 분별할 수 있다.

둘째, 로깅을 체계적으로 적용한다. MCP 서버의 도구 호출 시점, 입력값, 반환값, 에러 정보를 로그로 기록하면 문제 추적이 용이해진다. Python의 `logging` 모듈을 사용하되, 로그 레벨(DEBUG, INFO, WARNING, ERROR)을 적절히 구분한다.

셋째, 단계별로 검증한다. MCP 서버 단독 동작 → Skills 단독 동작 → 통합 동작의 순서로 검증하면, 통합 과정에서 발생하는 문제를 빠르게 격리할 수 있다.

넷째, 에러 메시지를 정확히 읽는다. 스택 트레이스의 마지막 줄뿐 아니라 전체 호출 경로를 확인하면, 에러의 근본 원인(root cause)에 더 빨리 도달할 수 있다.

## 2회차: 프로젝트 집중 개발

### 14.5 개발 우선순위 결정

2회차는 프로젝트를 완성하는 마지막 실습 시간이다. 남은 시간을 효과적으로 활용하기 위해 다음 우선순위에 따라 작업한다.

**표 14.3** 개발 우선순위

| 우선순위 | 작업 | 기준 |
|:--------:|------|------|
| 1 | 핵심 기능 완성 | 데모에서 반드시 보여줄 기능 |
| 2 | 에러 처리 보강 | 시연 중 오류 발생 방지 |
| 3 | 발표 자료 작성 | 슬라이드 10장 이내 |
| 4 | README.md 작성 | 실행 방법과 의존성 목록 |
| 5 | 부가 기능 추가 | 시간이 남는 경우에만 |

핵심 원칙은 "동작하는 데모"를 최우선으로 확보하는 것이다. 모든 기능이 완벽하지 않더라도 핵심 워크플로우가 종단간(end-to-end)으로 동작하는 것이 가장 중요하다. 부가 기능은 핵심 기능이 안정적으로 동작한 이후에만 추가한다.

### 14.6 최종 디버깅과 안정화

데모 시연의 안정성을 확보하기 위해 다음 체크리스트를 점검한다.

- MCP 서버가 정상적으로 시작되고 종료되는가
- 모든 도구가 정상 입력과 비정상 입력 모두에 대해 적절히 응답하는가
- Skills의 각 단계가 순서대로 실행되는가
- API 키와 환경 변수가 올바르게 설정되어 있는가
- 네트워크 연결이 끊어진 경우에도 에러 메시지가 적절히 표시되는가

시연 환경은 본인의 개발 환경과 다를 수 있으므로, 가능하면 다른 컴퓨터에서 한 번 이상 실행하여 환경 의존성 문제를 사전에 발견하는 것이 바람직하다.

### 14.7 발표 자료 준비

발표 자료는 13주차에서 안내한 구성(문제 정의 → 아키텍처 → 데모 → 한계점)을 따른다. 슬라이드 작성 시 다음 사항에 유의한다.

- 슬라이드 한 장에 하나의 핵심 메시지만 담는다
- 아키텍처 다이어그램은 텍스트보다 시각적으로 표현한다
- 데모 시연 순서를 슬라이드에 미리 정리하여 진행 흐름을 잃지 않도록 한다
- 실시간 시연이 실패할 경우를 대비하여 스크린샷 또는 녹화 영상을 백업한다
- 발표 시간(10분)을 반드시 리허설로 확인한다 — 초과 시 감점 요인이 된다

### 14.8 최종 제출

14주차 2회차 종료 시점까지 다음 사항을 완료해야 한다.

- GitHub 저장소에 모든 코드를 최종 커밋한다
- README.md에 프로젝트 소개와 실행 방법을 기술한다
- 발표 자료 초안을 완성한다 (최종 수정은 15주차 발표 전까지 가능)

---

## 과제

**제출물**: 프로젝트 코드 최종 커밋 + 발표 자료 초안

- 코드: GitHub 저장소에 최종 커밋 (14주차 2회차 수업 종료 시까지)
- 발표 자료: 슬라이드 초안 (PDF 형식으로 저장소에 포함)
- 필수 포함: MCP 서버 코드, SKILL.md, mcp.json, README.md

---

## 핵심 정리

- 중간 점검 발표(5분)를 통해 진행 상황을 공유하고 피드백을 받는다
- 피드백을 건설적으로 수용하여 프로젝트에 반영하는 것이 최종 품질을 좌우한다
- 개발 우선순위는 핵심 기능 완성 → 에러 처리 → 발표 자료 → README → 부가 기능 순서이다
- "동작하는 데모"가 가장 중요하다 — 종단간 워크플로우의 안정성을 최우선으로 확보한다

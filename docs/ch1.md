# 제1장. AI 협업이 바꾸는 개발 프로세스

## 학습목표

- AI와 협업하는 개발에서 사람이 책임져야 하는 판단과 검증의 범위를 설명한다.
- 에이전틱 코딩 도구의 유형(IDE 기반, 터미널 기반)과 선택 기준을 이해한다.
- 바이브 코딩의 개념과 실무 한계를 파악한다.
- 환각, 과신, 컨텍스트 누락 같은 실패 패턴을 식별하고 대응 원칙을 정리한다.

## 선수지식

기본적인 파이썬 실행 경험(가상환경, 패키지 설치)이 있으면 충분하다. 본 장은 특정 프레임워크를 깊게 다루기보다, 이후 장에서 반복될 "검증 가능한 개발 흐름"을 먼저 만든다.

---

## 1.1 개발 패러다임의 변화: 타이핑에서 설계·검증으로

AI 코딩 도구를 처음 접하면 "코드를 대신 써준다"는 점이 가장 눈에 띈다. 그러나 실무에서 더 큰 변화는 타이핑 속도 자체가 아니라, 설계 대안을 탐색하고 검증을 반복하는 비용이 크게 낮아진다는 데 있다. 같은 시간 안에 더 많은 가설을 세우고, 더 많은 구현 후보를 만들고, 더 많은 검증을 수행할 수 있다면 결과적으로 실패를 더 빨리 발견할 가능성이 커진다.

이때 개발자의 역할은 사라지지 않는다. 오히려 요구사항을 명확히 하고, 위험을 판단하고, 검증 기준을 설계하는 책임이 더 중요해진다. AI는 "좋아 보이는 코드"를 제안할 수 있지만, 그 코드가 지금의 제약(환경, 버전, 보안 정책, 운영 조건)을 만족하는지는 사람이 확인해야 한다.

2024년부터 AI 코딩 도구는 단순한 코드 자동 완성을 넘어, 여러 파일을 동시에 수정하고 터미널 명령을 실행하며 오류를 스스로 수정하는 "에이전틱(agentic)" 단계로 진입했다. 이 변화가 개발자에게 의미하는 바는 명확하다. 생성 능력이 강해질수록, 무엇을 생성할지 판단하고 생성된 결과를 검증하는 능력이 차별화 요인이 된다.

이 책은 이 관점을 출발점으로 삼고, 이후 장에서 MCP와 에이전트 프레임워크를 다루면서도 언제나 "검증 가능한 산출물"을 남기는 습관을 기본값으로 유지한다.

## 1.2 에이전틱 코딩 도구 전경

2025년을 기점으로 주요 AI 코딩 도구가 일제히 에이전트 모드(Agent Mode)를 도입했다. 에이전트 모드란 개발자가 목표를 제시하면, 도구가 코드베이스를 분석하고 여러 파일을 수정하고 터미널 명령을 실행하며 오류를 자체 수정하는 자율적 작업 방식을 말한다. 기존의 "한 줄 자동 완성"과는 근본적으로 다른 패러다임이다.

### 1.2.1 IDE 기반 도구

IDE 기반 도구는 에디터 안에서 코드 편집, 파일 탐색, 터미널 실행을 통합적으로 수행한다.

**GitHub Copilot 에이전트 모드**는 VS Code에 내장된 형태로 2025년 2월에 프리뷰로 공개되었다. 코드베이스를 분석하고, 여러 파일에 걸친 수정을 제안하며, 터미널에서 명령을 실행하고, 테스트가 통과할 때까지 반복하는 자율 루프를 수행한다. 클라우드 기반 코딩 에이전트(Coding Agent)는 GitHub Issues를 할당받아 자율적으로 드래프트 PR을 생성하는 기능으로, GitHub Actions 위에서 비동기로 실행된다.

**Cursor**는 VS Code를 포크한 전용 IDE로, 2023년에 출시되어 2025년 10월 Cursor 2.0으로 업그레이드되었다. Composer 모드에서 여러 파일을 동시에 편집하며, 독립적인 git 워크트리(worktree)에서 최대 8개의 에이전트를 병렬로 실행하는 멀티 에이전트 아키텍처를 갖추고 있다.

**Windsurf**는 Codeium이 2024년 11월에 출시한 IDE로, 핵심 기능인 Cascade는 장기 계획 에이전트와 단기 실행 모델이 분리된 이중 구조다. 실시간으로 코드베이스 상태를 추적하면서 다단계 작업을 수행한다.

### 1.2.2 터미널 기반 도구

터미널 기반 도구는 명령줄에서 동작하며, 기존 워크플로우(git, 셸 스크립트 등)와 자연스럽게 통합된다.

**Claude Code**는 Anthropic이 2025년 2월에 리서치 프리뷰로 공개한 CLI 도구다. 터미널에서 자연어로 지시하면 코드베이스 전체를 이해하고, 파일 읽기/쓰기, git 작업, 셸 명령 실행을 수행한다. 2026년 2월에는 하나의 리드 에이전트가 여러 병렬 세션을 생성하여 협업하는 에이전트 팀(Agent Teams) 기능이 추가되었다.

**Gemini CLI**는 Google이 2025년 6월에 Apache 2.0 라이선스로 공개한 오픈소스 CLI 도구다. 개인 Google 계정으로 무료 사용이 가능하며, MCP 서버 연동을 기본 지원한다. 100만 토큰 컨텍스트 윈도우를 활용할 수 있다는 점이 강점이다.

**OpenAI Codex CLI**는 2025년 4월에 오픈소스로 공개된 Rust 기반 CLI 도구다. 샌드박스 실행 모드를 제공하여 로컬 환경에서 안전하게 에이전트를 실행할 수 있다.

### 1.2.3 도구 선택 기준

에이전틱 코딩 도구의 선택은 프로젝트 성격과 팀 워크플로우에 따라 달라진다. IDE 기반 도구는 시각적 피드백과 에디터 통합이 강점이어서 프론트엔드 개발이나 빠른 프로토타이핑에 적합하다. 터미널 기반 도구는 기존 CLI 워크플로우(git, CI/CD, 셸 스크립트)와 자연스럽게 결합하므로 백엔드 개발이나 대규모 리팩토링에 유리하다.

**표 1.1** 에이전틱 코딩 도구 비교 (2025–2026)

| 도구 | 유형 | 출시 | 무료 티어 | 핵심 특징 |
|:---|:---|:---|:---|:---|
| GitHub Copilot | IDE + 클라우드 | 2025.02 | 월 50회 프리미엄 | 자율 루프 + 클라우드 코딩 에이전트 |
| Cursor | IDE (VS Code 포크) | 2023 (v2.0: 2025.10) | 제한적 | Composer 모델 + 8 병렬 에이전트 |
| Windsurf | IDE (VS Code 포크) | 2024.11 | 월 25 크레딧 | Cascade 이중 구조 에이전트 |
| Claude Code | 터미널 (CLI) | 2025.02 | 웹만 제한적 | 터미널 네이티브 + Agent Teams |
| Gemini CLI | 터미널 (오픈소스) | 2025.06 | 관대한 무료 | 오픈소스 + MCP 기본 지원 |
| OpenAI Codex CLI | 터미널 (오픈소스) | 2025.04 | 구독 필요 | Rust 기반 + 샌드박스 실행 |

어떤 도구를 선택하든 핵심은 동일하다. 에이전트가 생성한 코드를 사람이 검증하는 구조를 유지하는 것이다. 도구의 자율성이 높아질수록 검증 습관의 중요성도 비례하여 커진다.

## 1.3 바이브 코딩의 가능성과 한계

### 1.3.1 바이브 코딩이란

"바이브 코딩(vibe coding)"은 OpenAI 공동 창업자 안드레이 카르파시(Andrej Karpathy)가 2025년 2월 X(구 트위터)에 올린 글에서 처음 사용한 표현이다. 그는 "코드가 존재한다는 사실조차 잊고, 분위기(vibe)에 맡긴다"고 묘사했다. AI에게 자연어로 원하는 것을 말하고, 생성된 코드를 읽지 않고 모두 수락하며, 오류가 나면 오류 메시지를 그대로 붙여넣어 해결하는 방식이다.

이 표현은 수주 만에 뉴욕 타임스, 아스 테크니카, 가디언 등 주요 매체에 소개되었고, 위키피디아 문서가 생성될 만큼 널리 퍼졌다. 프로토타입이나 일회성 스크립트를 빠르게 만들어야 할 때, 바이브 코딩은 진입 장벽을 극적으로 낮춘다. 프로그래밍 경험이 적은 사람도 동작하는 소프트웨어를 만들 수 있다는 점에서 가능성이 있다.

### 1.3.2 실무에서의 한계

그러나 바이브 코딩을 프로덕션 코드에 적용하면 심각한 문제가 발생한다.

**보안 취약점 문제가 첫 번째다.** AI가 생성한 코드에 OWASP Top 10 취약점이 포함될 확률이 높다는 보고가 있다. 코드를 읽지 않고 수락하면 SQL 인젝션, XSS, 인증 우회 같은 취약점이 프로덕션에 그대로 배포될 수 있다.

**유지보수 불가가 두 번째다.** 바이브 코딩으로 생성된 코드베이스는 일관된 구조가 없다. 개발자 자신도 코드의 동작 원리를 이해하지 못하는 상태가 되면, 버그 수정이나 기능 추가 시 전체를 다시 생성해야 하는 상황이 반복된다. 사이먼 윌리슨(Simon Willison)은 "우리가 소프트웨어 엔지니어로서 하는 일의 대부분은 기존 시스템을 진화시키는 것이며, 이때 코드의 품질과 이해 가능성이 결정적"이라고 지적했다.

**생산성 착각이 세 번째다.** METR의 2025년 연구에 따르면, 숙련된 오픈소스 개발자 16명이 실제 작업 246건을 수행한 결과 AI 도구를 사용했을 때 오히려 19% 느려졌다. 흥미로운 점은 개발자들이 스스로 24% 빨라졌다고 예측했고, 실험 후에도 20% 빨라졌다고 느꼈다는 것이다. 체감 속도와 실제 속도 사이에 상당한 괴리가 존재한다.

### 1.3.3 AI 보조 프로그래밍과의 구분

바이브 코딩과 AI 보조 프로그래밍(AI-assisted programming)은 다른 개념이다. AI 보조 프로그래밍은 개발자가 생성된 코드를 이해하고, 검토하고, 필요하면 수정한다. 주도권이 사람에게 있다. 반면 바이브 코딩은 코드 이해를 포기하고 결과만 확인하는 방식이다. 이 책에서 다루는 모든 실습은 AI 보조 프로그래밍 관점을 따른다. 에이전트가 코드를 생성하더라도, 사람이 설계를 판단하고 결과를 검증하는 구조를 유지한다.

## 1.4 AI 협업의 역할 분담: 무엇을 맡기고 무엇을 검증하는가

AI 협업을 잘하는 방법은 "잘 시키기" 이전에 "무엇을 맡기고 무엇을 남길지"를 정하는 일이다. AI가 잘하는 영역은 초안 생성, 대안 탐색, 리팩토링 후보 제안처럼 탐색적 작업이다. 반대로 사람이 책임져야 하는 영역은 요구사항 확정, 위험 판단, 검증 설계, 운영 책임처럼 결과의 책임이 따르는 작업이다.

이 구분을 팀의 규칙으로 만들기 위해서는 작업 산출물을 '문장'이 아니라 '파일'로 남기는 습관이 필요하다. 어떤 코드를 선택했는지, 어떤 입력에서 확인했는지, 어떤 실패를 고려했는지를 남기면 다음 단계(리뷰, 수정, 재현)가 쉬워진다.

**표 1.2** 사람과 AI의 역할 분담(권장)

| 구분 | 사람이 담당 | AI가 보조(권장) |
|:---|:---|:---|
| 목표/범위 | 문제 정의, 완료 조건 확정 | 요구사항 질문 목록 제안 |
| 설계 | 제약/리스크 판단, 의사결정 | 대안 아키텍처 초안 제안 |
| 구현 | 핵심 로직 검증, 통합/운영 고려 | 코드 초안 생성, 리팩토링 후보 |
| 검증 | 테스트/재현 절차 정의, 결과 해석 | 테스트 코드 초안, 에러 원인 후보 |
| 안전 | 비밀정보/권한/파괴적 동작 통제 | 체크리스트 제안, 누락 탐지 |

## 1.5 실패 패턴과 대응: 환각/과신/컨텍스트/보안

AI 협업의 실패는 대체로 "답이 틀렸다"가 아니라 "확인 없이 믿었다"에서 시작한다.

**환각(hallucination)**은 존재하지 않는 API, 옵션, 참고자료를 사실처럼 말하는 형태로 나타난다. 존재하지 않는 라이브러리 함수를 자신 있게 제안하거나, 가짜 DOI가 포함된 논문을 인용하는 경우가 대표적이다.

**과신(overconfidence)**은 "된다"는 주장과 실제 실행의 괴리가 커지는 형태로 나타난다. AI가 "이 코드는 모든 엣지 케이스를 처리합니다"라고 주장하더라도, 실제 실행 없이는 확인할 수 없다.

**컨텍스트 누락**은 파일 구조, 버전, 운영 환경 차이를 반영하지 못해 발생한다. 프로젝트에 이미 존재하는 유틸리티 함수를 무시하고 새로 작성하거나, 프로덕션 환경의 제약을 고려하지 않는 코드를 제안하는 경우가 이에 해당한다.

**보안 문제**는 비밀정보 노출이나 파괴적 동작(덮어쓰기, 삭제 등)을 무심코 실행하는 형태로 나타난다. API 키를 코드에 하드코딩하거나, `rm -rf` 같은 위험한 명령을 검증 없이 제안하는 경우다.

이 장에서는 이를 막기 위한 최소 규칙을 네 가지로 정리한다. 첫째, 목표와 산출물을 합의한다. 둘째, 입력과 제약을 확인한다. 셋째, 실행 가능한 재현 절차를 남긴다. 넷째, 안전과 보안을 기본값으로 둔다. 다음 절의 미니 실습은 이 규칙을 "체크리스트 파일"로 만들어 프로젝트 산출물로 남기는 데 목적이 있다.

## 1.6 미니 실습: 체크리스트 기반으로 작은 자동화를 만들기

이 절의 목표는 자동화 자체가 아니라, 자동화를 만들 때 반드시 따라야 하는 검증 습관을 파일로 남기는 것이다. 실습 스크립트는 실행하면 체크리스트 문서를 생성해 `practice/chapter1/data/output/ch01_checklist.md`에 저장한다.

_전체 코드는 practice/chapter1/code/1-4-checklist-generator.py 참고_

본문에서는 출력 폴더를 만들고 파일을 쓰는 핵심만 확인한다.

```python
output_dir = chapter_dir / "data" / "output"
output_dir.mkdir(parents=True, exist_ok=True)
output_path = output_dir / "ch01_checklist.md"
output_path.write_text(CHECKLIST_TEXT, encoding="utf-8")
```

실습을 수행한 뒤에는 생성된 체크리스트를 그대로 사용하는 것만으로 끝내지 말고, 이후 장의 실습에서도 "작업 범위/입력·제약/검증/안전" 항목을 실제로 채워 넣는 방식으로 습관화하는 것이 중요하다.

## 핵심정리

- AI 협업 개발의 핵심은 "빠른 생성"이 아니라 "빠른 검증"에 있다.
- 에이전틱 코딩 도구는 IDE 기반(Copilot, Cursor, Windsurf)과 터미널 기반(Claude Code, Gemini CLI, Codex CLI)으로 나뉘며, 프로젝트 성격에 따라 선택한다.
- 바이브 코딩은 프로토타입에는 유효하나, 프로덕션 코드에는 보안·유지보수·생산성 측면에서 한계가 명확하다.
- 사람의 책임(요구사항, 위험 판단, 검증 설계)은 줄어들지 않고, 오히려 더 중요해진다.
- 산출물은 텍스트 주장보다 파일로 남길 때 재현과 검토가 가능해진다.

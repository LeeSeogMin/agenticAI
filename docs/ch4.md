# 제4장. 첫 MCP 서버 만들기: 외부 API 래핑

## 학습목표

- 외부 API를 MCP 서버로 래핑할 때 고려할 사항(인증/실패/로깅/테스트)을 설명한다.
- `.env` 파일로 인증키를 분리하고, 코드에 하드코딩하지 않는 패턴을 적용한다.
- 타임아웃, 재시도, 에러 응답 형식을 포함한 실패 처리 전략을 구현한다.
- 로깅을 통해 요청/응답을 추적 가능하게 만들고, 테스트 가능한 구조로 설계한다.

## 선수지식

3장에서 다룬 MCP 도구(tool) 설계 원칙과 명세 작성 방법을 이해하고 있어야 한다. Python 기본 문법과 비동기 프로그래밍(async/await)에 대한 기초 지식이 있으면 따라오기 수월하다.

## 4.1 외부 API 래핑의 목표와 고려사항

MCP 서버로 외부 API를 래핑하는 것은 단순히 "호출을 감싸는 것"이 아니라, 에이전트가 안정적으로 사용할 수 있는 "계약(contract)"을 만드는 것이다. 3장에서 다룬 도구 명세가 "무엇을 할 것인가"였다면, 이 장에서는 "어떻게 안전하게 할 것인가"를 다룬다.

외부 API는 우리가 통제할 수 없는 영역이다. 네트워크는 언제든 끊길 수 있고, 서버는 과부하 상태일 수 있으며, 인증 토큰은 만료될 수 있다. 이런 불확실성을 MCP 서버 내부에서 처리하지 않으면, 에이전트는 예측할 수 없는 실패를 경험하게 된다.

**표 4.1** 외부 API 호출 시 문제 유형과 대응 전략

| 문제 유형 | 원인 | 대응 전략 |
|----------|------|----------|
| 네트워크 오류 | 연결 실패, DNS 오류 | 재시도(지수 백오프) |
| 인증 실패 | 잘못된 키, 만료된 키 | 명확한 에러 메시지, 키 검증 |
| 레이트 리밋 | 요청 과다 | 대기 후 재시도, 캐싱 |
| 응답 지연 | 서버 과부하 | 타임아웃 설정, 취소 |
| 잘못된 응답 | API 스키마 변경 | 응답 검증, 폴백 처리 |

이 장에서는 OpenWeatherMap API를 예제로 사용한다. 이 API는 무료 티어를 제공하고, API 키 발급이 간단하며, 요청/응답 구조가 명확하다. 날씨 정보는 실시간 데이터이므로 실제 실행 결과를 확인하기에 적합하다.

## 4.2 인증키 관리: .env와 환경 변수

API 키를 코드에 하드코딩하면 여러 보안 문제가 발생한다. 가장 흔한 실수는 API 키가 포함된 코드를 Git 저장소에 커밋하는 것이다. 공개 저장소라면 즉시 노출되고, 비공개 저장소라도 팀원 변경이나 저장소 유출 시 위험해진다. 로그 파일에 요청 URL이 기록될 때 쿼리 파라미터로 포함된 키가 함께 노출되는 경우도 있다.

이 문제를 해결하는 표준적인 방법은 환경 변수를 사용하는 것이다. Python에서는 `python-dotenv` 패키지를 사용해 `.env` 파일에서 환경 변수를 로드할 수 있다.

```python
from dotenv import load_dotenv
import os

load_dotenv()
api_key = os.getenv("OPENWEATHERMAP_API_KEY")
```

_전체 코드는 practice/chapter4/code/4-6-weather-mcp-server.py 참고_

`.env` 파일은 반드시 `.gitignore`에 추가해야 한다. 대신 `.env.example` 파일을 만들어 필요한 환경 변수 목록을 공유하는 것이 좋은 관례이다. 실습 디렉토리에는 다음과 같은 예제 파일이 포함되어 있다.

```
# .env.example
OPENWEATHERMAP_API_KEY=your_api_key_here
```

환경 변수에는 우선순위가 있다. 시스템에 이미 설정된 환경 변수가 있으면 `.env` 파일의 값보다 우선한다. 이 동작은 `python-dotenv`의 기본 설정이며, 프로덕션 환경에서 `.env` 파일 없이 시스템 환경 변수만으로 동작하게 할 때 유용하다.

## 4.3 실패 처리: 타임아웃, 재시도, 에러 응답

외부 API는 언제든 실패할 수 있으므로, "성공 경로"뿐 아니라 "실패 경로"도 설계해야 한다. 실패 처리의 핵심은 세 가지다. 첫째, 무한 대기를 방지하는 타임아웃 설정. 둘째, 일시적 오류에 대응하는 재시도 전략. 셋째, 클라이언트가 이해할 수 있는 에러 응답 형식이다.

### 타임아웃 설정

네트워크 요청에는 반드시 타임아웃을 설정해야 한다. 타임아웃 없이 요청을 보내면, 서버가 응답하지 않을 때 무한히 대기하게 된다. `httpx` 라이브러리에서는 `timeout` 파라미터로 초 단위 타임아웃을 지정할 수 있다. 일반적으로 외부 API 호출에는 10~30초 사이의 타임아웃이 적절하다.

### 재시도 전략

모든 오류가 재시도 대상은 아니다. 인증 실패(401)나 잘못된 요청(400)처럼 재시도해도 결과가 달라지지 않는 오류는 즉시 실패를 반환해야 한다. 반면 서버 오류(5xx)나 네트워크 오류는 재시도하면 성공할 가능성이 있다.

재시도 시에는 지수 백오프(exponential backoff)를 적용하는 것이 좋다. 첫 번째 재시도는 1초 후, 두 번째는 2초 후, 세 번째는 4초 후처럼 대기 시간을 지수적으로 늘리는 방식이다. 이렇게 하면 서버가 과부하 상태일 때 부하를 가중시키지 않으면서도 복구 후 빠르게 요청을 처리할 수 있다.

```python
for attempt in range(max_retries):
    response = requests.get(url, timeout=10)
    if response.ok:
        break
    time.sleep(2 ** attempt)  # 지수 백오프: 1, 2, 4초
```

_전체 코드는 practice/chapter4/code/4-6-weather-mcp-server.py 참고_

### MCP 에러 응답 형식

MCP 도구가 실패할 때는 클라이언트가 이해할 수 있는 형식으로 에러를 반환해야 한다. 단순히 예외를 발생시키는 것보다, 구조화된 JSON 응답을 반환하는 것이 에이전트가 실패를 처리하기 쉽다.

```json
{
  "success": false,
  "error": "API 요청 실패: 인증키가 유효하지 않습니다"
}
```

## 4.4 로깅: 요청과 응답을 추적 가능하게

운영 환경에서 문제가 발생했을 때 원인을 파악하려면 로깅이 필수다. 언제, 어떤 요청을 보냈고, 응답이 무엇이었는지 기록하지 않으면, 문제 재현조차 어려워진다.

Python의 `logging` 모듈을 사용하면 로그 레벨별로 메시지를 분류하고, 파일이나 콘솔에 출력할 수 있다. 로그 레벨은 DEBUG(상세 디버깅), INFO(일반 정보), WARNING(경고), ERROR(오류) 순으로 심각도가 높아진다.

```python
import logging

logging.basicConfig(
    filename="logs/mcp_server.log",
    level=logging.INFO,
    format="%(asctime)s - %(levelname)s - %(message)s"
)
```

_전체 코드는 practice/chapter4/code/4-6-weather-mcp-server.py 참고_

로깅 시 주의할 점이 있다. API 키, 사용자 개인정보, 인증 토큰 같은 민감 정보는 로그에 그대로 기록하면 안 된다. 실습 코드에서는 API 키를 마스킹하는 함수를 구현했다. 예를 들어 `abcd1234efgh5678` 같은 키는 `abcd...5678`으로 변환하여 기록한다.

MCP 서버에서 특히 주의할 점이 있다. STDIO 기반 MCP 서버에서는 `print()` 함수를 사용하면 안 된다. MCP 프로토콜은 표준 출력(stdout)을 통해 통신하므로, `print()`로 출력한 내용이 프로토콜 메시지와 섞여 통신이 깨질 수 있다. 대신 `logging` 모듈을 사용하면 표준 에러(stderr)로 출력되어 안전하다.

### 실패 사례: 로그에 API 키가 노출된 경우

한 개발팀에서 날씨 MCP 서버를 운영하다가 보안 문제가 발생했다. 디버깅을 위해 요청 URL 전체를 로그에 기록했는데, 쿼리 파라미터에 포함된 API 키가 그대로 노출된 것이다. 로그 파일이 모니터링 시스템으로 전송되면서 더 넓은 범위로 유출되었다.

이 문제의 대응책은 두 가지다. 첫째, 로그 기록 전에 민감 정보를 마스킹한다. 둘째, 요청 URL 대신 엔드포인트와 파라미터를 분리하여 기록하고, API 키는 기록에서 제외한다. 실습 코드의 `_mask_api_key()` 함수가 이 패턴을 보여준다.

## 4.5 테스트 가능한 구조: 의존성 분리

외부 API에 의존하는 코드는 테스트하기 어렵다. API 서버가 항상 동작한다고 보장할 수 없고, 테스트할 때마다 실제 요청을 보내면 비용이 발생하거나 레이트 리밋에 걸릴 수 있다. 이 문제를 해결하려면 API 호출 로직을 분리하여 모킹(mocking) 가능한 구조로 만들어야 한다.

핵심은 의존성 주입이다. API 클라이언트를 함수 내부에서 직접 생성하는 대신, 외부에서 주입받도록 설계한다. 그러면 테스트 시에는 실제 클라이언트 대신 가짜(mock) 클라이언트를 주입할 수 있다.

실습 코드에서는 `WeatherAPIClient` 클래스로 API 호출 로직을 캡슐화했다. 이 클래스의 메서드들은 실제 API 호출 없이도 단위 테스트가 가능하다.

## 4.6 실습: OpenWeatherMap API를 MCP 서버로 래핑

이 절에서는 앞서 설명한 원칙들을 적용하여 실제 MCP 서버를 구현한다. OpenWeatherMap의 Current Weather API를 래핑하여, 위도와 경도를 입력받아 현재 날씨 정보를 반환하는 도구를 만든다.

### 환경 설정

먼저 실습 환경을 준비한다. Python 가상환경을 생성하고 의존성을 설치한다.

```bash
cd practice/chapter4
python3 -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r code/requirements.txt
```

`.env` 파일을 생성하고 OpenWeatherMap API 키를 설정한다. API 키는 https://openweathermap.org/api 에서 무료로 발급받을 수 있다.

```
OPENWEATHERMAP_API_KEY=your_api_key_here
```

### MCP 서버 구조

실습 코드는 FastMCP를 사용하여 MCP 서버를 구현한다. FastMCP는 Python 타입 힌트와 docstring을 분석하여 자동으로 도구 스키마를 생성한다.

```python
mcp = FastMCP("weather-server")

@mcp.tool()
async def get_current_weather(latitude: float, longitude: float) -> str:
    """지정된 위치의 현재 날씨를 조회합니다."""
```

_전체 코드는 practice/chapter4/code/4-6-weather-mcp-server.py 참고_

### 핵심 구현 요소

1. **인증키 관리**: `.env` 파일에서 API 키를 로드하고, 로그에는 마스킹된 형태로 기록한다.

2. **입력 검증**: 위도는 -90~90, 경도는 -180~180 범위를 벗어나면 에러를 반환한다.

3. **실패 처리**: 타임아웃 10초, 최대 3회 재시도, 지수 백오프 적용.

4. **로깅**: 모든 요청과 응답을 `logs/mcp_server.log`에 기록한다.

5. **응답 구조화**: 원시 API 응답에서 필요한 정보만 추출하여 일관된 형태로 반환한다.

### 테스트 실행

테스트 클라이언트를 실행하여 핵심 로직을 검증할 수 있다.

```bash
python3 code/4-6-test-client.py
```

테스트가 성공하면 `data/output/ch04_test_results.json`에 결과가 저장된다.

### 의사결정 포인트

MCP 서버를 구현할 때 몇 가지 선택이 필요하다.

**HTTP 클라이언트 선택**: `requests`와 `httpx` 중 어떤 것을 사용할까? MCP 서버는 비동기로 동작하므로, 비동기를 기본 지원하는 `httpx`가 더 적합하다. `requests`도 사용할 수 있지만, 별도의 스레드 풀이 필요해 복잡해진다.

**재시도 횟수와 타임아웃**: 너무 짧으면 일시적 오류에 취약하고, 너무 길면 사용자 경험이 나빠진다. 일반적인 웹 API에는 타임아웃 10초, 재시도 3회가 적절한 출발점이다. 실제 운영에서는 API의 특성에 맞게 조정해야 한다.

**에러 메시지 상세도**: 너무 상세하면 내부 구현이 노출될 수 있고, 너무 간략하면 디버깅이 어렵다. 클라이언트에게는 "인증 실패" 같은 개요만 전달하고, 상세 정보는 서버 로그에 기록하는 것이 좋다.

## 핵심정리

- 외부 API를 MCP 서버로 래핑할 때는 인증, 실패 처리, 로깅, 테스트 가능한 구조를 함께 고려해야 한다.
- API 키는 `.env` 파일로 분리하고, `.gitignore`에 추가하여 저장소에 커밋되지 않도록 한다.
- 타임아웃과 재시도(지수 백오프)로 일시적 오류에 대응하고, 재시도 불가능한 오류는 즉시 반환한다.
- 로깅은 문제 추적에 필수이지만, 민감 정보는 마스킹해야 한다.
- 의존성 분리를 통해 외부 API 없이도 핵심 로직을 테스트할 수 있다.

## 10.1 LLM 환각의 이해

환각이란 모델이 학습 데이터에 근거하지 않거나 주어진 컨텍스트와 일치하지 않는 정보를 생성하는 현상을 말한다. 2025년 OpenAI의 연구에 따르면, 환각은 단순한 버그가 아니라 next-token 예측이라는 학습 목표에서 비롯된 구조적 특성이다. 모델은 정확한 답변보다 "자신감 있어 보이는" 답변에 더 높은 보상을 받도록 학습되기 때문이다.

### 환각의 유형

환각은 크게 세 가지 유형으로 분류된다.

첫째, 사실 오류(Factual Hallucination)는 실제와 다른 정보를 생성하는 것이다. 예를 들어, "파이썬은 1995년에 출시되었다"라고 답변하는 경우가 이에 해당한다(실제 출시 연도는 1991년이다).

둘째, 논리 오류(Logical Hallucination)는 주어진 전제로부터 도출될 수 없는 결론을 제시하는 것이다. RAG 시스템에서 검색된 문서에 "A는 B보다 빠르다"라고 되어 있는데, "따라서 B가 더 효율적이다"라고 결론짓는 경우가 이에 해당한다.

셋째, 출처 조작(Source Fabrication)은 존재하지 않는 참고문헌을 인용하거나, 실제 논문의 내용을 왜곡하여 인용하는 것이다. 학술 정보 시스템에서 특히 위험한 유형이다.

### 환각 발생 원인

환각이 발생하는 주요 원인은 다음과 같다. 학습 데이터의 편향으로 인해 모델이 특정 패턴을 과도하게 학습할 수 있다. 또한 주어진 컨텍스트가 불충분할 때 모델은 내부 지식으로 공백을 채우려 하며, 이 과정에서 오류가 발생한다. 토큰 확률 분포에 기반한 생성 방식 자체가 때로는 사실보다 유창함을 우선시하게 만든다.

Nature에 발표된 Semantic Entropy 연구는 모델이 동일한 질문에 대해 의미적으로 다른 답변을 생성할 때 높은 불확실성을 보인다는 점을 활용하여 환각을 탐지하는 방법을 제안했다. 이 접근은 단어 수준이 아닌 의미 수준에서 불확실성을 측정한다는 점에서 획기적이다.

## 10.2 교차검증 전략

교차검증(Cross-Verification)은 동일한 질문에 대해 여러 경로로 답변을 생성하고, 그 일관성을 확인하는 방법이다. 일관성이 낮다면 모델이 확신 없이 답변하고 있을 가능성이 높다.

### Self-Consistency

Self-Consistency는 Wang et al.(2023)이 제안한 방법으로, 동일한 모델에게 같은 질문을 여러 번 하되 높은 temperature(예: 0.8~1.2)를 사용하여 다양한 답변을 유도한다. 이후 다수결(majority voting)로 최종 답변을 선택한다.

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.8)
samples = [llm.invoke(prompt) for _ in range(3)]
```

이 방법의 핵심 아이디어는 모델이 확신하는 답변은 여러 번 물어도 일관되게 나타난다는 것이다. 반면 불확실한 영역에서는 샘플링마다 다른 답변이 나온다.

### Chain-of-Verification (CoVe)

Meta AI 연구팀이 개발한 Chain-of-Verification은 4단계 프로세스를 따른다. 먼저 초기 답변을 생성하고, 그 답변을 검증하기 위한 질문을 계획한다. 각 검증 질문에 독립적으로 답변한 후, 검증 결과를 바탕으로 최종 답변을 수정한다.

실험 결과, CoVe를 적용한 LLaMA-65B 모델은 FactScore가 63.7에서 71.4로 향상되었다. 기술 문서와 코드 리뷰에서는 40%의 정확도 향상이 보고되었다.

### 전략 비교

**표 10.1** 교차검증 전략 비교

| 전략 | 방법 | API 호출 | 신뢰도 향상 |
|:---|:---|:---:|:---:|
| Self-Consistency | 동일 모델 N회 샘플링 | N회 | 중간 |
| Multi-Model | 서로 다른 모델로 검증 | 모델 수만큼 | 높음 |
| Chain-of-Verification | 검증 전용 프롬프트 추가 | 3~4회 | 중간~높음 |

## 10.3 출처 일치 검증

RAG 시스템에서 특히 중요한 것은 생성된 답변이 실제로 검색된 문서에 근거하는지 확인하는 것이다. 이를 Grounding 또는 Faithfulness 검증이라고 한다.

### Grounded Generation

Grounded Generation은 모델이 답변을 생성할 때 반드시 주어진 컨텍스트에서 근거를 찾도록 강제하는 접근이다. 프롬프트에 "컨텍스트에 없는 정보는 추측하지 마세요"와 같은 지시를 포함하는 것이 기본이다.

그러나 이러한 지시만으로는 충분하지 않다. 모델은 여전히 내부 지식을 활용하여 컨텍스트를 "보완"하려 할 수 있다. 따라서 생성 후 검증이 필요하다.

### NLI 기반 검증

자연어 추론(Natural Language Inference, NLI) 모델을 활용하면 "답변이 컨텍스트로부터 논리적으로 도출되는가"를 자동으로 평가할 수 있다. SummaC는 문서-요약 쌍에서 문장 수준 NLI 점수를 집계하는 방법을 제안했으며, 이는 RAG 답변 검증에도 적용할 수 있다.

### Faithfulness vs Correctness

주의할 점은 Correctness와 Faithfulness가 다르다는 것이다. 답변이 사실적으로 맞더라도(Correct), 그 정보가 주어진 컨텍스트에서 온 것이 아니라면 Unfaithful한 것이다. RAG 시스템의 신뢰성은 모델이 실제로 검색된 문서를 사용했는지에 달려 있다.

연구에 따르면, 일부 RAG 최적화 모델은 내부 지식으로 답변을 생성한 후 토큰 유사성을 기반으로 사후적으로 인용을 붙이는 경향이 있다. 이러한 "post-hoc citation" 문제를 탐지하려면 생성 과정 자체를 검증해야 한다.

## 10.4 품질 게이트 설계

품질 게이트(Quality Gate)는 생성 파이프라인의 특정 지점에 배치되어 출력이 기준을 충족하는지 검사하는 검문소이다. 게이트를 통과하지 못하면 재시도하거나 에스컬레이션한다.

### 게이트 배치 위치

효과적인 품질 게이트는 다음 세 지점에 배치한다.

**그림 10.1** 품질 게이트가 포함된 생성 파이프라인

```
[입력] → [생성] → [Gate 1: 형식] → [Gate 2: 출처] → [Gate 3: 일관성] → [출력]
              ↓ 실패        ↓ 실패          ↓ 실패
           [재시도]      [재시도]       [사람 승인]
```

Gate 1은 형식 검사로, JSON 파싱 가능 여부나 필수 필드 존재 여부를 확인한다. Gate 2는 출처 일치 검증으로, 답변이 컨텍스트에 근거하는지 평가한다. Gate 3은 일관성 검사로, Self-Consistency를 통해 답변의 안정성을 확인한다.

### 게이트 통과 기준

각 게이트는 수치적 임계값을 기준으로 통과 여부를 결정한다. 본 장의 실습에서는 일관성 점수 0.25 이상, 출처 점수 0.7 이상을 통과 기준으로 설정했다. 이 값은 도메인과 요구 사항에 따라 조정해야 한다.

임계값이 너무 높으면 정상적인 답변도 거부하는 False Positive가 증가하고, 너무 낮으면 문제 있는 답변이 통과하는 False Negative가 증가한다. 실제 운영에서는 로그를 분석하여 최적의 균형점을 찾아야 한다.

### 실패 시 처리 전략

게이트 실패 시 가장 단순한 전략은 재시도이다. 동일한 요청을 다시 처리하면 LLM의 비결정적 특성으로 인해 다른 결과가 나올 수 있다. 그러나 무한 재시도는 비효율적이므로 최대 재시도 횟수(예: 3회)를 설정한다.

최대 재시도를 초과하면 사람에게 에스컬레이션하거나, 답변 불가 메시지를 반환한다. 어떤 전략을 선택할지는 시스템의 요구 사항에 따라 달라진다.

## 10.5 검증 파이프라인 구현

LangGraph를 활용하면 검증 로직을 워크플로우로 구조화할 수 있다. 각 검증 단계를 노드로 정의하고, 조건부 엣지로 재시도와 에스컬레이션 흐름을 구현한다.

### 상태 정의

검증 워크플로우의 상태는 질문, 컨텍스트, 답변뿐 아니라 검증 점수, 재시도 횟수, 사람 승인 여부 등을 포함한다.

```python
class VerificationState(TypedDict):
    query: str
    context: str
    answer: str
    consistency_score: float
    grounding_score: float
    retry_count: int
    needs_human_approval: bool
```

### 검증 노드 설계

각 검증 단계는 독립된 노드로 구현한다. 일관성 검사 노드는 Self-Consistency를 수행하고, 출처 검증 노드는 LLM을 활용하여 Grounding 점수를 계산한다. 품질 게이트 노드는 모든 검증 결과를 종합하여 다음 단계를 결정한다.

```python
graph.add_node("consistency", check_consistency)
graph.add_node("grounding", check_grounding)
graph.add_node("gate", quality_gate)
```

### 조건부 라우팅

품질 게이트 노드는 세 가지 경로 중 하나를 선택한다. 모든 검증을 통과하면 종료하고, 재시도 횟수가 남아 있으면 생성 노드로 돌아가며, 재시도를 초과하면 사람 승인 노드로 이동한다.

```python
graph.add_conditional_edges(
    "gate",
    route_after_gate,
    {"generate": "generate", "human": "human", "end": END}
)
```

## 10.6 OWASP AI Agent Security

에이전트 시스템은 전통적인 LLM 애플리케이션과 다른 고유한 보안 위협에 노출된다. OWASP(Open Worldwide Application Security Project)는 이러한 위협을 체계적으로 분류한다.

### 10.6.1 에이전트 고유 보안 위협

에이전트는 도구를 호출하고 외부 시스템에 영향을 미치므로, 단순한 텍스트 생성과는 차원이 다른 위험이 존재한다. 주요 위협은 다음과 같다.

**프롬프트 인젝션(Prompt Injection)**: 악의적 입력이 에이전트의 시스템 프롬프트를 우회하거나 덮어쓰는 공격이다. 직접 인젝션(사용자 입력에 지시문 삽입)과 간접 인젝션(검색된 문서나 도구 응답에 숨겨진 지시문)으로 나뉜다. 에이전트가 도구를 호출하는 능력을 갖추었기 때문에, 인젝션 성공 시 피해가 텍스트 생성보다 훨씬 크다.

**도구 오용(Tool Misuse)**: 에이전트가 의도하지 않은 방식으로 도구를 호출하는 것이다. 파일 시스템 접근 도구가 민감한 경로를 읽거나, API 호출 도구가 인가되지 않은 엔드포인트에 요청을 보내는 경우가 이에 해당한다.

**데이터 유출(Data Exfiltration)**: 에이전트가 접근한 내부 데이터를 외부로 전송하는 것이다. 도구 호출의 파라미터에 민감 정보를 포함시키거나, 외부 API 응답으로 데이터를 누출할 수 있다.

**과도한 자율성(Excessive Agency)**: 에이전트에게 필요 이상의 권한이 부여되어, 의도하지 않은 범위의 작업을 수행하는 것이다. 최소 권한 원칙의 위반에서 비롯된다.

## 10.7 다층 가드레일 아키텍처

가드레일(guardrail)은 에이전트의 동작을 안전한 범위 안에 유지하는 보호 장치다. 단일 계층이 아니라, 입력부터 출력까지 여러 단계에 걸친 다층 방어가 효과적이다.

### 10.7.1 입력 가드레일

사용자 입력이 에이전트에 전달되기 전에 검사한다. PII(개인식별정보) 자동 제거, 탈옥(jailbreak) 시도 탐지, 입력 길이 제한이 이 계층에 해당한다. 프롬프트 인젝션 탐지 모델을 배치하여 악의적 입력을 사전에 차단할 수 있다.

### 10.7.2 실행 샌드박싱

에이전트가 코드를 실행하거나 외부 시스템에 접근할 때, 격리된 환경에서 수행한다. gVisor나 Firecracker microVM으로 실행 환경을 격리하고, 네트워크 이그레스 허용목록(allowlist)으로 외부 접근 범위를 제한한다. 에이전트가 접근할 수 있는 파일 경로, API 엔드포인트, 네트워크 범위를 명시적으로 정의한다.

### 10.7.3 출력 가드레일

에이전트의 출력이 사용자에게 전달되기 전에 정책 규칙을 검사한다. 유해 콘텐츠 필터링, 민감 정보 마스킹, 조직 정책 준수 여부를 확인한다. 이 계층은 환각 방지보다는 정책 위반 방지에 초점을 맞춘다.

### 10.7.4 ID/접근 제어

에이전트별로 최소 권한 IAM(Identity and Access Management)을 적용한다. 연구자 에이전트에게는 읽기 권한만, 실행 에이전트에게는 특정 도구 호출 권한만 부여한다. 도구 호출 시마다 호출 주체의 권한을 확인하는 런타임 검증이 필요하다.

### 10.7.5 프레임워크 내장 가드레일

주요 프레임워크는 가드레일을 내장 기능으로 제공한다. OpenAI Agents SDK는 `guardrails` 프리미티브로 입력/출력 가드레일을 에이전트 정의에 포함할 수 있다. Google ADK는 safety 설정으로 유해 콘텐츠 카테고리별 차단 수준을 조정할 수 있다. 프레임워크 내장 기능을 활용하되, 조직 정책에 맞는 커스텀 가드레일을 추가하는 것이 바람직하다.

## 10.8 에이전트 평가 프레임워크

에이전트 시스템은 단순한 텍스트 생성과 달리, 멀티스텝 계획, 도구 사용, 정책 준수 등 복합적인 능력을 평가해야 한다. 전용 벤치마크와 평가 방법론이 발전하고 있다.

**표 10.2** 주요 에이전트 평가 벤치마크

| 벤치마크 | 평가 대상 | 핵심 특징 |
|---------|----------|----------|
| GAIA | 멀티스텝 계획 | 웹 검색·파일 처리를 결합한 실세계 질문 |
| WebArena | 웹 상호작용 | 실제 웹사이트에서 작업 수행 |
| τ-bench | 정책 준수 | 도메인 정책을 따르며 사용자와 대화 |
| Terminal-Bench | CLI 워크플로우 | 터미널 환경에서 작업 수행 |

### 10.8.1 CLEAR 프레임워크

CLEAR는 에이전트를 5가지 차원에서 종합 평가하는 프레임워크다. **C**ost(비용), **L**atency(지연시간), **E**fficacy(효과), **A**ssurance(보증), **R**eliability(신뢰성)를 측정한다. 정확도만으로는 프로덕션 적합성을 판단할 수 없으므로, 비용·지연시간·신뢰성을 함께 평가해야 한다는 관점이다.

### 10.8.2 LLM-as-Judge

LLM을 평가자로 활용하는 접근은 스케일이 필요한 평가에서 빠르게 확산되고 있다. LLM이 에이전트의 출력을 채점하고, 사람 평가자와의 일치도가 약 90%에 달한다는 연구 결과가 있다. 다만 위치 편향(첫 번째 응답을 선호), 장문 편향(긴 답변을 선호), 자기 편향(같은 모델의 출력을 선호) 등의 한계가 있으므로, 사람 평가와 병행하는 것이 권장된다.

## 10.9 실습: 검증 워크플로우 구현

이 실습에서는 교차검증과 출처 일치 검증이 포함된 완전한 검증 파이프라인을 구현한다. 검증 실패 시 자동 재시도하고, 최대 재시도를 초과하면 사람 승인을 요청하는 흐름을 구성한다.

_전체 코드는 practice/chapter10/code/10-6-verification.py 참고_

### 실습 환경 설정

```bash
cd practice/chapter10
python3 -m venv venv
source venv/bin/activate  # macOS/Linux
pip install -r code/requirements.txt
```

### 핵심 구현

검증 워크플로우는 다섯 개의 노드로 구성된다.

```python
graph.add_node("generate", generate_answer)
graph.add_node("consistency", check_consistency)
graph.add_node("grounding", check_grounding)
graph.add_node("gate", quality_gate)
graph.add_node("human", request_human_approval)
```

generate 노드는 RAG 기반으로 답변을 생성하고, consistency 노드는 3회 샘플링으로 일관성을 검사한다. grounding 노드는 LLM을 활용하여 답변이 컨텍스트에 근거하는지 평가한다. gate 노드는 두 검증 결과를 종합하여 통과, 재시도, 에스컬레이션 중 하나를 결정한다.

### 실행 결과

3개의 테스트 질문에 대해 검증 워크플로우를 실행한 결과는 다음과 같다.

**표 10.3** 검증 워크플로우 실행 결과

| 질문 | 일관성 점수 | 출처 점수 | 재시도 | 통과 여부 |
|:---|:---:|:---:|:---:|:---:|
| 이벤트 루프의 역할 | 0.293 | 1.000 | 0회 | 통과 |
| 코루틴과 태스크 차이 | 0.295 | 1.000 | 0회 | 통과 |

모든 질문이 첫 시도에서 통과했다. 평균 일관성 점수는 0.294, 평균 출처 점수는 1.000이었다. 출처 점수가 1.0인 것은 모든 답변이 주어진 컨텍스트에 완전히 근거했음을 의미한다.

## 10.10 실패 사례 수집과 개선

검증 파이프라인은 문제를 탐지할 뿐 아니라, 탐지된 문제로부터 학습하는 피드백 루프를 구성해야 한다. 실패 사례를 체계적으로 수집하고 분석하면 시스템을 지속적으로 개선할 수 있다.

### 실패 사례 분류

실패 사례는 유형과 심각도에 따라 분류한다. 유형은 일관성 실패, 출처 실패, 형식 오류 등으로 구분하고, 심각도는 사소한 문제, 주요 문제, 치명적 문제로 나눈다. 이러한 분류는 우선순위 결정과 자원 배분에 도움이 된다.

### 패턴 분석

축적된 실패 사례에서 패턴을 분석한다. 특정 유형의 질문에서 일관성이 낮다면 해당 도메인의 컨텍스트가 부족한 것일 수 있다. 특정 소스에서 출처 점수가 낮다면 해당 문서의 품질이나 청킹 전략을 검토해야 한다.

### 피드백 루프

분석 결과는 세 가지 방향으로 피드백된다. 프롬프트 개선은 가장 빠르게 적용할 수 있는 방법이다. 검색 전략 수정은 청킹 크기, 검색 알고리즘, 리랭킹 모델 등을 조정한다. 검증 기준 조정은 임계값을 도메인에 맞게 튜닝하는 것이다.

## 10.11 실패 사례와 교훈

검증 파이프라인을 운영하면서 발생할 수 있는 문제와 대응 방법을 정리한다.

### 과도한 검증으로 인한 지연

모든 답변에 3회 샘플링과 출처 검증을 수행하면 응답 시간이 크게 증가한다. 본 실습에서도 단일 질문에 11~17초가 소요되었다. 실시간 응답이 중요한 시스템에서는 검증 수준과 응답 속도 사이의 균형을 찾아야 한다.

한 가지 접근은 위험도에 따라 검증 수준을 차등화하는 것이다. 일반적인 질문은 가벼운 검증만 수행하고, 중요한 의사결정과 관련된 질문은 엄격한 검증을 적용한다.

### False Positive와 False Negative

키워드 기반 일관성 검사는 의미적으로 유사하지만 다른 단어를 사용한 답변을 실패로 처리할 수 있다. 반대로 임계값이 너무 낮으면 실제 문제 있는 답변이 통과할 수 있다. 실제 운영에서는 의미론적 유사도를 활용하거나, 도메인별로 임계값을 조정해야 한다.

### 사람 승인 병목

모든 실패가 사람에게 에스컬레이션되면 병목이 발생한다. 이를 완화하려면 자동으로 처리할 수 있는 실패 유형을 늘리거나, 답변 불가 메시지를 반환하는 전략을 고려해야 한다.

## 핵심 정리

- 환각은 LLM의 구조적 특성에서 비롯되며, 완전히 제거하기보다 탐지하고 관리하는 것이 현실적이다.
- 에이전트는 프롬프트 인젝션, 도구 오용, 데이터 유출, 과도한 자율성 등 고유한 보안 위협에 노출된다.
- 다층 가드레일(입력·실행·출력·접근 제어)로 에이전트의 동작을 안전한 범위에 유지한다.
- 품질 게이트는 생성 파이프라인의 검문소로, 재시도와 에스컬레이션 흐름을 제어한다.
- GAIA, WebArena, τ-bench, CLEAR 등 전용 벤치마크로 에이전트의 복합적 능력을 평가한다.

---

10장에서는 검증 파이프라인을 통해 LLM 출력의 신뢰성을 높이는 방법을 학습했다. 검증에 실패하고 재시도를 초과한 경우, 최종 결정은 사람에게 위임되었다. 이 장에서는 이러한 사람 개입(Human-in-the-Loop, HITL)을 본격적으로 다룬다.

AI 에이전트가 아무리 정교해도, 되돌릴 수 없는 작업이나 윤리적 판단이 필요한 상황에서는 사람의 최종 승인이 필수이다. 결제 실행, 데이터 삭제, 외부 시스템 변경과 같은 고위험 작업은 AI가 제안하더라도 사람이 확인해야 한다.

---

## 11.1 Human-in-the-loop의 필요성

Human-in-the-loop(HITL)은 AI 시스템의 특정 지점에서 사람이 개입하여 결정을 내리거나 승인하는 패턴을 말한다. 완전 자동화가 목표가 아니라, AI의 효율성과 사람의 판단력을 조합하는 것이 핵심이다.

### HITL이 필요한 상황

첫째, 고위험 작업에서 HITL이 필요하다. 결제 처리, 데이터 삭제, 계약 체결과 같이 되돌리기 어렵거나 비용이 큰 작업은 사람의 승인을 거쳐야 한다.

둘째, 불확실성이 높은 상황에서 필요하다. 10장에서 다룬 것처럼 검증 점수가 낮거나 모델이 확신하지 못하는 경우, 사람에게 판단을 위임하는 것이 적절하다.

셋째, 윤리적 판단이 필요한 상황에서 필요하다. 법적 책임이 따르는 결정, 개인정보 처리, 콘텐츠 검열과 같은 영역에서는 AI가 판단을 내리더라도 사람이 최종 책임을 져야 한다.

넷째, 외부 시스템에 영향을 미치는 작업에서 필요하다. 이메일 발송, API 호출, 서드파티 서비스 연동 등 외부에 변경을 가하는 작업은 신중하게 검토해야 한다.

### AI 자율성과 사람 통제의 균형

모든 작업에 사람 승인을 요구하면 HITL의 의미가 퇴색한다. 반복적이고 저위험인 작업은 자동화하고, 중요한 결정에만 사람이 개입하는 것이 바람직하다. 이를 위해 작업을 위험도에 따라 분류하고 차등 승인 정책을 적용한다.

## 11.2 HITL 패턴 분류

HITL은 개입 방식에 따라 여러 패턴으로 분류된다.

### 주요 패턴

**표 11.1** HITL 패턴별 적용 시나리오

| 패턴 | 설명 | 적용 예시 |
|:---|:---|:---|
| Approval | 실행 전 명시적 승인 | 결제, 이메일 발송, 데이터 삭제 |
| Review | 결과 확인 후 진행 | 문서 요약, 보고서 생성 |
| Edit | 상태나 결과를 직접 수정 | 프롬프트 수정, 데이터 보정 |
| Escalation | 상위 권한자에게 위임 | 고액 결제, 민감 정보 접근 |
| Override | AI 결정을 사람이 덮어씀 | 자동 분류 결과 수정 |

Approval 패턴은 가장 흔한 형태로, AI가 작업을 제안하면 사람이 승인하거나 거부한다. Review 패턴은 AI가 작업을 완료한 후 사람이 결과를 확인하고 필요시 수정한다. Edit 패턴은 사람이 AI의 중간 상태나 입력을 직접 수정하는 것이다.

Escalation 패턴은 일반 승인자가 결정할 수 없는 경우 상위 권한자에게 위임하는 것이다. Override 패턴은 AI가 이미 내린 결정을 사람이 번복하는 것으로, 자동 분류 시스템에서 오분류를 수정할 때 사용한다.

## 11.3 LangGraph의 interrupt와 Command

LangGraph에서 `interrupt` 함수는 HITL 워크플로우 구현의 권장 방식이다. 이 함수를 사용하면 그래프 실행을 특정 노드에서 일시 중지하고, 사람의 입력을 받은 후 재개할 수 있다.

### interrupt 함수의 동작 원리

`interrupt`를 호출하면 다음과 같은 일이 발생한다. 그래프 실행이 일시 중지되고, 현재 스레드가 "interrupted" 상태로 표시된다. interrupt에 전달한 데이터는 persistence layer에 저장된다. 이후 `Command(resume=...)`를 통해 사람의 응답을 전달하면 그래프가 재개된다.

```python
from langgraph.types import interrupt, Command

def approval_node(state):
    decision = interrupt({"action": state["action"], "risk": "high"})
    if decision == "approved":
        return Command(goto="execute")
    return Command(goto="reject")
```

### Command 객체

Command 객체는 그래프의 흐름을 제어하는 도구이다. 상태를 업데이트하면서 다음 노드를 지정할 수 있어, 동적 라우팅이 가능하다. 사전에 정의된 엣지 없이도 런타임에 흐름을 결정할 수 있다.

### 요구사항

interrupt를 사용하려면 Checkpointer가 필요하다. Checkpointer는 각 단계 후 그래프 상태를 저장하여, 중단된 지점에서 정확히 재개할 수 있게 한다. 메모리 기반, 파일 기반, 데이터베이스 기반 등 다양한 Checkpointer 구현이 있다.

## 11.4 승인 요청 UX 설계

사람에게 승인을 요청할 때는 충분한 정보를 제공해야 한다. 정보가 부족하면 올바른 판단을 내릴 수 없고, 정보가 과다하면 핵심을 놓칠 수 있다.

### 승인 요청 메시지 구성

효과적인 승인 요청 메시지는 다음 요소를 포함한다. 작업 내용(무엇을 할 것인가), 대상(어디에 영향을 미치는가), 위험도(왜 승인이 필요한가), 컨텍스트(어떤 상황에서 요청되었는가), 선택지(승인, 거부, 수정 등)가 그것이다.

### 타임아웃과 기본 동작

승인 요청이 무한정 대기할 수는 없다. 타임아웃을 설정하고, 타임아웃 시 기본 동작을 정의해야 한다. 고위험 작업은 기본적으로 거부하고, 저위험 작업은 자동 승인하는 것이 일반적이다.

### 비동기 승인

모든 승인이 실시간일 필요는 없다. 저우선순위 작업은 Slack, 이메일, 대시보드 등 비동기 채널로 라우팅할 수 있다. 이 경우 워크플로우는 승인이 완료될 때까지 대기 상태로 유지된다.

## 11.5 감사 로그와 승인 이력

규정 준수와 책임 추적을 위해 모든 승인 이력을 기록해야 한다. 감사 로그(Audit Log)는 "누가, 언제, 무엇을, 왜" 했는지를 추적한다.

### 필수 로깅 항목

효과적인 감사 로그는 다음 정보를 포함한다.

- **Actor Identity**: 누가 행위를 수행했는가 (사람, 시스템, AI 에이전트)
- **Timestamp**: 언제 발생했는가
- **Action**: 무엇을 했는가 (작업 유형, 대상)
- **Decision**: 결과는 무엇인가 (승인, 거부, 실행, 차단)
- **Reason**: 왜 그런 결정을 내렸는가

AI 시스템에서는 추가로 프롬프트, 검색된 컨텍스트, 도구 호출 파라미터, 안전성 점수 등을 기록하는 것이 권장된다.

### 불변 로그

감사 로그는 변경 불가능(immutable)해야 한다. 한 번 기록된 이벤트는 수정되거나 삭제될 수 없어야 책임 추적이 가능하다. 암호화된 저장소와 해시 체인을 활용하면 무결성을 보장할 수 있다.

### 보존 기간

SOC 2, PCI DSS, HIPAA 등 규정에 따라 3~7년 보존이 요구되기도 한다. 조직의 규정 준수 요구사항에 맞춰 보존 정책을 설정해야 한다.

## 11.6 고위험 작업 분리

모든 작업에 동일한 승인 수준을 적용하는 것은 비효율적이다. 위험도에 따라 작업을 분류하고 차등 승인 정책을 적용해야 한다.

### 위험도 분류 기준

위험도는 다음 기준으로 평가한다. 되돌릴 수 있는가(가역성), 비용이 얼마나 드는가(영향도), 외부에 영향을 미치는가(범위)가 주요 기준이다.

**표 11.2** 위험도별 승인 레벨

| 위험도 | 특성 | 승인 레벨 | 예시 |
|:---:|:---|:---|:---|
| 낮음 | 되돌릴 수 있음, 내부 영향 | 자동 승인 | 로그 조회, 검색, 조회 |
| 중간 | 일부 되돌릴 수 있음 | 단일 승인 | 설정 수정, 파일 변경 |
| 높음 | 되돌릴 수 없음, 외부 영향 | 필수 승인 | 삭제, 결제, 외부 전송 |

### 승인 레벨 정의

자동 승인은 저위험 작업에 적용되며, 사람 개입 없이 진행된다. 단일 승인은 한 명의 승인자가 확인하면 진행된다. 필수 승인(또는 다중 승인)은 반드시 지정된 권한자의 승인이 있어야 하며, 승인 없이는 절대 실행되지 않는다.

### 긴급 우회 절차

예외적인 상황에서는 일시적으로 승인을 우회해야 할 수 있다. 그러나 모든 우회는 철저히 기록되어야 하며, 사후 감사 대상이 된다. 우회 권한은 최소한의 인원에게만 부여해야 한다.

## 11.7 점진적 배포 전략

에이전트 시스템은 전통적인 소프트웨어와 달리 비결정적(non-deterministic)이다. 동일한 입력에 대해 다른 도구를 호출하거나 다른 순서로 실행할 수 있으므로, 코드 배포처럼 한 번에 전환하는 방식은 위험하다. 점진적 배포(Gradual Rollout)는 에이전트의 행동을 단계별로 검증하며 실제 환경에 투입하는 전략이다.

### 5단계 배포 파이프라인

**표 11.3** 에이전트 점진적 배포 5단계

| 단계 | 이름 | 에이전트 행동 | 검증 기준 | 롤백 조건 |
|:---:|:---|:---|:---|:---|
| 1 | Shadow Mode | 분석만 수행, 실행 안 함 | 기존 시스템과 결과 비교 | 일치율 < 90% |
| 2 | 샌드박스 | 격리 환경에서 실행 | 오류율, 비용, 응답 시간 | 오류율 > 5% |
| 3 | 스테이징 | 실제 데이터, 제한된 영향 | 전체 메트릭 기준 충족 | 이상 징후 탐지 |
| 4 | 소규모 프로덕션 | 트래픽 5-10% 라우팅 | A/B 비교, 사용자 피드백 | 성능 저하 감지 |
| 5 | 전체 배포 | 100% 트래픽 처리 | 지속적 모니터링 | 자동 롤백 임계값 |

### Shadow Mode의 핵심 원리

Shadow Mode는 1단계로서 가장 안전한 검증 방법이다. 에이전트가 실제 요청을 받아 분석하고 응답을 생성하지만, 그 결과를 사용자에게 전달하지 않는다. 대신 기존 시스템의 응답과 비교하여 일치율, 품질 점수, 비용을 측정한다.

```python
def shadow_mode_node(state):
    agent_response = agent.invoke(state["query"])
    legacy_response = legacy_system.process(state["query"])
    comparison = compare_responses(agent_response, legacy_response)
    log_shadow_result(comparison)  # 기록만, 실행 안 함
    return {"shadow_result": comparison}
```

Shadow Mode에서 충분한 데이터가 축적되면(일반적으로 1-2주), 일치율과 품질 점수를 기준으로 다음 단계 진입 여부를 판단한다.

### 카나리 배포와 자동 롤백

4단계(소규모 프로덕션)에서는 카나리 배포 패턴을 적용한다. 전체 트래픽의 5-10%만 새 에이전트로 라우팅하고, 나머지는 기존 시스템이 처리한다. 핵심 메트릭(오류율, 응답 시간, 비용)이 임계값을 초과하면 자동으로 트래픽을 기존 시스템으로 복원한다.

자동 롤백은 에이전트 배포에서 특히 중요하다. LLM의 비결정적 특성으로 인해 테스트에서 발견하지 못한 엣지 케이스가 프로덕션에서 발생할 수 있기 때문이다.

## 11.8 감사 로그 설계: 무결성과 추적 가능성

11.5절에서 감사 로그의 기본 개념과 필수 항목을 다루었다. 이 절에서는 프로덕션 환경에서 요구되는 암호화 서명, 변조 방지, 에이전트 행동의 완전한 추적 가능성을 설계한다.

### 암호화 서명 기반 로그 무결성

각 로그 엔트리에 디지털 서명을 적용하면 사후 변조를 탐지할 수 있다. HMAC(Hash-based Message Authentication Code)을 사용하여 로그 내용과 타임스탬프를 함께 서명한다.

```python
import hmac, hashlib, json

def sign_log_entry(entry: dict, secret_key: bytes) -> str:
    payload = json.dumps(entry, sort_keys=True).encode()
    return hmac.new(secret_key, payload, hashlib.sha256).hexdigest()
```

서명된 로그는 검증 시 동일한 키로 서명을 재계산하고, 저장된 서명과 비교하여 변조 여부를 판단한다.

### 해시 체인: 블록체인 원리의 적용

해시 체인(Hash Chain)은 각 로그 엔트리가 이전 엔트리의 해시를 포함하는 구조이다. 블록체인의 핵심 원리와 동일하며, 중간 로그가 삭제되거나 변경되면 체인이 깨져 즉시 탐지된다.

해시 체인의 구조는 다음과 같다. 첫 번째 엔트리는 초기 해시(genesis hash)를 가지고, 이후 각 엔트리는 `hash(이전_해시 + 현재_내용 + 타임스탬프)`로 연결된다. 검증 시에는 첫 엔트리부터 순차적으로 해시를 재계산하여 체인의 무결성을 확인한다.

### 에이전트 행동의 완전한 추적

에이전트 시스템에서는 단순한 입출력 기록을 넘어, 중간 추론 과정까지 추적해야 한다. 완전한 추적 가능성(Full Traceability)을 위해 다음 정보를 기록한다.

**표 11.4** 에이전트 감사 로그 확장 스키마

| 계층 | 기록 항목 | 목적 |
|:---|:---|:---|
| 요청 | 원본 입력, 세션 ID, 사용자 ID | 요청 식별 |
| 추론 | 프롬프트, 모델 응답, 토큰 수, 지연 시간 | 의사결정 추적 |
| 도구 | 호출된 도구, 파라미터, 반환값, 실행 시간 | 행동 감사 |
| 승인 | 승인 요청, 승인자, 결정, 사유 | 책임 추적 |
| 결과 | 최종 출력, 품질 점수, 후속 조치 | 결과 검증 |

이 스키마를 적용하면 "에이전트가 왜 이 도구를 호출했는가", "누가 이 작업을 승인했는가"와 같은 질문에 완전하게 답할 수 있다. 규제 대응, 사고 조사, 시스템 개선의 근거가 된다.

## 11.9 실습: 승인 워크플로우 구현

이 실습에서는 위험도에 따른 차등 승인 로직과 감사 로그가 포함된 HITL 워크플로우를 구현한다. 저위험 작업은 자동 진행하고, 중위험은 단일 승인, 고위험은 반드시 승인 후 실행되는 구조이다.

_전체 코드는 practice/chapter11/code/11-7-hitl-workflow.py 참고_

### 실습 환경 설정

```bash
cd practice/chapter11

```

### 핵심 구현

위험도 분류와 승인 라우팅은 LangGraph 조건부 엣지로 구현한다.

```python
graph.add_conditional_edges(
    "classify",
    route_after_classification,
    {"approve": "approve", "execute": "execute"}
)
```

저위험 작업은 classify 노드에서 바로 execute 노드로 진행한다. 중위험/고위험 작업은 approve 노드를 거쳐 승인 여부에 따라 execute 또는 block으로 분기한다.

### 감사 로그 구현

모든 이벤트는 AuditLog 클래스를 통해 기록된다.

```python
audit_log.log("approval_request", {
    "request_id": state["request_id"],
    "action_type": state["action_type"],
    "decision": approval_status
})
```

**표 11.5** HITL 워크플로우 실행 결과

| 작업 유형 | 대상 | 위험도 | 승인 필요 | 결과 |
|:---|:---|:---:|:---:|:---:|
| read (조회) | users_table | 낮음 | 아니오 | 실행됨 |

## 11.10 실패 사례와 교훈

HITL 워크플로우 운영 시 발생할 수 있는 문제와 대응 방법을 정리한다.

### 승인 지연으로 인한 병목

승인 요청이 쌓이면 전체 시스템이 느려진다. 이를 완화하려면 승인자 풀을 확대하거나, 비동기 승인 채널을 활용하거나, 타임아웃 후 자동 에스컬레이션하는 정책을 적용한다.

### 승인 피로 (Approval Fatigue)

너무 많은 승인 요청은 승인자의 피로를 유발하고, 형식적인 승인으로 이어질 수 있다. 위험도 분류를 정교화하여 정말 중요한 작업에만 승인을 요구해야 한다. 저위험 작업의 자동화 비율을 높이는 것이 핵심이다.

### 권한 남용과 우회

긴급 우회 절차가 남용되면 HITL의 의미가 사라진다. 모든 우회는 예외 없이 로깅하고, 정기적으로 감사해야 한다. 우회 권한은 최소 인원에게만 부여하고, 사용 시마다 사유를 기록하게 한다.

### 로그 누락과 데이터 무결성

감사 로그가 누락되면 책임 추적이 불가능하다. 로깅은 동기적으로 수행하고, 로그 저장 실패 시 작업도 실패하도록 구현하는 것이 안전하다. 해시 체인이나 블록체인 기반 로깅을 통해 무결성을 강화할 수도 있다.

## 핵심 정리

- HITL은 AI의 효율성과 사람의 판단력을 조합하는 패턴이다. 위험도에 따라 자동·단일·필수 승인으로 차등화한다.
- LangGraph의 interrupt와 Command로 승인 워크플로우를 구현하며, Checkpointer가 상태를 보존한다.
- 에이전트 배포는 Shadow Mode → 샌드박스 → 스테이징 → 소규모 프로덕션 → 전체 배포의 5단계로 점진적으로 진행한다.
- 감사 로그는 HMAC 서명과 해시 체인으로 무결성을 보장하고, 요청·추론·도구·승인·결과 5개 계층을 기록한다.
- 승인 피로를 방지하기 위해 저위험 작업은 자동화하고, 카나리 배포와 자동 롤백으로 프로덕션 안정성을 확보한다.

## 12.1 배포 옵션과 고려사항

에이전트 시스템의 배포 방식은 팀 규모, 트래픽, 예산에 따라 결정된다.

### 로컬 실행

개발과 테스트 환경에서는 로컬 실행이 가장 간단하다. 빠른 피드백과 디버깅이 가능하지만, 확장성이 없고 단일 장애점이 된다.

### Docker 컨테이너

Docker는 이식성과 재현성을 제공한다. 동일한 이미지를 개발, 스테이징, 프로덕션에서 실행할 수 있어 환경 차이로 인한 문제를 줄인다. 소규모 프로덕션에 적합하며, Kubernetes 등 오케스트레이션 도구와 결합하면 확장도 가능하다.

### 클라우드 배포

대규모 트래픽과 고가용성이 필요한 경우 클라우드 배포가 적합하다. AWS, GCP, Azure 등은 서버리스 함수, 컨테이너 서비스, 관리형 Kubernetes를 제공한다. 단, 복잡성과 비용이 증가한다.

**표 12.1** 배포 옵션 비교

| 옵션 | 장점 | 단점 | 적합한 상황 |
|:---|:---|:---|:---|
| 로컬 | 간단, 빠른 피드백 | 확장 불가, 단일 장애점 | 개발, 테스트 |
| Docker | 이식성, 재현성 | 오케스트레이션 필요 | 소규모 프로덕션 |
| 클라우드 | 확장성, 고가용성 | 복잡성, 비용 | 대규모 프로덕션 |

## 12.2 관측 가능성 (Observability)

관측 가능성(Observability)은 시스템의 내부 상태를 외부에서 파악할 수 있는 능력이다. LLM 애플리케이션은 비결정적이므로 전통적인 모니터링만으로는 부족하다.

### 관측 가능성의 세 요소

첫째, 로깅(Logging)은 이벤트를 시간순으로 기록한다. 요청 내용, 응답 내용, 에러 메시지 등을 포함한다.

둘째, 트레이싱(Tracing)은 요청의 전체 흐름을 추적한다. 에이전트가 여러 단계를 거칠 때 각 단계의 입출력과 소요 시간을 기록한다.

셋째, 메트릭(Metrics)은 수치화된 측정값이다. 요청 수, 응답 시간, 에러율, 토큰 사용량 등을 시계열로 수집한다.

### 주요 도구

**Langfuse**는 가장 많이 사용되는 오픈소스 LLM 관측 도구이다. 포괄적인 트레이싱, 평가, 프롬프트 관리, 메트릭을 제공한다. MIT 라이선스로 셀프호스팅이 가능하며, OpenTelemetry와 LangChain에 통합된다.

**LangSmith**는 LangChain 팀의 관리형 관측 스위트이다. LangChain/LangGraph 생태계와 심층 통합되며, 대시보드, 알림, HITL 평가를 제공한다. 셀프호스팅은 Enterprise 플랜에서만 가능하다.

```python
from langfuse.decorators import observe

@observe()
def process_query(query: str) -> str:
    # 자동으로 트레이스가 기록됨
    return llm.invoke(query)
```

## 12.3 운영 메트릭 설계

효과적인 모니터링을 위해 추적할 메트릭을 정의해야 한다.

### 핵심 메트릭

LLM 에이전트 시스템에서 추적해야 할 핵심 메트릭은 다음과 같다. 요청 수는 시스템 사용량을 나타낸다. 응답 시간은 사용자 경험을 결정한다. 토큰 사용량은 비용과 직결된다. 에러율은 시스템 안정성을 보여준다. 성공률은 작업 완료 비율이다.

### SLI와 SLO

서비스 수준 지표(SLI)는 측정 가능한 메트릭이다. 예를 들어 "p95 응답 시간"이나 "성공률"이 SLI가 된다. 서비스 수준 목표(SLO)는 SLI에 대한 목표값이다. "p95 응답 시간 < 5초", "성공률 > 99%"와 같이 정의한다.

### 알림 설정

임계값을 기반으로 알림을 설정한다. 에러율이 5%를 초과하면 즉시 알림을 보내고, 응답 시간이 10초를 초과하면 경고를 발송한다. 일일 비용이 예산의 80%에 도달하면 주의 알림을 보낸다.

## 12.4 비용 추정과 최적화

LLM API는 토큰 단위로 과금되므로 비용 관리가 중요하다.

### 토큰 기반 과금 구조

대부분의 LLM API는 입력 토큰과 출력 토큰을 별도로 과금한다. 출력 토큰이 입력 토큰보다 비싸므로, 응답 길이 제어가 비용 절감에 효과적이다.

**표 12.2** 주요 LLM API 비용 (2025년 기준, USD/1M 토큰)

| 모델 | 입력 | 출력 | 특징 |
|:---|---:|---:|:---|
| GPT-4o | $2.50 | $10.00 | 고성능 멀티모달 |
| GPT-4o-mini | $0.15 | $0.60 | 비용 효율 |
| Claude 3.5 Sonnet | $3.00 | $15.00 | 긴 컨텍스트 |
| Claude Haiku | $0.25 | $1.25 | 최저 비용 |

### 비용 추정 공식

월간 비용은 다음과 같이 계산한다.

```
월간 비용 = (입력 토큰 × 입력 단가 + 출력 토큰 × 출력 단가) × 30일
```

예를 들어, GPT-4o-mini로 일일 10만 입력 토큰, 5만 출력 토큰을 사용하면 일일 비용은 약 $0.045, 월간 비용은 약 $1.35이다.

### 최적화 전략

첫째, 모델 선택을 최적화한다. 모든 작업에 GPT-4o가 필요한 것은 아니다. 단순 분류, 요약, 대화에는 GPT-4o-mini나 Claude Haiku로 충분할 수 있다. 복잡한 추론에만 고성능 모델을 사용한다.

둘째, 프롬프트를 최적화한다. 불필요한 지시문과 예시를 제거하여 토큰을 절약한다. 시스템 프롬프트 압축만으로 62%의 크기 감소가 보고되었다.

셋째, 캐싱을 활용한다. 동일한 프롬프트에 대한 응답을 캐싱하면 API 호출을 줄일 수 있다. Anthropic의 프롬프트 캐싱은 캐시 읽기가 기본 비용의 10%에 불과하다.

넷째, 응답 길이를 제어한다. max_tokens를 적절히 설정하여 불필요하게 긴 응답을 방지한다.

## 12.5 보안과 키 관리

API 키는 외부 서비스에 대한 접근 권한이므로 철저한 관리가 필요하다.

### API 키 관리 원칙

API 키를 코드에 절대 하드코딩하지 않는다. 환경 변수, .env 파일, 또는 AWS Secrets Manager, HashiCorp Vault 같은 시크릿 매니저를 사용한다.

```python
import os
from dotenv import load_dotenv

load_dotenv()
api_key = os.getenv("OPENAI_API_KEY")
```

### 키 로테이션

정기적으로 API 키를 교체한다. 키가 노출되었거나 직원이 퇴사한 경우 즉시 교체한다. 새 키 발급 → 애플리케이션 업데이트 → 구 키 폐기 순서로 진행하여 서비스 중단을 방지한다.

### 최소 권한 원칙

필요한 권한만 부여한다. 읽기 전용 작업에는 읽기 권한만 가진 키를 발급한다. 프로덕션과 개발 환경의 키를 분리한다.

### 접근 제어와 감사

누가 키에 접근했는지 기록한다. 키 사용 내역을 정기적으로 감사한다. 비정상적인 사용 패턴을 탐지하면 알림을 발송한다.

## 12.6 프로덕션 체크리스트

프로덕션 배포 전 확인해야 할 항목을 체크리스트로 정리한다.

### 배포 전 체크리스트

환경 변수와 시크릿이 올바르게 설정되었는지 확인한다. API 키가 코드에 하드코딩되지 않았는지 검증한다. 의존성 버전이 고정되어 있는지 확인한다. 에러 핸들링과 재시도 로직이 구현되어 있는지 점검한다.

### 모니터링 체크리스트

요청/응답 로깅이 활성화되어 있는지 확인한다. 민감 정보가 마스킹되는지 검증한다. 핵심 메트릭(요청 수, 응답 시간, 에러율, 비용)이 추적되는지 확인한다. 알림 임계값이 설정되어 있는지 점검한다.

### 보안 체크리스트

API 키가 시크릿 매니저에 저장되어 있는지 확인한다. 키 로테이션 절차가 수립되어 있는지 검증한다. 최소 권한 원칙이 적용되어 있는지 점검한다. 감사 로그가 활성화되어 있는지 확인한다.

### 비용 관리 체크리스트

예상 월간 비용이 계산되어 있는지 확인한다. 비용 한도가 설정되어 있는지 검증한다. 비용 이상 알림이 설정되어 있는지 점검한다.

def estimate_cost(self, input_tokens: int, output_tokens: int):

## 12.8 실패 사례와 교훈

프로덕션 운영 시 발생할 수 있는 문제와 대응 방법을 정리한다.

### 예상치 못한 비용 폭증

새로운 기능 출시 후 트래픽이 급증하거나, 무한 루프로 API를 반복 호출하면 비용이 폭증할 수 있다. 일일/월간 비용 한도를 설정하고, 한도 도달 시 자동으로 요청을 차단하는 서킷 브레이커를 구현해야 한다.

### API 키 노출 사고

GitHub에 API 키가 포함된 코드를 푸시하면 수분 내에 악용될 수 있다. git-secrets나 pre-commit hook을 사용하여 키 커밋을 방지한다. 노출된 키는 즉시 폐기하고 새 키를 발급한다.

### 모니터링 누락으로 인한 장애 지연 탐지

모니터링이 없으면 장애를 사용자 신고로 알게 된다. 핵심 메트릭에 대한 알림을 반드시 설정하고, 정기적으로 알림이 작동하는지 테스트한다.

### 용량 초과 문제

API 레이트 리밋에 도달하면 요청이 거부된다. 레이트 리밋을 파악하고 요청을 분산하거나, 지수 백오프로 재시도한다. 필요시 레이트 리밋 증가를 요청한다.

## 핵심 정리

1. 배포 옵션은 팀 규모, 트래픽, 예산에 따라 로컬, Docker, 클라우드 중 선택한다.
2. 관측 가능성의 세 요소는 로깅, 트레이싱, 메트릭이다.
3. LLM API는 토큰 단위로 과금되며, 모델 선택과 프롬프트 최적화로 비용을 절감한다.
4. API 키는 절대 코드에 하드코딩하지 않고 시크릿 매니저를 사용한다.
5. 프로덕션 배포 전 체크리스트로 환경, 모니터링, 보안, 비용을 점검한다.

## 마무리

이 책에서는 AI 협업 개발의 기초부터 프로덕션 배포까지의 여정을 다루었다. 1장에서 AI와의 역할 분담을 이해하고, MCP로 도구를 표준화하고, LangChain/LangGraph로 워크플로우를 구축했다. RAG와 GraphRAG로 지식 기반 시스템을 만들고, 검증 파이프라인과 HITL로 신뢰성을 확보했다.

핵심은 AI가 모든 것을 자동화하는 것이 아니라, AI의 효율성과 사람의 판단력을 적절히 조합하는 것이다. 고위험 작업에는 사람 승인을, 반복 작업에는 자동화를, 불확실한 영역에는 검증 파이프라인을 적용한다.

이 책에서 다룬 원칙과 패턴이 실무에서 AI 에이전트 시스템을 설계하고 운영하는 데 도움이 되기를 바란다.

# 10주차. Skills 심화

> **1회차** (강의 90분): 스킬 설계 패턴(체크리스트형/워크플로우형/가드레일형), 커뮤니티 스킬 활용, MCP와 Skills의 관계, 스킬 조합 전략
> **2회차** (실습 90분): 테스트 작성 스킬 + 자유 주제 도메인 특화 스킬 설계 및 구현

---

## 학습목표

1. 스킬 설계 패턴(체크리스트형, 워크플로우형, 가드레일형)을 구분하고, 상황에 맞는 패턴을 선택할 수 있다
2. 커뮤니티 스킬을 탐색하고, 팀 상황에 맞게 커스터마이징하여 도입할 수 있다
3. MCP(도구)와 Skills(절차)의 역할 차이를 이해하고, 두 기술의 조합 전략을 설명할 수 있다
4. 특정 도메인에 특화된 스킬을 체계적인 방법론에 따라 설계하고 구현할 수 있다

## 선수지식

- 9주차: SKILL.md 파일 구조(YAML 프론트매터 + 마크다운 본문), 3단계 점진적 공개 모델
- 9주차 과제: 코드 리뷰 스킬, API 문서화 스킬 작성 경험
- 4~7주차: MCP 서버의 개념과 구현 경험 (Tool, Resource, Prompt 이해)

## 1회차: 강의

### 10.1 스킬 설계 원칙 복습과 확장

9주차에서 SKILL.md의 구조와 작성법을 익혔다. 코드 리뷰 스킬과 API 문서화 스킬이라는 두 가지 실습을 통해, 스킬이 Copilot의 출력을 팀의 기준에 맞추어 표준화하는 메커니즘임을 확인하였다. 이번 주에서는 이 경험을 바탕으로 더 체계적인 스킬 설계 방법론으로 확장한다.

9주차 과제를 수행하면서 다음과 같은 질문이 떠올랐을 수 있다.

- "코드 리뷰 스킬과 API 문서 스킬은 구조가 꽤 다른데, 스킬에도 유형이 있는 것인가?"
- "커뮤니티가 공유하는 스킬을 활용하면 처음부터 만들지 않아도 되지 않는가?"
- "MCP 서버와 스킬을 함께 쓰면 어떤 시너지가 생기는가?"
- "스킬을 여러 개 동시에 쓸 때 충돌하지 않는가?"

이번 강의에서는 이 질문들에 대한 답을 하나씩 제시한다.

#### 효과적인 스킬 설계의 6가지 원칙

스킬 설계 패턴에 들어가기 전에, 9주차에서 암묵적으로 적용한 설계 원칙을 명시적으로 정리한다.

**원칙 1: description이 모든 것을 결정한다.** Copilot은 스킬의 본문을 읽기 전에 description만으로 활성화 여부를 판단한다. 사용자가 입력할 법한 키워드와 문맥을 description에 충분히 포함시켜야 한다.

**원칙 2: 하나의 스킬은 하나의 도메인에 집중한다.** 코드 리뷰와 API 문서 생성을 하나의 스킬에 합치면 description이 모호해지고, Copilot이 불필요한 상황에서 스킬을 활성화하거나 필요한 상황에서 활성화하지 못하는 문제가 발생한다. 관심사 분리(Separation of Concerns) 원칙은 스킬 설계에도 그대로 적용된다.

**원칙 3: 규칙을 서술하기보다 예시를 보여 준다.** "변수명은 명확하게 짓는다"라는 추상적 규칙보다 "good: `user_count`, bad: `uc`"처럼 구체적인 예시를 제시하면 Copilot이 더 정확하게 의도를 파악한다.

**원칙 4: 번들 파일을 적극적으로 활용한다.** SKILL.md 본문에 모든 내용을 넣으면 컨텍스트 윈도우를 불필요하게 소비한다. 상세한 체크리스트, API 문서, 템플릿은 별도 파일로 분리하고 SKILL.md에서 참조만 한다.

**원칙 5: 스킬을 버전 관리하고 팀과 공유한다.** `.github/skills/` 디렉터리에 저장한 스킬은 Git으로 관리되므로, PR을 통해 팀 리뷰를 거칠 수 있다. 스킬 자체도 코드와 마찬가지로 지속적으로 개선해 나가야 하는 자산이다.

**원칙 6: 스킬 간 충돌을 주의한다.** 유사한 도메인을 다루는 스킬이 여러 개 있으면 Copilot이 동시에 여러 스킬을 활성화할 수 있다. 각 스킬의 description이 서로 명확히 구분되도록 작성해야 한다.

### 10.2 스킬 설계 패턴

9주차에서 작성한 코드 리뷰 스킬은 점검 항목을 나열하고 순서대로 확인하는 형태였고, API 문서화 스킬은 입력(소스 코드)을 받아 특정 형식의 출력(문서)을 생성하는 형태였다. 이처럼 스킬은 해결하려는 문제의 성격에 따라 서로 다른 구조적 패턴을 갖는다. 여기서는 실무에서 자주 등장하는 세 가지 설계 패턴을 체계적으로 분류하고, 각 패턴의 특성과 적합한 사용 상황을 정리한다.

#### 패턴 1: 체크리스트형 (Checklist Pattern)

체크리스트형 스킬은 **정해진 점검 항목을 하나씩 확인하고 결과를 보고하는** 구조이다. 9주차에서 작성한 코드 리뷰 스킬이 이 패턴에 해당한다.

**구조적 특징**:
- 점검 항목의 목록이 본문의 핵심이다
- 각 항목은 독립적으로 평가할 수 있다 (순서가 바뀌어도 결과가 동일)
- 출력은 항목별 통과/실패 또는 심각도 등급이다
- 상세 체크리스트는 `references/` 폴더에 번들로 분리하기 적합하다

**적합한 사용 상황**:
- 코드 리뷰 (보안, 성능, 가독성 점검)
- 배포 전 검증 (환경 변수, 설정 파일, 의존성 확인)
- 접근성 검사 (WCAG 기준 항목별 점검)
- 보안 감사 (OWASP Top 10 항목 점검)
- PR 머지 전 확인 사항

**SKILL.md 핵심 구조 예시**:

```markdown
## 점검 항목

### 범주 A: 보안
- [ ] 점검 항목 A-1: 상세 설명
- [ ] 점검 항목 A-2: 상세 설명

### 범주 B: 성능
- [ ] 점검 항목 B-1: 상세 설명

## 출력 형식
각 항목에 대해 [통과/실패/해당없음]과 근거를 보고한다.
```

체크리스트형 스킬을 설계할 때 주의할 점은 항목의 수를 적정하게 유지하는 것이다. 항목이 너무 많으면 Copilot이 모든 항목을 꼼꼼히 확인하지 못할 수 있다. 핵심 항목은 SKILL.md 본문에 두고, 상세 항목은 `references/` 파일로 분리하여 3단계 점진적 공개 모델을 활용하는 것이 효과적이다.

#### 패턴 2: 워크플로우형 (Workflow Pattern)

워크플로우형 스킬은 **정해진 순서에 따라 단계적으로 작업을 수행하는** 구조이다. 9주차에서 작성한 API 문서화 스킬이 이 패턴에 가깝다. 입력을 받아 여러 단계를 거쳐 특정 형식의 출력을 생산한다.

**구조적 특징**:
- 단계의 순서가 중요하다 (순서가 바뀌면 결과가 달라질 수 있다)
- 이전 단계의 출력이 다음 단계의 입력이 된다
- 최종 출력의 형식이 명확하게 정의되어 있다
- 템플릿 파일을 `assets/` 폴더에 번들로 포함하기 적합하다

**적합한 사용 상황**:
- API 문서 생성 (코드 분석 → 정보 추출 → 템플릿 적용 → 문서 출력)
- 테스트 작성 (대상 분석 → 테스트 케이스 설계 → 코드 생성 → 검증)
- 커밋 메시지 작성 (변경 사항 분석 → 분류 → Conventional Commits 형식 적용)
- 리팩토링 절차 (문제 식별 → 영향 분석 → 수정 계획 → 구현)
- 릴리스 노트 생성 (커밋/이슈 수집 → 분류 → 형식 적용 → 출력)

**SKILL.md 핵심 구조 예시**:

```markdown
## 작업 절차

### 1단계: 입력 분석
- 대상 파일/코드에서 필요한 정보를 식별한다
- [구체적인 식별 기준]

### 2단계: 정보 추출
- 1단계에서 식별한 항목의 세부 정보를 추출한다
- [추출할 정보의 종류와 방법]

### 3단계: 변환/생성
- `assets/template.md`의 형식을 적용한다
- 추출한 정보로 각 섹션을 채운다

### 4단계: 검증
- 생성된 결과물의 완성도를 점검한다
- 누락된 정보가 있으면 2단계로 돌아간다

## 출력 형식
[최종 산출물의 구조를 명시]
```

워크플로우형 스킬의 핵심은 각 단계 사이의 **데이터 흐름**을 명확히 정의하는 것이다. "1단계에서 추출한 엔드포인트 목록을 2단계에서 상세화한다"처럼, 이전 단계의 결과물이 다음 단계에서 어떻게 사용되는지를 명시해야 Copilot이 절차를 정확히 따른다.

#### 패턴 3: 가드레일형 (Guardrail Pattern)

가드레일형 스킬은 **특정 행동을 금지하거나 제약 조건을 설정하는** 구조이다. 체크리스트형이나 워크플로우형과 달리, 가드레일형은 "무엇을 해야 하는가"보다 **"무엇을 하지 말아야 하는가"** 또는 **"어떤 범위 안에서 행동해야 하는가"**를 정의한다.

**구조적 특징**:
- 금지 사항, 제약 조건, 경계 규칙이 본문의 핵심이다
- "~하지 말아야 한다", "~을 초과하지 않는다", "반드시 ~해야 한다"와 같은 규칙이 중심이다
- 다른 패턴의 스킬과 함께 사용되는 경우가 많다
- 위반 시 경고를 출력하도록 지시할 수 있다

**적합한 사용 상황**:
- 코딩 표준 강제 (금지된 패턴, 필수 패턴)
- 보안 가드레일 (eval 사용 금지, 환경변수 직접 참조 금지)
- 아키텍처 경계 (특정 레이어 간 직접 참조 금지)
- 성능 제약 (최대 응답 시간, 최대 쿼리 수)
- 라이선스 준수 (사용 금지 라이브러리 목록)

**SKILL.md 핵심 구조 예시**:

```markdown
## 금지 사항 (절대 규칙)
- eval(), exec() 함수를 사용하지 않는다
- SQL 문자열을 직접 조합(string concatenation)하지 않는다
- API 키를 소스 코드에 하드코딩하지 않는다

## 제약 조건
- 함수 하나의 줄 수는 50줄을 초과하지 않는다
- 함수의 매개변수는 5개를 초과하지 않는다
- 중첩 깊이(nesting depth)는 3단계를 초과하지 않는다

## 필수 사항
- 모든 public 함수에는 docstring을 작성한다
- 에러 처리는 구체적인 예외 타입을 사용한다 (bare except 금지)
- 외부 API 호출에는 반드시 타임아웃을 설정한다

## 위반 시 행동
위 규칙에 위반하는 코드를 발견하면 다음과 같이 경고한다:
⚠️ [규칙 ID] 위반 내용 + 수정 방안
```

가드레일형 스킬의 강점은 다른 패턴의 스킬과 자연스럽게 결합된다는 점이다. 코드 리뷰 스킬(체크리스트형)이 점검 항목을 확인하는 동안, 코딩 표준 가드레일 스킬이 금지 패턴 위반을 감시하는 이중 구조가 가능하다.

#### 패턴 비교 요약

**표 10.1** 스킬 설계 패턴 비교

| 특성 | 체크리스트형 | 워크플로우형 | 가드레일형 |
|------|:----------:|:----------:|:---------:|
| 핵심 질문 | "무엇을 점검하는가?" | "어떤 순서로 수행하는가?" | "무엇을 하지 말아야 하는가?" |
| 순서 중요도 | 낮음 | 높음 | 해당 없음 |
| 출력 형식 | 항목별 결과 보고 | 최종 산출물 | 위반 경고 |
| 주 번들 유형 | references/ (체크리스트) | assets/ (템플릿) | references/ (규칙 목록) |
| 독립 사용 | 가능 | 가능 | 다른 패턴과 조합 권장 |

### 10.3 커뮤니티 스킬 활용

스킬을 처음부터 모두 직접 만들 필요는 없다. GitHub의 `github/awesome-copilot` 저장소와 커뮤니티 채널에는 커뮤니티가 공유한 40개 이상의 스킬이 모여 있으며, 다양한 도메인을 다룬다.

#### 커뮤니티 스킬 설치 방법

커뮤니티 스킬을 도입하는 절차는 간단하다.

1. `awesome-copilot-skills` 저장소에서 원하는 스킬 폴더를 찾는다
2. 해당 폴더를 다운로드하여 프로젝트의 `.github/skills/` 디렉터리에 복사한다
3. SKILL.md의 내용을 검토하고, 필요시 팀 상황에 맞게 수정한다
4. Copilot Chat에서 관련 프롬프트로 동작을 확인한다

예를 들어, 커밋 메시지 규칙 스킬을 도입하려면 해당 스킬 폴더를 `.github/skills/commit-convention/`에 넣는다. 이후 "이 변경 사항에 대한 커밋 메시지를 작성해 줘"라고 요청하면 스킬에 정의된 Conventional Commits 형식을 따른다.

#### 커뮤니티에서 유용한 스킬 유형

커뮤니티에서 특히 유용하다고 평가받는 스킬 유형은 다음과 같다.

| 스킬 유형 | 설명 | 패턴 분류 |
|----------|------|:---------:|
| Git 커밋 컨벤션 | Conventional Commits 형식 강제 | 워크플로우형 |
| 웹 접근성 검사 | WCAG 기준 적용 | 체크리스트형 |
| DB 마이그레이션 | 스키마 변경 절차 표준화 | 워크플로우형 |
| 다이어그램 생성 | Mermaid 문법으로 다이어그램 작성 | 워크플로우형 |
| 보안 코딩 규칙 | OWASP 기준 코딩 가이드라인 | 가드레일형 |

#### 커스터마이징이 핵심이다

커뮤니티 스킬을 **그대로 사용하기보다는 팀의 상황에 맞게 수정**하는 것을 권장한다. 수정이 필요한 대표적인 영역은 다음과 같다.

1. **description 수정**: 커뮤니티 스킬의 description은 영문인 경우가 많다. 팀이 한국어로 프롬프트를 입력한다면 한국어 키워드를 추가하거나 description을 한국어로 재작성해야 활성화 확률이 높아진다.
2. **규칙 조정**: 커뮤니티의 일반적인 규칙을 프로젝트의 구체적인 관례로 대체한다. 예를 들어, 커밋 메시지 스킬의 타입 목록에 팀이 자체적으로 사용하는 타입을 추가한다.
3. **번들 파일 교체**: 범용 템플릿을 팀의 문서 형식에 맞는 템플릿으로 교체한다.

커뮤니티 스킬은 완성품이 아니라 **출발점**으로 활용하는 것이 가장 효과적이다.

#### 보안 유의사항

커뮤니티 스킬을 도입할 때 반드시 지켜야 할 보안 습관이 있다. 스킬의 SKILL.md 본문을 **반드시 검토**해야 한다. 스킬은 Copilot의 행동 방식에 직접 영향을 미치므로, 의도하지 않은 지시가 포함되어 있지 않은지 확인하는 과정이 필요하다. 이는 서드파티 라이브러리의 코드를 리뷰하는 것과 동일한 원칙이다.

특히 `scripts/` 폴더에 실행 가능한 스크립트가 포함된 경우, 해당 스크립트가 어떤 동작을 수행하는지 반드시 확인한다. 파일 삭제, 네트워크 요청, 환경변수 접근 등 민감한 작업이 포함되어 있을 수 있다.

### 10.4 MCP와 Skills의 관계

4~7주차에서 다룬 MCP와 9~10주차의 Skills는 Copilot을 확장하는 두 축이다. 이 둘의 관계를 명확히 이해하면 적절한 확장 전략을 수립할 수 있다.

#### 도구 vs 지식

MCP 서버는 Copilot에 **도구(Tool)**를 제공한다. 데이터베이스를 조회하거나, 외부 API를 호출하거나, 파일 시스템을 조작하는 것처럼 Copilot이 실제로 무언가를 "실행"할 수 있게 만든다.

Skills는 Copilot에 **지식(knowledge)**과 **절차(procedure)**를 제공한다. 코드를 어떤 기준으로 리뷰해야 하는지, 문서를 어떤 형식으로 작성해야 하는지처럼 Copilot이 올바르게 "판단"할 수 있게 만든다.

**표 10.2** MCP와 Skills의 역할 비교

| 구분 | MCP | Skills |
|------|-----|--------|
| 제공하는 것 | 도구 (실행 능력) | 지식과 절차 (판단 기준) |
| 비유 | 새 도구를 손에 쥐어 주는 것 | 매뉴얼을 읽게 하는 것 |
| 예시 | DB 조회, API 호출, 파일 변환 | 코딩 표준, 리뷰 체크리스트, 문서 템플릿 |
| 활성화 | 도구 호출 시 | 프롬프트 매칭 시 |
| 구현 난이도 | 높음 (서버 개발 필요) | 낮음 (마크다운 파일 작성) |
| 런타임 의존성 | 서버 프로세스 실행 필요 | 파일 시스템 접근만 필요 |

#### MCP + Skills 조합 패턴

실무에서의 전형적인 조합 패턴을 살펴본다.

**패턴 A: 데이터 수집 + 형식화**

MCP 서버가 외부 시스템에서 데이터를 가져오고, Skills가 그 데이터를 팀의 형식에 맞추어 정리하는 구조이다.

```
예시: 릴리스 노트 자동 생성
- MCP 서버 → Jira에서 이슈 목록을 가져온다 (도구)
- Skills   → 이슈 데이터를 릴리스 노트 형식으로 정리한다 (절차)
```

이때 MCP 없이는 Copilot이 Jira 데이터에 접근할 수 없고, Skills 없이는 Copilot이 팀의 문서 형식을 알 수 없다. 양쪽이 모두 갖추어져야 의미 있는 자동화가 가능하다.

**패턴 B: 환경 확인 + 판단**

MCP 서버가 현재 시스템 상태를 확인하고, Skills가 그 상태에 따른 행동 방침을 결정하는 구조이다.

```
예시: 배포 전 점검 자동화
- MCP 서버 → 스테이징 서버 상태를 조회한다 (도구)
- Skills   → 배포 가능 여부를 체크리스트에 따라 판단한다 (절차)
```

**패턴 C: 코드 분석 + 생성**

MCP 서버가 코드베이스를 분석하는 도구를 제공하고, Skills가 분석 결과를 바탕으로 코드를 생성하는 절차를 정의하는 구조이다.

```
예시: 자동 테스트 생성
- MCP 서버 → 코드 커버리지 도구로 미테스트 함수를 식별한다 (도구)
- Skills   → 팀의 테스트 패턴에 따라 테스트를 생성한다 (절차)
```

이 조합 패턴은 11주차 "MCP + Skills 통합 프로젝트"에서 실제로 구현한다.

### 10.5 스킬 조합 전략

프로젝트가 복잡해지면 하나의 스킬로 모든 요구를 충족시키기 어렵다. 여러 스킬을 함께 사용할 때의 전략과 주의사항을 정리한다.

#### 보완적 조합

서로 다른 관심사를 다루는 스킬을 함께 배치하는 전략이다. 각 스킬의 description이 충분히 구분되어 있으므로 Copilot이 상황에 따라 적절한 스킬만 활성화한다.

```
.github/skills/
├── code-review/          # 코드 점검 (체크리스트형)
├── api-docs/             # API 문서 생성 (워크플로우형)
├── test-writer/          # 테스트 생성 (워크플로우형)
└── coding-standards/     # 코딩 규칙 강제 (가드레일형)
```

이 구성에서 "코드 리뷰해 줘"라는 프롬프트에는 `code-review`와 `coding-standards`가 동시에 활성화될 수 있으며, 이는 의도한 동작이다. 체크리스트형 리뷰와 가드레일형 규칙이 서로 보완하며 더 철저한 리뷰 결과를 생성한다.

#### 계층적 조합

범용 스킬과 특화 스킬을 계층적으로 구성하는 전략이다.

```
.github/skills/
├── code-review/               # 범용 코드 리뷰
├── frontend-code-review/      # React 컴포넌트 리뷰 (특화)
└── backend-code-review/       # FastAPI 엔드포인트 리뷰 (특화)
```

이 경우 각 특화 스킬의 description에 대상 기술을 명시하여 구분한다. "React 컴포넌트를 리뷰해 줘"라는 프롬프트에는 `frontend-code-review`가, "FastAPI 엔드포인트를 리뷰해 줘"라는 프롬프트에는 `backend-code-review`가 활성화된다. 단순히 "코드 리뷰해 줘"라는 프롬프트에는 범용 `code-review`가 활성화된다.

#### 충돌 방지 원칙

스킬 간 충돌을 방지하기 위한 핵심 원칙은 다음과 같다.

1. **description의 명확한 분리**: 각 스킬의 description에 사용하는 키워드가 서로 겹치지 않도록 한다
2. **상충하는 지시 금지**: 두 스킬이 동시에 활성화되었을 때, 하나는 "함수를 길게 작성하라", 다른 하나는 "함수를 짧게 작성하라"처럼 상반된 지시를 하지 않도록 한다
3. **우선순위 명시**: 여러 스킬이 동시에 활성화될 가능성이 있다면, 각 스킬의 본문에 "다른 스킬과 충돌 시 이 스킬의 보안 규칙을 우선한다"와 같은 우선순위를 명시한다

### 10.6 도메인 특화 스킬 설계 방법론

2회차 실습에서 자유 주제의 도메인 특화 스킬을 설계해야 한다. 여기서는 어떤 도메인이든 적용할 수 있는 체계적인 설계 방법론을 제시한다.

#### 5단계 설계 프로세스

**1단계: 문제 정의**

"현재 Copilot이 이 작업을 수행할 때 어떤 점이 부족한가?"를 명확히 정의한다. 스킬은 문제를 해결하기 위한 도구이므로, 문제 정의가 불명확하면 효과적인 스킬을 만들 수 없다.

질문 체크리스트:
- 이 작업을 Copilot에 요청했을 때, 기대와 다른 결과가 나오는 부분은 무엇인가?
- 팀원마다 이 작업의 결과물이 다른 이유는 무엇인가?
- 이 작업에서 반복적으로 수정을 요청하는 부분은 무엇인가?

**2단계: 패턴 선택**

문제의 성격에 따라 적합한 설계 패턴을 선택한다.

| 문제 유형 | 적합한 패턴 |
|----------|:----------:|
| "점검해야 할 항목이 있다" | 체크리스트형 |
| "정해진 절차에 따라 산출물을 만들어야 한다" | 워크플로우형 |
| "특정 행동을 금지하거나 제약해야 한다" | 가드레일형 |
| "점검 + 산출물 생성 모두 필요하다" | 복합형 (패턴 조합) |

**3단계: description 설계**

사용자가 이 스킬이 필요한 상황에서 실제로 입력할 법한 프롬프트를 5개 이상 나열하고, 그 프롬프트들에 공통으로 포함된 키워드를 추출하여 description을 작성한다.

```
사용자 프롬프트 예상:
1. "이 코드의 테스트를 작성해 줘"
2. "단위 테스트를 추가해 줘"
3. "테스트 커버리지를 높여 줘"
4. "이 함수를 테스트해 줘"
5. "pytest로 테스트를 만들어 줘"

→ 추출 키워드: 테스트, 작성, 단위 테스트, 커버리지, pytest
→ description: "테스트를 작성하거나, 단위 테스트를 추가하거나, 테스트 커버리지를 높일 때 사용한다."
```

**4단계: 본문 및 번들 설계**

선택한 패턴에 따라 본문의 구조를 잡고, 상세 내용을 번들 파일로 분리할 지점을 결정한다.

- 체크리스트형: 핵심 항목은 본문, 상세 항목은 `references/`
- 워크플로우형: 절차는 본문, 템플릿은 `assets/`, 예시는 `examples/`
- 가드레일형: 핵심 규칙은 본문, 상세 규칙은 `references/`

**5단계: 반복 검증**

작성한 스킬을 Copilot Chat에서 테스트하고, 다음 세 가지를 확인한다.

1. **활성화 정확도**: 관련 프롬프트에서만 활성화되는가? 관련 없는 프롬프트에서 오활성화되지 않는가?
2. **지시 충실도**: 본문의 절차와 규칙을 정확히 따르는가?
3. **출력 일관성**: 같은 유형의 입력에 대해 일관된 형식의 결과를 생성하는가?

검증 결과에 따라 description, 본문, 번들 파일을 반복적으로 개선한다.

## 2회차: 실습

### 실습 1: 테스트 작성 스킬

프로젝트의 테스트 관례에 맞는 단위 테스트를 Copilot이 작성하도록 가르치는 스킬을 만든다. 테스트 작성은 팀마다 프레임워크, 네이밍 규칙, 픽스처 패턴이 다르기 때문에 스킬로 표준화하기에 적합한 영역이다. 이 실습에서는 워크플로우형 패턴을 적용한다.

#### 단계 1: 디렉터리 생성

```bash
mkdir -p .github/skills/test-writer/examples
```

#### 단계 2: SKILL.md 작성

`.github/skills/test-writer/SKILL.md` 파일을 작성한다.

```yaml
---
name: test-writer
description: >
  테스트를 작성하거나, 단위 테스트를 추가하거나,
  테스트 커버리지를 높일 때 사용한다.
  pytest와 프로젝트의 테스트 패턴을 따른다.
---

## 테스트 작성 절차

테스트 작성 요청을 받으면 다음 순서를 따른다.

### 1단계: 대상 분석
- 테스트 대상 모듈/함수의 역할을 파악한다
- 입력 파라미터의 타입과 범위를 확인한다
- 반환값의 타입과 가능한 값을 확인한다
- 외부 의존성(DB, API, 파일 시스템)을 식별한다

### 2단계: 테스트 케이스 설계
- 정상 케이스(happy path)를 먼저 설계한다
- 경계값(boundary) 케이스를 추가한다
- 예외/에러 케이스를 추가한다
- 각 케이스에 대해 기대 결과를 명시한다

### 3단계: 코드 생성
- 아래의 네이밍 규칙과 구조 규칙을 따라 테스트 코드를 작성한다
- 외부 의존성은 반드시 mock 처리한다 (pytest-mock 사용)
- 예시 코드는 `examples/` 폴더를 참고한다

### 4단계: 자가 검증
- 작성한 테스트가 실제로 실행 가능한지 확인한다
- 누락된 import가 없는지 확인한다
- mock 설정이 올바른지 확인한다

## 네이밍 규칙
- 파일명: `test_{모듈명}.py`
- 함수명: `test_{기능}_{시나리오}_{기대결과}`
- 클래스명: `Test{대상클래스명}` (클래스 기반 테스트 사용 시)

## 구조 규칙
- 각 테스트는 **Arrange-Act-Assert** 패턴을 따른다
  - Arrange: 테스트에 필요한 데이터와 조건을 준비한다
  - Act: 테스트 대상 함수/메서드를 실행한다
  - Assert: 결과가 기대값과 일치하는지 검증한다
- 테스트 간 의존성을 만들지 않는다 (각 테스트는 독립적으로 실행 가능해야 한다)
- 픽스처는 `conftest.py`에 정의하여 재사용한다

## 예시

good:
```python
def test_calculate_total_with_discount_returns_reduced_price():
    # Arrange
    items = [{"price": 100}, {"price": 200}]
    discount = 0.1

    # Act
    result = calculate_total(items, discount)

    # Assert
    assert result == 270.0
```

bad:
```python
def test1():  # 무의미한 이름
    assert calculate_total([{"price": 100}], 0) == 100  # AAA 패턴 미준수
```
```

#### 단계 3: 예시 파일 작성

`examples/test_user_service.py` 파일에 잘 작성된 테스트 예시를 넣어 둔다.

```python
"""사용자 서비스 테스트 예시.

이 파일은 test-writer 스킬이 참고하는 예시 테스트이다.
팀의 테스트 패턴(AAA, 네이밍 규칙, mock 처리)을 보여 준다.
"""
import pytest
from unittest.mock import AsyncMock


class TestUserService:
    """사용자 서비스에 대한 테스트."""

    def test_get_user_with_valid_id_returns_user(self, mock_db):
        # Arrange
        mock_db.find_one.return_value = {"id": 1, "name": "홍길동"}

        # Act
        result = user_service.get_user(1)

        # Assert
        assert result["name"] == "홍길동"

    def test_get_user_with_invalid_id_raises_not_found(self, mock_db):

        # Act & Assert

```

Copilot은 "보여 주기(show)" 방식의 예시로부터 패턴을 학습하므로, 추상적인 규칙만 나열하는 것보다 구체적인 예시 코드를 포함하는 것이 훨씬 효과적이다.

#### 단계 4: 비교 테스트

스킬의 효과를 체감하기 위해, 같은 대상 모듈에 대해 스킬이 없는 상태와 있는 상태에서의 Copilot 응답을 비교한다.

1. 먼저 `.github/skills/test-writer/` 폴더의 이름을 임시로 변경하여 스킬을 비활성화한다
2. Copilot Chat에서 "이 모듈의 테스트를 작성해 줘"라고 요청한다
3. 폴더 이름을 원래대로 복원하여 스킬을 활성화한다
4. 동일한 프롬프트를 다시 입력한다
5. 두 결과를 비교한다: 네이밍 규칙, AAA 구조, mock 처리 방식의 차이를 확인한다

스킬이 없을 때 Copilot은 일반적인 pytest 패턴으로 테스트를 생성하지만, 스킬이 활성화되면 팀의 네이밍 규칙, Arrange-Act-Assert 구조, mock 처리 방식이 정확히 반영된다.

### 실습 2: 도메인 특화 스킬 설계 및 구현 (자유 주제)

10.6절에서 학습한 5단계 설계 프로세스를 적용하여, 자유 주제의 도메인 특화 스킬을 직접 설계하고 구현한다. 이 실습은 본 과목의 핵심 역량인 "도메인 지식을 스킬로 체계화하는 능력"을 평가하는 과제와 직결된다.

#### 주제 선택 가이드

자유 주제이지만, 다음 기준에 부합하는 주제를 선택하면 실습의 학습 효과가 높다.

1. **반복적인 작업**: 매번 비슷한 패턴으로 수행하는 작업 (매주 작성하는 보고서, 반복되는 코드 패턴 등)
2. **팀 내 기준 불일치**: 사람마다 결과물이 다른 작업 (코드 스타일, 문서 형식 등)
3. **절차가 있는 작업**: 특정 순서를 따라야 하는 작업 (배포 절차, 코드 이관 등)

#### 주제 예시

| 주제 | 적합한 패턴 | description 키워드 예시 |
|------|:-----------:|----------------------|
| Git 커밋 메시지 규칙 | 워크플로우형 | 커밋 메시지, 커밋, 변경사항 |
| README 작성 | 워크플로우형 | README, 프로젝트 소개, 문서화 |
| 에러 핸들링 표준 | 가드레일형 | 에러 처리, 예외, try-catch |
| CSS/디자인 리뷰 | 체크리스트형 | 디자인 리뷰, CSS, 스타일 |
| 데이터베이스 쿼리 최적화 | 체크리스트형 | 쿼리, SQL, 성능, 최적화 |
| PR 설명 작성 | 워크플로우형 | PR, 풀 리퀘스트, 변경사항 설명 |
| 환경 변수 관리 | 가드레일형 | 환경변수, .env, 설정, 시크릿 |

#### 실습 진행 절차

**1단계 (15분): 문제 정의 및 패턴 선택**

선택한 도메인에서 해결하려는 문제를 한 문장으로 정의하고, 적합한 설계 패턴을 선택한다. 종이 또는 텍스트 파일에 다음을 작성한다.

```
문제 정의: ___________________________________________
선택 패턴: [ ] 체크리스트형  [ ] 워크플로우형  [ ] 가드레일형
선택 근거: ___________________________________________
```

**2단계 (15분): description 설계 및 SKILL.md 골격 작성**

예상 프롬프트 5개를 나열하고, 키워드를 추출하여 description을 작성한다. 디렉터리를 생성하고 SKILL.md의 프론트매터와 본문 골격을 작성한다.

**3단계 (30분): 본문 작성 및 번들 파일 준비**

선택한 패턴에 따라 본문을 상세하게 작성한다. 필요한 번들 파일(체크리스트, 템플릿, 예시 코드)을 준비한다.

**4단계 (15분): 테스트 및 개선**

Copilot Chat에서 최소 3가지 다른 프롬프트로 스킬을 테스트한다. 활성화, 지시 준수, 출력 일관성을 확인하고 필요시 수정한다.

**5단계 (15분): 설계 근거 문서 작성**

과제 제출용 설계 근거 보고서를 작성한다. 문제 정의, 패턴 선택 이유, description 설계 과정, 테스트 결과와 개선 사항을 포함한다.

## 과제

### 제출 과제 (2회차 종료 후)

1. **테스트 작성 스킬**
   - `.github/skills/test-writer/SKILL.md`
   - `examples/` 폴더에 예시 테스트 코드 최소 1개

2. **도메인 특화 스킬 (자유 주제)**
   - `.github/skills/{스킬명}/SKILL.md`
   - 번들 파일 최소 1개 (references/, assets/, examples/ 중 선택)

3. **설계 근거 보고서**
   다음 항목을 포함하는 보고서를 제출한다 (A4 1~2매 분량).
   - 문제 정의: 스킬로 해결하려는 문제는 무엇인가
   - 패턴 선택: 어떤 패턴을 선택했으며 그 이유는 무엇인가
   - description 설계 과정: 예상 프롬프트, 추출 키워드, 최종 description
   - 테스트 결과: 스크린샷 + 활성화 정확도, 지시 충실도, 출력 일관성 평가
   - 개선 사항: 테스트 과정에서 수정한 내용과 그 이유

### 제출 방법

- GitHub 리포지토리에 `.github/skills/` 디렉터리를 포함하여 커밋
- 설계 근거 보고서는 리포지토리의 `docs/` 폴더 또는 과제 제출 시스템에 업로드

## 핵심 정리

- 스킬 설계 패턴은 체크리스트형(항목별 점검), 워크플로우형(단계적 산출물 생성), 가드레일형(금지/제약 규칙)의 세 가지로 분류된다. 문제의 성격에 따라 적합한 패턴을 선택해야 한다.
- 커뮤니티 스킬(`awesome-copilot-skills`)은 유용한 출발점이지만, 팀 상황에 맞게 description, 규칙, 템플릿을 커스터마이징하여 사용해야 한다. 도입 전 SKILL.md 보안 검토는 필수이다.
- MCP는 "도구(실행 능력)"를, Skills는 "지식(판단 기준)"을 Copilot에 제공한다. 두 기술은 대체 관계가 아니라 보완 관계이며, 조합될 때 비로소 완전한 자동화 에이전트가 가능하다.
- 여러 스킬을 함께 사용할 때는 보완적 조합(서로 다른 관심사)과 계층적 조합(범용 + 특화)을 고려하되, description이 명확히 구분되고 지시가 상충하지 않도록 설계해야 한다.
- 도메인 특화 스킬 설계는 문제 정의 → 패턴 선택 → description 설계 → 본문/번들 작성 → 반복 검증의 5단계 프로세스를 따른다.
